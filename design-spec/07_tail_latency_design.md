# 07 - 尾延迟与自适应过载控制决策规范

> 回答"当瓶颈不是均值 / 吞吐、而是 **P99 / P999 长尾**与过载下的稳定性时，怎么判形态、选治理手段"。
>
> **核心命题**：前六视角压的是**均值与吞吐**（消灭往返、批量、并行、削峰）；长尾由一组**方向相反**的手段治理——对冲请求**主动增加**一次往返、自适应限流**主动让渡**吞吐换稳定。把均值压到极致后，决定用户体验与系统存活的，是尾部那一小撮请求和过载那一瞬间的行为。
>
> **边界先声明**：熔断算法 / 差异化参数 / 接入（[`docs-spec/12`](../docs-spec/12_circuit_breaker_spec.md)）、全局 SLA / 热路径 / 背压 / 容量（[`docs-spec/22`](../docs-spec/22_performance_contract_spec.md)）、跨服务超时协调 / 下游合约（[`docs-spec/24`](../docs-spec/24_cross_service_contract_spec.md)）、优雅关闭 / 探针 / 启动时序（[`docs-spec/27`](../docs-spec/27_deployment_runtime_spec.md)）等**规则真相源在对应 docs-spec**，本篇只引用不复制；静态池 sizing / Little 定律在 [`03`](03_concurrency_design.md)、截止预算逐跳传播与扇出失败语义在 [`01 §3.3/§3.4`](01_io_design.md)。本篇新增决策资产：**尾延迟三源识别、对冲 / 备份请求决策树、静态池 → 自适应并发限制升级、优先级 / 截止时间感知 load shedding、重试预算**。

## 1. 设计目标

- **压尾不压均值**：本视角优化 P99 / P999 而非 P50。接受为此付出的代价——对冲的**冗余请求**、自适应限流的**吞吐让渡**。均值已被前六视角压到极致时，再优化均值收益递减，长尾才是天花板。
- **过载有序降级，不雪崩**：系统饱和时按优先级 / 截止时间**主动丢弃**部分请求，而非让所有请求一起变慢直至集体超时（避免 metastable failure：延迟飙升后久久不能自愈）。
- **第一性约束（不可妥协）**：**任何尾延迟手段都不得放大故障**。对冲不得在下游已饱和时翻倍打、重试不得无预算放大、load shedding 必须在占用下游资源之前尽早丢。**治尾的手段一旦误用，反向制造的是雪崩。**

> **为什么尾延迟自成一视角**：长尾不由"单次操作多贵"决定（那是前六视角的战场），而由两条与单次成本无关的定律支配——① **扇出序统计**：响应并行聚合 N 个分支取最慢时，整体时延是 N 个样本的 **max**，单分支的小尾巴被放大成请求级常态；② **排队非线性**：资源近满载时排队时延 ∝ `1/(1−ρ)`（ρ=利用率），随"剩余余力"倒数发散、临界点成悬崖。两条都压不动单次成本——这正是前六视角覆盖不到、本视角必须独立的根因。图形直觉、数值与适用边界见 §9.6。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 高扇出聚合取最慢 | "一个响应并行等 N 个下游 / 分片，整体时延 = 最慢那个" | 对冲 / 扇出 partial |
| P99 ≫ P50 长尾 | "均值很好但 P99 / P999 明显长尾""偶发慢请求查不出规律" | 对冲 / 自适应限制 |
| 共享资源排队 | "热路径偶发卡顿，疑似 GC / 锁 / 连接池 / CPU 排队" | 自适应并发限制 |
| 过载会雪崩 | "流量尖峰后延迟飙升、久久不恢复" | load shedding + 自适应 |
| 重试放大风险 | "下游一抖动，重试就翻倍把它打得更死" | 重试预算 |
| 冷启尾延迟 | "发版 / 扩容后头几秒 P99 飙高" | 预热（缓存 [`05 §3.8`](05_caching_design.md) / 连接池 [`27 §4`](../docs-spec/27_deployment_runtime_spec.md)） |
| 严格端到端 SLA + 多跳 | "串 N 跳下游、有端到端时延上限、要保 P99" | 截止预算传播（[`01 §3.3`](01_io_design.md)）+ 对冲 |

> 识别口诀：**"均值好但偶发慢"→ 对冲 / 自适应；"一过载就崩、久不恢复"→ load shedding；"一抖动就被重试打死"→ 重试预算。**

## 3. 决策树（强制 / 本篇核心）

### 3.0 先定位长尾来源（治尾的前提是判准它从哪来）

不同来源的长尾要用**不同且不可互换**的手段，套错会放大故障：

```
P99 / P999 长尾来自哪？
│
├─ 个别下游 / 分片偶发慢（扇出取最慢） ──► 对冲 / 备份请求（§3.2）＋ 扇出 best-effort（01 §3.4）
├─ 本进程共享资源排队（GC / 锁 / 连接池 / CPU 调度） ──► 自适应并发限制（§3.3）＋ 运行时调优（22 §3 运行时旋钮）
├─ 系统性过载（到达率 λ 超过服务能力） ──► 优先级 / 截止感知 load shedding（§3.4）
├─ 下游瞬时抖动被重试放大 ──► 重试预算 + backoff + jitter（§3.5）
└─ 冷启动（发版 / 扩容头部窗口） ──► 缓存预热（05 §3.8）+ 连接池预热（27 §4），readiness 晚于预热
```

### 3.1 与 06 的分流（先判这是"治尾"还是"扛故障"）

```
稳定性诉求是哪一类？
│
├─ 依赖故障了怎么不被拖垮 ─────► 走 06：熔断 + 差异化降级（依赖挂了返什么）
├─ 入口超额怎么挡（固定天花板） ─► 走 06：限流（令牌桶 / 漏桶 / 滑窗）
├─ 正常运行下怎么削长尾 ────────► 本篇 §3.2 对冲 / §3.3 自适应限制
└─ 系统饱和怎么有序丢（动态找天花板） ─► 本篇 §3.4 优先级 load shedding
```

> **一句话分界**：限流（06）是**静态天花板**，自适应限制（07）是**动态找天花板**；降级（06）是依赖挂了返什么，load shedding（07）是系统满了丢谁。详见 §8 边界表。

### 3.2 对冲 / 备份请求决策树（削"取最慢"的长尾）

```
该调用要不要对冲？（对冲 = 超过 tie-breaking 分位后再并发发一个备份，取先到者）
│
├─ 调用幂等 / 只读？ ──否──► 禁止对冲（重复副作用）；先幂等化或改"超时 + 受控重试"
│                     └是─┐
├─ 下游有空闲容量余量？ ──否──► 禁止对冲（饱和时对冲翻倍打 = 放大过载）→ 先 §3.4 load shedding
│                        └是─┐
├─ P99 ≫ P50 且长尾来自少数慢实例 / 分片 ──► 对冲：主请求超过 tie-delay（取 P95~P99 分位）才发备份
│                                            ↳ 首响应胜出，其余立即 cancel（释放下游资源）
│                                            ↳ 对冲量设硬上限（如 ≤ 触发请求的 5%），并计入下游并发预算
└─ 多副本 / 多分片可取任一份 ──► tied request：同时发两个，任一**开始处理**即回信 cancel 另一个
```

> **对冲的反直觉点**：对冲与本支柱主线"**消灭往返**"（[`01`](01_io_design.md)）**直接冲突**——它故意多打一次。所以它是**长尾专用的最后一招**，只在"均值已优化、长尾不可压、下游有余量、调用幂等"四条同时成立时启用；任一不成立都退回超时 + 重试。

### 3.3 静态池 sizing → 自适应并发限制（削"排队"的长尾）

```
在途并发上限怎么定？
│
├─ 负载稳定、λ / W 可预估 ──► 静态：Little 定律算池容量（03 §3.2），固定 worker pool
│                              （这是默认；稳定负载别上自适应，徒增震荡与复杂度）
├─ λ / W 波动大、下游能力随时间漂移 ──► 自适应并发限制：按观测 RTT 反馈实时调在途上限
│      ├─ AIMD（乘性减 / 加性增）：实现简单，类 TCP 拥塞控制
│      └─ 梯度式（min_rtt 与当前 rtt 的比值定 limit）：响应更平滑、少震荡
└─ 上限被打满时怎么办？ ──► 不排无界队列：有界排队 + 触发 §3.4 load shedding，拒绝返明确 429 + Retry-After
```

> 自适应限制是对 03 静态 Little 定律的**升级**而非替代：静态 sizing 假设 λ 与 W 恒定；当二者波动快于静态假设时，固定池要么过小（排队长尾）要么过大（下游被打爆），自适应限制让在途上限随实测 RTT 自己找到拐点。

### 3.4 优先级 / 截止时间感知 load shedding（过载时有序丢）

```
系统饱和（队列积压 / CPU 满 / 自适应 limit 触顶）时丢谁？
│
├─ 请求已超自身截止预算（注定超时） ──► 立即丢（deadline-aware：别为"死人"干活，01 §3.3）
├─ 有请求优先级分层？ ──► 先丢低优先级（可重试 / 可降级 / 后台），保高优先级（交互 / 付费 / 关键）
├─ 无优先级、需均匀降载 ──► 概率丢弃（与 06 §3.2 自适应熔断同构，但作用于**入口**而非依赖）
└─ 丢弃必须快且廉价 ──► 在入口尽早丢（占用下游资源**之前**），返明确 429 / 503 + Retry-After，不静默 hang
```

> load shedding 与限流（06）的区别：限流是**事前**按固定速率挡；load shedding 是**事中**在真实饱和信号下，按**优先级 / 截止时间**选择性丢——保的是"过载时系统仍对高优先级有序响应"，而非"把吞吐钉死在某个数"。

### 3.5 重试预算（防重试风暴）

```
要不要重试 / 重试几次？
│
├─ 操作幂等 / 安全？ ──否──► 禁止重试（先幂等化，见 04 幂等设计）
├─ 重试预算未超？（全局重试数 ≤ 总请求的 X%，如 10%） ──否──► 不重试（预算耗尽即停，防放大）
├─ 退避：指数 backoff ＋ 抖动 jitter（防多客户端重试同步成群、二次冲击）
└─ 与熔断联动：熔断 Open 期间**不重试**（06）；重试只救"偶发瞬时失败"，不救"持续故障"
```

> 无预算的重试是**级联失败的头号放大器**：下游抖动 → 上游重试翻倍 → 下游更死 → 重试更多。重试预算把"重试总量"锁在请求量的一个小比例内，与熔断（06）叠加形成双保险。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 单下游调用 | 超时（06） | P99 ≫ P50 ＋ 下游有余量 ＋ 幂等 → 对冲（§3.2） | 下游饱和 / 非幂等 → **禁对冲** |
| 高扇出聚合 | 并行 + best-effort（[`01 §3.4`](01_io_design.md)） | 个别分支长尾拖垮整体 → 对冲慢分支 | 全员对冲 → 翻倍负载 |
| 在途并发上限 | 静态 Little 定律池（[`03 §3.2`](03_concurrency_design.md)） | λ / W 波动大 → 自适应并发限制（§3.3） | 稳定负载 → **别上自适应**（震荡 + 复杂度） |
| 过载保护 | 入口限流（06） | 需保高优先级 / 防雪崩 → 优先级 load shedding（§3.4） | 低负载 → 别丢（误杀） |
| 重试 | 不重试 / 06 基线重试 1 | 偶发瞬时失败 ＋ 幂等 → 重试预算 + backoff + jitter（§3.5） | 无预算重试 → 重试风暴 |
| 冷启 | readiness 晚于资源初始化（27） | 头部 P99 飙高 → 缓存（05）+ 连接池（27）预热，readiness 晚于预热 | 已稳态 → 无需预热 |

> **反向纪律是本视角的命脉**：对冲 / 自适应 / 重试**全部"非默认"**。它们每一个误用都从"治尾"反转成"造灾"——对冲翻倍负载、自适应在稳定负载上无谓震荡、重试无预算放大故障。**规范必须同时规定"何时不做"**，与正向手段同等重要。

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 尾延迟手段权衡矩阵

| 手段 | 治尾效果 | 成本 | 过载安全性 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| 对冲请求 | 高（削最慢分支） | 冗余请求（+5%~+100%） | **差**（饱和时放大）→ 必配预算 | 中 | P99≫P50 ＋ 下游有余量 ＋ 幂等 |
| tied request | 高 | 低（任一开始即 cancel） | 中 | 中 | 多副本 / 分片可取任一 |
| 自适应并发限制 | 中（防排队长尾） | 吞吐让渡 | 好 | 高 | λ / W 波动大 |
| 优先级 load shedding | —（保稳定非降时延） | 丢部分请求 | **好**（防雪崩） | 中 | 系统性过载 |
| 重试预算 | 低（救偶发瞬时） | 额外负载（有界） | 中（预算护栏） | 低 | 幂等 ＋ 瞬时失败 |
| 静态池（对照，03/06） | —（不治尾） | — | 中 | 低 | 稳定负载 |

### 5.2 借纪律不借实现：按尾延迟来源重选手段（强制）

同一条"削 P99"的纪律，**手段随长尾来源变**（[`00 §2.4`](00_design_overview.md)）：扇出取最慢 → 对冲；进程内排队 → 自适应限制；系统性过载 → load shedding。**把对冲硬套到过载场景会直接放大故障**（饱和时再翻倍打）。可迁移的是"先定位三源、再分别下药 + 永远给护栏（预算 / 上限 / deadline）"这套纪律，不是某个具体的对冲分位或 AIMD 系数。

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向 |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）：重试无 `jitter`（grep backoff 拼装缺随机项）、对冲 / 重试包裹非幂等调用（污点：写类方法名进对冲 / 重试入口）、`make(chan T, ...)` 无界兜底——命中即硬拦截。"预算是否合理""优先级分层是否齐"等语义类违规走选型审查（软提醒）。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节的数值、分位、系数、代码均为某真实工程 / 公开文献（Dean & Barroso《The Tail at Scale》, CACM 2013；Netflix concurrency-limits）的具体实现，**只借纪律不借实现**——落地工程时按自己的长尾来源与实测分布重选手段与参数。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 对冲请求（削扇出取最慢的长尾）

```go
// 对冲：主请求超过 tie-delay（取 P95 分位）才发备份；首响应胜出，其余 cancel
// 前置闸门：调用必须幂等 / 只读；下游须有容量余量；对冲量计入并发预算
func hedgedGet(ctx context.Context, key string) (Result, error) {
    ctx, cancel := context.WithCancel(ctx); defer cancel()
    ch := make(chan Result, 2)
    go fetch(ctx, replicaA, key, ch)            // 主请求
    timer := time.NewTimer(p95Latency)          // tie-breaking delay = 实测 P95
    defer timer.Stop()
    select {
    case r := <-ch:        return r, nil        // 主请求在 P95 前返回 → 不发备份（绝大多数请求走这）
    case <-timer.C:        go fetch(ctx, replicaB, key, ch) // 仅尾部 ~5% 触发备份
    case <-ctx.Done():     return Result{}, ctx.Err()
    }
    return <-ch, nil                            // 取先到者；cancel() 通过 ctx 终止落败的那个
}
```

> 公开实证（《The Tail at Scale》）：对 99th 分位后发一个备份请求，某 Google 服务 P999 从 ~1800ms 降到 ~74ms，额外负载仅约 +2%。关键在 tie-delay 取分位足够高 → 只有真·长尾才触发对冲，冗余可控。

### 9.2 自适应并发限制（梯度式 / AIMD）

```
梯度式（类 Netflix concurrency-limits）：
    gradient = min_rtt / sample_rtt              // ∈ (0,1]，越拥塞越小
    new_limit = current_limit × gradient + queue_size   // rtt 升高 → limit 自动收
AIMD（类 TCP 拥塞）：
    成功且未排队 → limit += 1（加性增）
    超时 / 排队 → limit = limit × 0.8（乘性减）
limit 触顶 → 新请求有界排队，溢出走 §9.3 load shedding（429 + Retry-After）
```

稳定负载下静态 Little 定律池足矣（[`03 §9.2`](03_concurrency_design.md)）；仅当下游能力随时段 / 故障漂移、固定池要么排队要么打爆时，才升级到自适应。

### 9.3 优先级 / 截止时间感知 load shedding

```
入口中间件按 X-Priority（interactive=0 / batch=1 / bestEffort=2）与剩余截止分层：
  if 已超请求自身 deadline:           shed（立即 429，别占下游资源）   // deadline-aware
  if 自适应 limit 触顶 / CPU>阈值:
        丢 priority 最大者（bestEffort 先丢）→ 仍过载则丢 batch → 最后才碰 interactive
  无优先级时退化为概率丢弃 p（与 06 §3.2 同构，但作用于入口）
```

### 9.4 重试预算（防风暴）

```
重试令牌桶：retryBudget = 请求量 × 10%，重试前 TryAcquire，桶空即不重试
退避：sleep = base × 2^attempt × (1 + rand[0,1)×jitterRatio)   // jitter 打散同步重试
联动：熔断 Open → 直接不重试（06）；仅对幂等操作 + 瞬时错误（超时 / 503）重试
```

### 9.5 冷启预热（削发版 / 扩容头部的尾延迟）

```
readiness gate 必须晚于：① 缓存预热（05 §3.8 加载热 key / 本地镜像）
                         ② 连接池预热（27 §4：MySQL/Redis 预拨 MinIdle 条连接，避开首请求握手长尾）
预热未完成 → readiness=false → 流量不打进来；预热完 → 置 true 才 accept
（与 27 §4 启动就绪时序、§5 优雅关闭对称：冷启不接、关闭排空）
```

### 9.6 两条尾延迟定律的图形直觉（为什么 07 自成一视角）

> 把 §1 两条定律画出来——长尾的"形状"一眼可见。数值为示意，重点是**曲线的形**不是具体数字。

**① 扇出放大尾部**（序统计 max-of-N）：单分支慢概率 p（示意 1%），扇出 N 个取最慢 → `P(请求慢)=1−(1−p)^N`

```
N=  1 │█ 1%
N= 10 │██████ 10%
N= 20 │███████████ 18%
N= 50 │███████████████████████ 39%
N=100 │█████████████████████████████████████ 63%  ← 已过半
N=200 │███████████████████████████████████████████████████ 87%
```

单分支"百里挑一"的罕见慢，扇出 100 后变成**过半请求都慢**——单分支 P99 ≈ 请求 P50。越往分布式 / 多分片扩越被尾巴吃，与单次成本无关。

**② 排队悬崖**（近满载非线性）：相对延迟 `= 1/(1−ρ)`（×服务时间），ρ = 利用率 λ/μ

```
ρ=0.50 │██ 2×
ρ=0.70 │███ 3.3×
ρ=0.80 │█████ 5×
ρ=0.90 │██████████ 10×
ρ=0.95 │████████████████████ 20×
ρ=0.99 │██████████████████████████████████████████████████ 100× →∞
```

`(1−ρ)` 就是"剩余追赶余力"，趋零 → 延迟冲天；膝盖≈0.8，过了掉崖。**尾分位先撞 SLO 红线**，是过载的早期警报。

> ⚠️ **适用边界（借纪律不借实现）**：`1/(1−ρ)` 是 **M/M/1**（单服务台 + 指数服务 + ρ<1）。**多服务台**（连接池 c 条 / 多核，M/M/c）膝盖更晚更缓；服务时间**恒定**（M/D/1）减半、**重尾**（M/G/1，×`(1+C²)/2`）更陡；ρ≥1 无稳态、队列无界增长（metastable）；真实 μ 随负载退化 → 实际更陡。**可迁移的是"扇出放大尾部、近满载非线性、留膝盖以下余力"这套纪律，不是这条具体曲线**——单线程 Redis / 单核热路径最贴 M/M/1，连接池 / 多核按 M/M/c 放松。
