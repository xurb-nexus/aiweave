# 03 - 并发模型设计决策规范

> 回答"面对一个需求，怎么选并发原语、给并行 fan-out 的协程池定容量、管好共享状态与 goroutine 生命周期"。并行是 IO 视角（[`01`](01_io_design.md)）"消灭独立串行"的执行底座。
>
> **边界先声明**：共享状态注册表、锁策略、Channel 规则、资源生命周期、危险操作清单、worker pool 容量等**约束规则的真相源在 [`docs-spec/20`](../docs-spec/20_concurrency_safety_spec.md)**；channel 容量 / 背压在 [`docs-spec/22 §6.1`](../docs-spec/22_performance_contract_spec.md)，本篇只引用不复制。本篇新增决策资产：并发原语选型、**协程池容量定量方法（Little 定律）**、阻塞不变量、内联兜底提交器。

---

## 1. 设计目标

- **无数据竞争 / 生命周期可控 / 背压可控**：共享可变状态有同步策略；每个 goroutine 有明确退出路径；无界扇出被 worker pool 约束。
- **池上限不成为瓶颈**：IO-bound 并行的 worker 上限必须足够大，绝不能小到把并行退化成同步等待。
- **提交永不丢、WaitGroup 永配平**：`Submit + WaitGroup` 编排不会因提交失败而永久挂死。

## 3. 决策树（强制）

### 3.1 并发原语选型

```
并发问题是哪种形态？
│
├─ 共享可变状态 ───────► 锁 vs 原子 vs channel 传递所有权 vs 不可变拷贝
│                          ↳ 高频计数 / 状态写? 优先单命令原子性（如 Redis HINCRBY），不在热路径加锁
├─ 并行 fan-out 一批 ──► worker pool / errgroup（受上限约束，容量见 §3.2）
├─ 常驻后台协程 ───────► ctx 控制退出；统一刷新用单后台协程（多任务顺序刷新而非并行）
└─ 取消 / 超时传播 ────► ctx 透传；异步任务用派生 ctx（脱离请求生命周期）
```

> **⚠️ "单后台协程"的多副本陷阱（强制过 [`10`](10_scheduling_coordination_design.md)）**：本节"单后台协程顺序刷新"是**单进程内**的并发结论；一旦服务多副本部署，"单后台协程"就是 **N 个副本各跑一份**。该任务"每副本各跑是否无害"是个独立的协调决策——无害（各刷各的本地 / 靠原子认领去重）则照常；有害（重复发通知 / 重复副作用）则须单例化（原子认领 / 分布式锁 / leader 选举 / 外部调度）。**判定与选型走 [`10 §3`](10_scheduling_coordination_design.md)**，本视角只管单进程内的协程生命周期。

### 3.2 协程池容量定量方法（Little 定律 / 强制）

IO-bound 任务的 worker 上限**必须远大于 CPU 核数**——goroutine 阻塞在 IO 上时不占 CPU，"按核数设上限"是错的：IO 越慢、单任务驻留越久，越需要更多 goroutine 同时挂在 IO 上等待。

```
需要的并发协程数  N ≈ 到达速率 λ × 单任务 IO 耗时 W      （Little 定律，给上限的下界）
                  IO 变慢（W↑）→ 需要的并发线性上升

上限 = max( N（Little 估算）, 下游可承载并发 )           // 取大者，保证上限不成为瓶颈
       再以「内存可控 + 不压垮下游」封顶，并强制阻塞模式做背压
```

只有 **CPU-bound** 任务才让 worker ≈ 核数。

> **CPU-bound 定容的前提是"真实可用核数"**：容器里 `GOMAXPROCS` 必须对齐 CPU limit（automaxprocs），否则默认 = 宿主核数 → CPU 池按虚高核数定容、实际被 cgroup 限流 → 调度争用反劣化。运行时旋钮（GOMAXPROCS / GOMEMLIMIT / GOGC）硬规则见 [`docs-spec/22 §3.4`](../docs-spec/22_performance_contract_spec.md) 与 [`27 §7`](../docs-spec/27_deployment_runtime_spec.md)。

### 3.3 协程池工程不变量（强制）

```
① 按职责物理隔离 + 容量分级 ──► 不同优先级/失败语义用独立池；best-effort 池要小且"池满即丢"，绝不挤占查询能力
② 阻塞模式不变量 ────────────► 查询池必须阻塞模式（禁非阻塞/丢弃）；否则池满静默丢任务 + WaitGroup → 死锁
③ 内联兜底提交器 ────────────► 禁止裸 Submit；封装唯一提交入口，Submit 失败则当前 goroutine 内联同步执行
                                （保证 task 永不丢、wg 永配平，根除 wg.Wait() 永久挂死）
```

### 3.4 池分类：按隔离维度拆池（强制）

> 一个任务流是否值得独立成池：看它在下列任一维度上是否与现有池不同——命中即拆（舱壁隔离），避免互相拖累 / 嵌套死锁 / 噪声邻居。维度正交，一个池常由多维度共同定义。

| 隔离维度 | 池原型 | 定容规则 | 失败语义 |
| --- | --- | --- | --- |
| 高优先级 · IO 短任务 | 在线读 / 查询池 | Little（≫ 核数 / 单库连接池） | 阻塞排队（不可丢） |
| 组合多读（防嵌套死锁） | 编排 / 扇出池 | 同查询量级；**须与读池物理分离** | 阻塞排队 |
| 必达写 | 关键写 / 持久化池 | min(Little, 下游写承载) | 阻塞 + 内联兜底（§3.3③） |
| 旁路 best-effort | 异步回写 / 上报池 | 小（不与前台抢） | 池满即丢 |
| 下游归属（舱壁） | 外部 RPC 池（每下游独立） | min(Little, 下游承载) | 快速失败 / 降级（熔断见 [`06`](06_resilience_design.md)） |
| CPU-bound | CPU 计算池 | ≈ 核数（§3.2 反例规则） | 阻塞排队 |
| 脱离请求生命周期 | 后台 / 刷新（多为单协程，非池） | —（顺序刷新） | 自带退避 / 重试 |

> 不是每个系统都要全部七类。**最小集是「在线读 +（关键写 / 回写）」两到三池**；下游多 / CPU 重 / 消息驱动的系统再逐步分出 RPC 舱、计算池、消费池（消费池 ≈ 下游归属 + 位点语义，关闭须排空，见部署 [`27 §5`](../docs-spec/27_deployment_runtime_spec.md)）。**先合后拆**：无争用证据时先用少池，观测到互抢 / 噪声邻居再拆——过度隔离同样浪费资源。

### 3.5 热点计数争用：分片计数 vs 单点原子 / 锁（含 false-sharing）

```
一个超高频计数 / 累加，并发争用怎么治？
│
├─ 单点 atomic（CAS / atomic.Add） ──► 中低频够用；超高频下所有核争同一缓存行 → 总线风暴
├─ 单点 Mutex ───────────────────► 临界区再短，高争用也排队；计数类绝不该用
├─ 分片计数（striped counter） ───► 拆 N 个独立计数器（按 P / CPU / hash 散列），读时求和
│        ↳ 写分散到 N 个缓存行 → 争用降 ~N 倍；读 O(N) 求和（写 ≫ 读时划算）
│        ↳ N 通常取 ~核数；与缓存写热 key 分片计数（05 §9）同构
└─ false-sharing 规避 ──► 分片 / 原子计数器按缓存行（64B）对齐填充，否则相邻计数器共享缓存行 → 假共享，分片白拆
```

> **分片计数的代价是读**：写从"N 核争一行"摊成"各写各行"，但读要扫 N 片求和 → 仅当**写 ≫ 读**时划算（计数器正是此形态）。**false-sharing 是隐形杀手**：把 N 个分片塞进同一缓存行，分片就白拆了——必须按缓存行对齐（padding / 独占 cache line）。原子 vs 锁：计数类恒选原子 / 分片，锁只留给"多字段一致更新"的临界区。分片计数登记与 false-sharing 危险操作的表示真相源在 [`docs-spec/20`](../docs-spec/20_concurrency_safety_spec.md)。

## 4. 默认选型与升级路径

| 场景 | 默认选型 | 升级触发 → 升级到 |
| --- | --- | --- |
| 共享状态 | `sync.Mutex` 守护 | 高频计数 → 原子 / 单命令原子性；读多写少 → RWMutex / 分片锁 |
| 超高频计数 | 单点原子（atomic.Add） | 单缓存行总线风暴 → 分片计数 + false-sharing 对齐（§3.5） |
| 并行处理 | errgroup + 上限 | 多缓存兜底混合编排 → 聚合器（01）；长流水线 → pipeline / 有界队列 |
| 池划分 | 单池 | 多优先级 / 多失败语义 / 多下游 → 按职责分池（见 §3.4 池原型） |
| 提交方式 | 内联兜底统一入口 | —（裸 Submit 禁止，无升级） |
| 后台协程 | ctx 控制退出 | 多任务刷新 → 单后台协程顺序刷新 |

## 5. 权衡矩阵

| 方案 | 安全性 | 吞吐 | 复杂度 | 适用 |
| --- | --- | --- | --- | --- |
| Mutex | 高 | 中 | 低 | 默认共享状态 |
| 原子 / 单命令原子性 | 高 | 高 | 低 | 高频计数 / 状态写 |
| channel 传递所有权 | 高 | 中 | 中 | 流水线 / 生产消费 |
| worker pool fan-out | 高（有界） | 高 | 中 | 并行 IO |
| 分片计数（striped） | 高 | 极高（写分散） | 中（读 O(N) 求和 + 对齐） | 超高频计数热点 |

> 池容量取值：IO-bound 取 `max(Little 估算, 下游可承载)`，远大于核数；best-effort 池小且池满即丢。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节数值 / 代码为某真实工程实现，**只借纪律不借实现**。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 三池物理隔离 + 容量分级

> 本例取 §3.4 池原型中的 **在线读 / 编排 / 异步回写** 三类，给一份物理隔离 + 容量分级的落地切片。

| 池 | 容量（相对） | 用途 | 失败语义 |
| --- | --- | --- | --- |
| 查询池（Repo 内多分表并行） | 大（Little 估算，远超核数 / 单库连接池） | 高优先级 | 池满阻塞排队 |
| 跨模块编排池（编排器 fan-out） | 大（同查询量级） | 高优先级 | 池满阻塞排队 |
| 回写池（缓存异步回写） | 小一档 | 低优先级 best-effort | 池满即丢 |

查询池容量远大于单库连接池——它汇聚跨多个分库、上百分表的总并发，单任务很短（单次 IN，周转快），按 Little 定律稳态并发需求很高；回写池小一个量级——best-effort，不跟查询抢资源。**借的是量级关系（查询池 ≫ 单库连接池、回写池小一档），绝对值按 §3.2 Little 定律据业务推导。**

### 9.2 Little 定律取值

```
某并行查询阶段：λ=2000 次/秒、单次 IO W=20ms → 稳态需 N ≈ 2000 × 0.02 = 40 goroutine 在飞
IO 变慢 W→40ms → 需要的并发翻倍到 80。池上限必须 ≥ 这个 N，否则同步等待。
```

### 9.3 内联兜底提交器

```go
// ✅ 统一提交入口，禁止裸 pool.Submit()
func Go(pool Pool, task func()) {
    if err := pool.Submit(task); err != nil {
        task()   // Submit 失败则在当前 goroutine 内联同步执行 → task 永不丢、wg 永配平
    }
}
```

### 9.4 嵌入式库并发（本地 SQLite，组件蓝图与原子发布不变量详见 [`05 §9.4`](05_caching_design.md)）

本地 SQLite 读是**进程内、命中私有 page cache 的 CPU-bound 访问（~1–50µs）**，不是网络 IO（ms 级）——所以读池按 §3.2 的 **CPU-bound 规则按核数定容**，而非套用网络池量级（~100）。

- **读池** `MaxOpenConns = min(max(2×NumCPU, 16), 64)` 且长存保温（连接复用 → 保住私有 page cache）：
  - 按核数定容——有用读并发 ≤ NumCPU；`2×` 留给"偶发冷页 OS 读"短暂转 IO-bound 的余量；
  - 下界 16 兜小核机 / 突发；上界 64 防超核连接只复制私有 page cache 白耗内存（超 NumCPU 不增吞吐）。
  - Little 定律佐证：即便 λ=10 万次/秒、W=50µs，稳态仅需 ≈5 条连接——吞吐从不是约束，故无需网络池量级。
- **写池** `MaxOpenConns=1` 把写并发在 Go 池层钳成串行，根除 `database is locked`。
- 单后台协程顺序刷新全部表（写本就串行，并行不更快）。
