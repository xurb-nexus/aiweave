# 01 - IO 设计决策规范 ⭐

> 本支柱的**旗舰范本**。回答"面对一个涉及数据访问的需求，怎么把它的 IO 形态判出来、选对编排方案"。核心命题：**把 IO 解决了，工程的技术天花板就高**（见 [`00 §2.2`](00_design_overview.md)）。
>
> **边界先声明**：三条 IO 铁律、聚合器五阶段、IO 往返预算、IO 回归测试等**硬规则的真相源在 [`docs-spec/25`](../docs-spec/25_io_aggregation_spec.md)**，本篇**只引用不复制**；并行原语的容量 / goroutine 生命周期在 [`docs-spec/20`](../docs-spec/20_concurrency_safety_spec.md) 与 [`03`](03_concurrency_design.md)。本篇新增的是上游决策资产：**三维度框架、读写路径分治、读/写两棵决策树、默认选型与升级路径、编排方案权衡矩阵、反模式速查**。

## 1. 设计目标：把 IO 的三个维度压到最小

IO 视角优化的第一性约束，拆成三个**可度量维度**——每个反模式都挂在某一维上，本篇全部决策都是对这三维做压缩：

| 维度 | 反模式 | 目标 | 主要手段 |
| --- | --- | --- | --- |
| **次数** | N 次单条查询（N+1） | 1 次批量 | 批量方法 `{...List}` / 批量缓存读 / 内存分组 |
| **串 / 并行** | 互不依赖的查询串行 | 同阶段并行 | 聚合器 / 扇出并行 / worker pool |
| **往返** | 每次访问一个远端 RTT | 0 往返（本地命中）或单次往返 | 多级缓存 / 本地镜像 / 消重复读 / 异步写回 |

> 这三维直接来自 PRINCIPLES §13.1——"重建出的代码默认就是 IO 高效的"。本篇的任务是把这三维从"事后审计"前移到"生成方案时就选对"。
>
> **横切补充**：三维度优化的是「均值 / 吞吐」；另有两条**横切纪律**保证压缩在生产可持续——**截止时间预算逐跳传播（§3.3）**让超时不向下游放大、**扇出失败语义（§3.4）**让并行的局部失败可控降级。

## 2. 输入信号识别（需求落入本视角的迹象）

一个需求只要命中下列任一迹象，**强制**过本视角的决策树：

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 集合处理 | "对一批 `{entity}` 逐个补全 / 校验 / 计算" | 读 |
| 多依赖编排 | "一个响应要同时拿 `{A}`、`{B}`、`{C}` 几份数据" | 读 |
| 跨实例 / 跨数据源读 | "从多个缓存实例 / 多个库取数据再合并" | 读 |
| 高频判定读 | "超高频读一个 `{key}` 做放行 / 判定" | 读 |
| 高频状态写 | "高频累加 / 扣减 `{核心数值字段}`" | 写 |
| 纯追加写 | "持续产生 `{流水}`，只追加不更新" | 写 |
| 扇出调用 | "对每个 `{item}` 调一次下游 / 外部 API" | 读 / 写 |
| 高频独立点查 | "超高频按 `{key}` 点查同一类资源（不同 key、来自不同请求）" | 读 |
| 多跳 + 端到端 SLA | "一个响应要串 N 跳下游，且有端到端时延上限" | 读 / 写 |

> 识别口诀：**"对每个…"或"同时要…"→ 读路径决策树；"高频写 / 持续追加"→ 写路径决策树。**

## 3. 读写路径分治 + 两棵决策树（强制 / 本篇核心）

### 3.0 先分治：读和写用相反的手段

读路径与写路径是**两种相反的 IO 模式**（[`00 §2.3`](00_design_overview.md)）。一个请求里同时存在读写时，**分别用两套手段，不要用统一"数据访问层"抹平**：

```
                    ┌──────────────────────────────────────┐
   请求里的数据访问 ─┤  是读还是写？                          │
                    └───────┬───────────────────┬──────────┘
                            ▼ 读                 ▼ 写
                   目标：极低延迟          目标：极高吞吐 / 削峰
                   优先级：消灭往返 >       手段：先写内存态 → 脏标记
                   合并往返                积攒 → 定时增量批量落库；
                                          纯追加走 MQ 批量
                            │                   │
                       §3.1 读路径树        §3.2 写路径树
```

### 3.1 读路径决策树（消灭往返 > 合并往返）

```
读访问是哪种形态？
│
├─ 单实体读（一个 key 拿一份） ──► 多级缓存：本地命中(0 网络) → 单次往返
│                                   ↳ 缓存是数据源? miss=拒绝，不回源（详见 05 非对称降级）
│
├─ 集合 enrich（对一批主体补关联） ──► 循环外收集 key → 一次批量读 → map 装配（铁律一 / 25 §2.1）
│                                   ↳ 关联来自多源? 每源一次批量读后 map 合并
│
├─ 多个互不依赖读（同时要 N 份） ──► 同阶段并行（铁律二 / 25 §2.2）
│                                   ↳ 多份带缓存兜底? 聚合器（收集→分组批量读→并行回源→单飞→异步写回）
│                                   ↳ 纯无缓存?       扇出并行（worker pool，容量见 03）
│
├─ 跨实例 / 跨数据源读 ──────────► 按实例分组，每实例一次批量读（聚合器）
│
├─ 高频独立点查（不同 key、跨请求、同一时间窗） ──► 跨请求微批加载器：短时窗内自动攒批 → 一次 MGET / 批量读
│                                   ↳ 单飞合并"同 key"、微批合并"不同 key"，两者正交叠加；点查稀疏 / 低并发禁套（徒增批窗延迟）
│
└─ 同一 key 在链路里被读多次 ──► 消灭重复读：一个 key 全链路只读一次，中间结果（派生值）透传复用
                                    ＋ 按分支裁剪跳过无关查询
```

> **DB 读走覆盖索引**：缓存 miss 回源 / 直查 DB 的读，让查询走**覆盖索引**（所需列全在索引 → index-only scan 免回表），是"消灭往返"在 DB 层的延伸（索引设计见 [`02 §3.5`](02_data_model_design.md)）。
>
> **读路径的反直觉点**：已被本地缓存命中的读，**绝不要再合并进 Pipeline**——那会绕过本地缓存反而劣化。读路径优先级恒为 **消灭往返 > 合并往返**。

### 3.2 写路径决策树（削峰 + 摊销）

```
写访问是哪种形态？
│
├─ 有中间态、需聚合（高频累加 / 扣减） ──► Redis-first：内存态原子命令(O(1) 无锁)
│                                          ＋ 脏标记集合积攒 ＋ 定时增量批量 flush 落库
│                                          ↳ 并发安全用单命令原子性，不在热路径加锁/Lua
│                                          ↳ 越界（超扣）在持久化边界兜底（见 04），不在热路径拦
│
├─ 纯追加、无中间态（流水 / 日志，量极大） ──► 投 MQ（按 `{key}` 保序）→ 消费端攒批
│                                          → 批量 INSERT（+ 分表）→ 重试 / 死信兜底
│
└─ 低频普通写 ──────────────────────────► 直接写（事务边界见 04），无需削峰
```

> **批量插入/更新混合** → 批量 Upsert（`ON DUPLICATE KEY UPDATE`/`ON CONFLICT`）一次往返，免 select-then-write；**大批量装载**（冷启 / 大迁移 / 离线导入）→ `LOAD DATA`/`COPY` 批量装载，比多行 INSERT 快约 10-100×。
>
> **写路径恒为 best-effort 异步优先**：能移出响应关键路径的写（缓存回写、流水落库）都异步化，并与请求生命周期物理解耦（派生 ctx）。

### 3.3 截止时间预算逐跳传播 + 预算耗尽降载（横切读写 / 强制）

§1 给的是往返**次数**预算；本节补往返**时间**预算——它横切读写两棵树，让"快"在过载时不反噬：

```
请求带端到端预算 deadline ──► 每跳下游拿到的是【剩余预算】，而非各自固定超时
│
├─ 剩余预算 ≥ 本跳成本 ──► 正常发起；ctx 带【剩余 deadline】透传给下游（逐跳递减）
├─ 剩余预算 < 本跳成本 ──► 不发起注定超时的调用：关键依赖 fast-fail，非关键依赖直接跳过（中途降载）
└─ 已超总预算 ──────────► 立即返回，绝不再向下游发起任何新 IO（消灭"上游已超时、下游还在干"的放大型无效工作）
```

> 与连接池 per-resource 固定超时（基线见 [`06 §9.2`](06_resilience_design.md)）的区别：固定超时是"单跳上限"，本节是"端到端预算在跳间递减分配"，两者叠加 → 实际超时 = `min(单跳上限, 剩余预算)`。链路超时协调 / 优雅降级层级的表示真相源见 [`docs-spec/22 §6.2`](../docs-spec/22_performance_contract_spec.md)、§6.3。

### 3.4 扇出失败语义：all-or-nothing vs best-effort partial（强制取舍）

并行扇出 / 聚合器并行回源时，部分分支失败怎么办？**先按依赖性质定失败语义，再选编排：**

```
扇出某分支失败 → 整体怎么处理？
│
├─ 关键依赖（缺它响应不成立） ──────────► all-or-nothing：firstErr 即整体失败（errgroup 默认语义）
├─ 可降级依赖（缺它可返回部分 / 默认值） ──► best-effort partial：收集成功分支 + 标记失败分支 → 返回部分结果
│                                          ↳ 失败分支按依赖性质降级：只读依赖返空 / best-effort 依赖静默（见 06）
└─ 混合 ──────────────────────────────► 关键分支 all-or-nothing 闸门 ＋ 可降级分支 best-effort 收集
```

> 默认 errgroup 是 all-or-nothing（一败俱败）；**需要部分可用时必须显式改写为 best-effort 收集**——别让一个可降级分支的失败拖垮整个响应，也别用 best-effort 放过"关键分支失败却返回部分脏结果"。降级三类法（按依赖性质）真相源见 [`docs-spec/12 §5.1`](../docs-spec/12_circuit_breaker_spec.md)；并行原语见 [`docs-spec/25 §4`](../docs-spec/25_io_aggregation_spec.md) / [`03`](03_concurrency_design.md)。

### 3.5 数据搬运最小化：计算下推 · 数据本地性 · 大写分批 · 传输层（横切读写 / 补充）

三维度压的是"往返次数与并行"；本节补的是**每次往返搬多少数据、数据离计算多近**——同样的 RTT 次数下，少搬数据 / 数据更近 = 更低延迟：

```
├─ 计算下推（把计算推到数据侧，而非把数据捞到计算侧） ──► 过滤 / 聚合 / 排序 / 计数能在 DB 做就别捞到应用层
│        ↳ WHERE / GROUP BY / COUNT / LIMIT 下推 DB（走索引）；禁"SELECT 全量 → 应用层 filter/sum"
│        ↳ 反向纪律：复杂业务逻辑别硬塞存储过程（不可测 / 难演进）——下推的是"无业务语义的数据密集算子"
├─ 数据本地性（让关联数据物理上靠近） ──► 常一起读的关联数据 co-locate 同分片（避免跨分片 / 跨网络扇出）
│        ↳ 分片键选"关联查询的连接键"，让 JOIN / 批量在单分片内完成（与 02 分库分表联动）
├─ 大写事务分批（巨型写拆小，避开复制延迟尖峰 + 长持锁） ──► 百万行 UPDATE/DELETE → keyset 分块小事务
│        ↳ 与 04 §3.3 group-commit 互补：那是把小事务【合】摊 fsync，这是把巨型写【拆】；都避"一次性大事务"
│        ↳ 在线回填走 11 §3.3 限流分批（监控复制延迟自动降速）
└─ 传输层（连接 / 协议层面的往返优化） ──► 连接复用（长连接 / 池）、HTTP/2 多路复用、gRPC streaming、批量 RPC
         ↳ 协议选型见 09 §3.6；避免"每次新建连接握手"的隐性往返（与连接池基线 06 §9.2 联动）
```

> **"消灭往返"的延伸是"消灭无效数据搬运"**：把 10 万行捞到应用层再 `sum` 是 1 次往返却搬了海量数据 + 吃 CPU/GC——往返次数对了，数据量错了。计算下推让"次数"与"载荷"同时最优。大写分批的硬规则（分块游标）见 [`02 §3.6`](02_data_model_design.md)，复制延迟取舍见 [`02 §3.7`](02_data_model_design.md)。

## 4. 默认选型与升级路径（强制）

### 4.1 三句话默认原则

任何请求路径上的数据访问，先按三句话定默认：**批量优先 · 并行优先 · 单次往返优先**。

| 默认 | 含义 | 反向纪律（防过度优化） |
| --- | --- | --- |
| **批量优先** | 对"一批"东西的查询，用批量方法一次完成；**数据层若缺批量方法，先补 `{...List}` 方法**（分表按 shard 分组并行），再在上层调用——禁止"因为没批量方法就退化为循环单查" | 单个对象的查询不要硬凑批量 |
| **并行优先** | 同阶段 ≥2 个互不依赖查询必须并行 | 同阶段**只有 1 个**查询 → **禁止**套并行编排器（徒增协程池开销与复杂度），直接调用 |
| **单次往返优先** | 能命中本地则 0 往返，否则合并成单次 RTT | 已被本地缓存命中的读，禁止再合并 Pipeline |

> **判定要点**："B 依赖 A 的结果"属**不同阶段**，串行正确；但该阶段内若仍有 2+ 独立查询，仍须并行。详见 [`docs-spec/25 §2.2`](../docs-spec/25_io_aggregation_spec.md) 真实依赖链豁免。

### 4.2 升级路径（从简到繁，每步要有触发理由）

| 场景 | 默认选型 | 升级触发 → 升级到 |
| --- | --- | --- |
| 集合 enrich | 一次批量读 + map 装配 | 关联多源 → 每源批量读后合并；源带缓存 → 聚合器 |
| 多个独立读 | 扇出并行（errgroup） | 读项有缓存兜底 / 需单飞合并 → 聚合器 |
| 跨实例读 | 按实例分组批量读 | 实例多 + 热点 → 聚合器 + 异步写回 |
| 高频状态写 | 直接写 | 写风暴 → Redis-first + delta 增量 flush |
| 纯追加写 | 直接批量 INSERT | 量极大 / 需隔离故障域 → MQ 批量消费；大批量装载 → `LOAD DATA`/`COPY` |
| 插入/更新混合 | select-then-write（先查有无再插/改） | → 批量 Upsert（`ON DUPLICATE KEY`/`ON CONFLICT`）一次往返 |
| 扇出 RPC | 下游批量接口 | 下游无批量接口 → 并行扇出（worker pool 限流） |
| 关联强依赖 + 同库可索引 | 单条 JOIN（≤ `{JOIN-limit}`） | 跨库 / 跨实例 / 弱依赖 → 聚合器并行多查询 |
| 高频独立点查 | 逐个点查（各自 GET） | 同类资源高并发、不同 key → 跨请求微批加载器攒批一次 MGET（§3.1） |
| 链路时延 | 各跳固定超时 | 多跳 + 端到端 SLA → 截止预算逐跳递减 + 耗尽降载（§3.3） |
| 扇出失败 | all-or-nothing（errgroup firstErr） | 含可降级依赖 → best-effort partial 收集（§3.4） |
| 数据密集算子（过滤/聚合/计数/排序） | 下推 DB（走索引，§3.5） | 跨库不可下推 → 应用层但先批量取最小列集 |
| 巨型写（百万行 UPDATE/DELETE） | keyset 分块小事务（§3.5） | 在线带流量 → 限流分批 + 监控复制延迟（[`11 §3.3`](11_evolution_migration_design.md)） |

> **反向纪律（写进规范，防教条化）**：性能纪律必须同时规定"何时不做"。"单查询禁套编排器""本地命中禁合并 Pipeline"与正向铁律同等重要——否则纪律被滥用会产生反向技术债。

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 编排方案权衡矩阵

| 方案 | 往返次数 | 并行度 | 复杂度 | 适用形态 | 主要代价 |
| --- | --- | --- | --- | --- | --- |
| 批量读 + map 装配 | 1 | — | 低 | 集合 enrich | 需先收齐 key |
| 扇出并行（errgroup） | N（并行） | 高 | 中 | 多个无缓存独立读 | goroutine 上限需管控（03） |
| 聚合器（Aggregator） | 按实例分组（少） | 高 | 高 | 多缓存兜底 + 少量直接任务 | Slot 只读、回源单飞 |
| 单条 JOIN | 1 | — | 低 | 强依赖同库可索引 | 跨库不可用、JOIN 表数受限 |
| Redis-first + delta flush | 热路径 0 落库 | — | 中 | 高频状态写 | 最终一致 + 边界兜底（04） |
| MQ 批量消费 | 每批 1 | — | 中 | 纯追加写 | 引入延迟窗口、需重试/死信 |
| 跨请求微批加载器 | 每批 1 | 高（攒批） | 中 | 高并发独立点查（不同 key） | 引入微小批窗延迟 |

### 5.2 借纪律不借实现：按数据形态重选聚合原语（强制）

同一条"把扇出收敛成一次往返"的纪律，**原语随数据形态变**（[`00 §2.4`](00_design_overview.md)）：

| 数据形态 | 推荐聚合原语 | 依据 |
| --- | --- | --- |
| 同质大对象（统一 `{表}:{id}` 编码、同实例） | **MGET**（一次取回 N 个） | 同质可一次 MGET；可叠压缩 |
| 异构小字段（Hash / Set / String 混合） | **Pipeline**（一次 RTT 合并多命令） | MGET 跨不了类型 |
| 跨抽象边界的多模块查询 | 声明式批量预取编排器 | 统一 key 命名空间 → 不同模块查询折叠成单次 MGET |
| 简单并发、无需跨边界合并 | 有界协程池 fan-out | 简单直接 |
| 高并发独立点查（不同 key、跨请求） | 跨请求微批加载器（攒批窗 → 一次 MGET） | 单飞只合并同 key，微批合并不同 key |

> **强制纪律**：复用本方法论时照搬纪律（三维度 / 铁律 / 判定标准），按自己的数据形态重选原语；把同质 MGET 硬搬到异构场景（或反之）会直接失效。各原语的硬约束（worker pool 上限、Slot 只读、channel 背压）真相源见 [`docs-spec/25 §3-§4`](../docs-spec/25_io_aggregation_spec.md) / [`20 §1.1`](../docs-spec/20_concurrency_safety_spec.md) / [`22 §6.1`](../docs-spec/22_performance_contract_spec.md)。

## 6. 反模式速查 + 机械闸门（引用不复制）

### 6.1 反模式速查

规则正文与豁免登记真相源在 [`docs-spec/25 §2`](../docs-spec/25_io_aggregation_spec.md)，本表只做"选型阶段速查 + 正确方向"：

| 反模式 | grep 锚 | 正确方向（详见 25） |
| --- | --- | --- |
| 循环内单条查询（N+1） | `R-IO-N-PLUS-1` | 循环外批量 + map 装配 |
| 独立读逐个串行 await | `R-IO-SERIAL-ORCH` | 聚合器 / 扇出并行 |
| 循环内单条写 | `R-IO-LOOP-WRITE` | 收集后批量写 / pipeline |
| 跨实例读未按实例分组 | `R-IO-NO-AGGREGATOR` | 按实例分组批量读 |
| 循环内单条 RPC | `R-IO-LOOP-RPC` | 批量接口 / 并行扇出 |
| 单查询套并行编排器 | （选型审查） | 去掉编排器，直接调用 |
| 本地命中的读被合并 Pipeline | （选型审查） | 命中即返回，不合并 |
| 裸提交协程池（漏配 WaitGroup） | `R-IO-RAW-SUBMIT` | 走"失败内联兜底"统一提交入口（见 03） |
| select 判有无再分别 insert/update | （选型审查） | 批量 Upsert（`ON DUPLICATE KEY`/`ON CONFLICT`）一次往返 |
| 大批量装载用逐行 / 多行 INSERT | （选型审查） | `LOAD DATA`/`COPY` 批量装载（快约 10-100×） |
| 超总预算后仍向下游发起新 IO（放大型无效工作） | （选型审查） | 截止预算逐跳递减，耗尽即 fast-fail / 降载（§3.3） |
| 高并发独立点查逐个 GET（可攒批却没攒） | （选型审查） | 跨请求微批加载器攒批一次 MGET（§3.1） |
| 可降级分支失败却整体失败（一败俱败） | （选型审查） | best-effort partial 收集（§3.4） |
| 关键分支失败却返回部分脏结果 | （选型审查） | 关键分支 all-or-nothing 闸门（§3.4） |
| SELECT 全量捞到应用层再 filter/sum/count | （选型审查） | 计算下推 DB（WHERE/GROUP BY/COUNT/LIMIT 走索引，§3.5） |
| 巨型写一条大事务（顶爆复制延迟 + 长持锁） | （选型审查） | keyset 分块小事务（§3.5 / [`02 §3.6`](02_data_model_design.md)） |

### 6.2 机械闸门下沉（L0）

可机械判定的 IO 违规必须下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）：N+1 检测需做**污点传播**（捕获"循环内拼接 / 派生 key → 调单 key 查询"，否则 `"{prefix}:" + r.X` 拼出的 key 全漏过）；裸提交检测 grep 绕过统一提交器的调用。命中即硬拦截，把修复要求回灌生成方。语义类违规（铁律二 / 文档同步）做软提醒。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节的系统名、数值、代码均为某真实工程的具体实现，**只借纪律不借实现**——落地工程时按自己的数据形态重选原语与参数。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 "同纪律不同实现"对照（借纪律不借实现的实证）

两个真实系统共享同一套 IO 纪律，却因数据形态不同选了不同原语：

| 维度 | 系统 A（读密集网关） | 系统 B（鉴权 / 计费） | 选择依据 |
| --- | --- | --- | --- |
| 聚合原语 | MGET | Pipeline | A 缓存同质 `表:id` JSON；B 热数据异构 Hash/Set/String |
| 并行 fan-out | 声明式批量预取编排器 | 有界协程池（100 worker） | A 需跨边界自动合并 MGET，B 只需简单并发 |
| 防击穿 | singleflight | 热路径不回源（miss=拒绝） | 数据源定位不同 |
| 数据编码 | JSON + LZ4 压缩 | 原始值无压缩 | A 缓存大对象，B 缓存小字段 |
| Redis 角色 | 纯缓存（miss 回源 DB） | 核心数据源（miss=拒绝） | 决定降级方向（见 05） |

**共同点（真正可迁移的纪律）**：三维度、铁律、判定标准、机械闸门、自适应熔断、缓存访问层、TTL 抖动、连接池基线、分层依赖、数据层禁日志——**全部一致；不同的只是按数据形态重选的原语。**

### 9.2 批量预取编排器三阶段（系统 A）

声明式三步：`Preload`（注册带缓存查询，参与跨 Repo MGET 合并）→ `Go`（注册无缓存并行任务）→ `Fetch`（一次性执行，只调一次）→ `Slot.Value()`（nil-safe 读结果）。执行机制：

```
Phase 1 mgetPhase   ：按 Redis 实例指针分组所有 Preload → 每组 1 次 MGET
                       → 命中 LZ4 解码 + 反序列化进 slot；未命中收集为 missedEntries
Phase 2 parallelPhase：missed + Go 任务全部提交协程池并行 → 同 key 用 singleflight 折叠成 1 次 DB
                       → WaitGroup 同步 + 聚合 firstErr → Fetch 返回（不等回写）
Phase 3 writeBack    ：ctx.Copy() 脱离请求生命周期 → 按实例分组 MSet（一次 RTT）→ 写回池满即静默丢弃
```

关键点：① 统一 `表:id` key 命名空间 → 不同模块查询折叠成单次 MGET；② 单次 MGET key 数 > `64` 自动 Pipeline 分批（Redis 单线程下单命令 key 过多会阻塞，64 足够小保亚毫秒、又覆盖绝大多数场景）；③ singleflight 的 waiter 拿同一指针 → **下游禁止就地改字段**（只读契约）；④ 写回用 `ctx.Copy()` + 派生超时，移出响应关键路径。**实测**：某 KYC 接口 16~18 次查询 → "1 次 MGET + 并行 DB"，延迟降 80%+。

### 9.3 读路径消灭重复读（系统 B 鉴权）

```go
// ❌ 同一个 key 读两次
remain := loadRemainQuota(key)      // HGETALL quota:{key}  第 1 次
mode   := determineBillingMode(key) // HGETALL quota:{key}  第 2 次（重复！）
// ✅ 读一次，派生值作副产物回传
mode, remainQuota := determineBillingMode(key)  // 只 HGETALL 一次，total-used 顺带返回
// 限流 INCR 的返回值即"当日已用量"，回传给配额判定，消除二次 GET
```

多级缓存：L1 进程内存 struct（`HGETALL` 一次 → 强类型 struct，后续 0 序列化 0 网络，只缓存"万级 + 低频变更"key、TTL≤15min 自然失效）；L2 Redis（miss 直接判"不存在"不穿透 DB，完整性靠预热 + 定时巡检）。鉴权 11 步校验链稳态**热路径零 Pipeline、纯串行**——有意为之。

### 9.4 写路径削峰（系统 B 计费）

```
有中间态（配额扣减）：热路径 HINCRBY used+1 + HINCRBY delta+1 + SADD dirty   // O(1) 原子无锁
                      定时每 5min：SPopN(dirty,100) → 按 expire_at ASC 增量 UPDATE 落库 → 一致性校验
                      不变量：Redis.used = DB.SUM(used) + Redis.delta
纯追加（调用流水）：  投 Kafka（key=appKey 保序）→ 消费端攒批（200 条或 5s）→ 按月分表 CreateInBatches
                      → 重试 3 次指数退避 → 死信队列
```

`SPOP` 弹出即消费的原子性 → 多实例并行 flush 不重复、无需分布式锁。超扣边界兜底见 [`04 §参考示例`](04_transaction_design.md)（`GREATEST`/`LEAST`）。

### 9.5 批量写极致：Upsert + 批量装载

```sql
-- 批量 Upsert：一次往返完成"有则更新无则插"，免 select-then-write
INSERT INTO t (k, v) VALUES (?,?),(?,?),…
  ON DUPLICATE KEY UPDATE v = VALUES(v);            -- MySQL
  -- ON CONFLICT (k) DO UPDATE SET v = EXCLUDED.v;  -- PostgreSQL
-- 大批量装载：冷启 / 迁移 / 离线导入，比多行 INSERT 快约 10-100×
LOAD DATA LOCAL INFILE 'f.csv' INTO TABLE t …;      -- MySQL
-- COPY t FROM STDIN …;                              -- PostgreSQL
```

### 9.6 截止时间预算逐跳递减

```go
// 入口按端到端 SLA 设 deadline；每跳传【剩余预算】而非各自固定超时
ctx, cancel := context.WithTimeout(req.Context(), 100*time.Millisecond); defer cancel()
// 下游调用前：剩余预算不足以覆盖本跳 → 关键依赖 fast-fail / 非关键依赖跳过
if dl, ok := ctx.Deadline(); ok && time.Until(dl) < estCost { return fallbackOrSkip() }
callDownstream(ctx, ...)   // ctx 透传，下游 select{<-ctx.Done()} 同步感知剩余预算
```

### 9.7 跨请求微批加载器

```
高并发下不同请求各自 GET user:{id} → 攒批窗（如 1ms 或满 64 个）合并成一次 MGET：
  loader.Load(id) ──► 投入当前批 ──► 批窗到期 / 满 ──► 一次 MGET → 按 id 分发结果给各 waiter
单飞（合并同 key）与微批（合并不同 key）正交叠加；点查稀疏 / 低并发时关掉微批（批窗延迟得不偿失）。
```

### 9.8 扇出 best-effort partial

```go
// 可降级依赖：失败不污染整体；关键依赖才进 errgroup（一败俱败）
results := make([]*Part, len(deps)); var wg sync.WaitGroup
for i, d := range deps { i, d := i, d; wg.Add(1)
    Go(pool, func(){ defer wg.Done(); if v, err := d.Fetch(ctx); err == nil { results[i] = v } }) // 失败留 nil → 标记降级
}
wg.Wait()   // 返回已成功分支的部分结果；失败分支按 06 差异化降级返空 / 默认
```
