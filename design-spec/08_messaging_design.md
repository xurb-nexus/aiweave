# 08 - 异步消息 / 事件驱动设计决策规范

> 回答"面对一个需求，怎么判断它该不该从请求生命周期里**解耦**出去走异步、选对投递语义 / 顺序 / 分区、把双写一致与消费幂等做对"。这是 [`00 §2.3`](00_design_overview.md) 读写分治的**第三面**——前六视角治"请求内怎么快"，本视角治"哪些工作根本不该留在请求内"。
>
> **边界先声明**：消息结构 / 攒批落库 / offset 提交语义 / 死信队列 / Producer 侧 / helpers 集成等**表示规则的真相源在 [`docs-spec/11`](../docs-spec/11_mq_consumer_spec.md)**；写路径削峰编排在 [`01 §3.2`](01_io_design.md)、消费池容量与排空在 [`03 §3.4`](03_concurrency_design.md) + [`docs-spec/27 §5`](../docs-spec/27_deployment_runtime_spec.md)、Outbox / 本地消息表的事务边界在 [`04`](04_transaction_design.md) + [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)，本篇只引用不复制。本篇新增决策资产：**同步 / 异步解耦判定、投递语义决策树、顺序与分区键选型、双写一致（Outbox / CDC）路径、消费幂等、消费者扩展与背压、扇出模式（工作队列 vs 发布订阅）**。

---

## 1. 设计目标

- **解耦得当**：能移出请求关键路径的工作（通知、流水、派生计算、跨服务事件）异步化，与请求生命周期物理解耦；但**强一致读后写、需即时反馈的写**不强行异步（异步不是银弹，引入投递语义 / 顺序 / 重复 / 延迟四类新复杂度）。
- **投递语义恰到好处**：按"丢一条的代价"选 at-most / at-least / exactly-once，**不为不在乎丢失的场景付 exactly-once 的复杂度**，也不为对账类场景图省事用 at-most-once。
- **双写不撕裂**：写库 + 发消息这对"双写"不靠"先写库再发消息"裸拼（中间崩溃即丢事件），用 Outbox / CDC 把"发消息"绑进"写库"的同一份原子性里。
- **消费幂等 + 可扩展 + 有背压**：at-least-once 下消费端必幂等；消费并行受分区数约束；积压（lag）有可观测与背压，不无限堆积。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 可异步的副作用 | "下单后**还要**发通知 / 加积分 / 同步索引" | 解耦判定 |
| 纯追加流水 | "持续产生 `{流水 / 事件}`，只追加、量极大" | 写路径削峰→MQ（[`01 §3.2`](01_io_design.md)） |
| 跨服务事件广播 | "`{实体}` 状态变了，多个下游 / 团队要各自响应" | 发布订阅扇出 |
| 双写一致 | "既要写 `{库}`、又要发 `{事件}`，不能只成一半" | Outbox / CDC |
| 削峰填谷 | "突发写流量远超下游 / DB 持续承载，要缓冲" | 工作队列 + 攒批 |
| 重复投递风险 | "下游 / MQ 可能重复投递，消费要幂等" | 消费幂等 |
| 顺序敏感 | "同一 `{key}` 的事件必须按发生序处理" | 分区有序 |
| 积压治理 | "消费跟不上生产、lag 持续涨" | 消费者扩展 + 背压 |
| 异构数据同步 | "把 OLTP 变更实时同步到检索 / 数仓 / 缓存" | CDC（联动 [`02`](02_data_model_design.md)） |

> 识别口诀：**"做完主操作之后还要做 X、且 X 晚一点做也行"→ 解耦走异步；"既写库又要发事件"→ Outbox / CDC；"同一 key 必须有序"→ 分区键有序。**

## 3. 决策树（强制 / 本篇核心）

### 3.0 先分治：这个工作该留在请求内，还是解耦出去？

异步化与 [`00 §2.3`](00_design_overview.md) 读写分治同源——它是"把工作移出请求生命周期"的决策。**先判该不该解耦，再谈用什么消息原语：**

```
请求里的某段工作 ──► 它必须在响应返回前完成吗？
│
├─ 是（强一致读后写 / 需即时反馈 / 失败要让调用方知道） ──► 留在请求内（走 01/04 同步路径，别异步）
└─ 否（副作用 / 派生 / 通知 / 流水 / 跨服务事件） ──────► 解耦异步（进入 §3.1 投递语义树）
        ↳ 解耦带来的延迟窗口业务可接受？ 否 → 退回同步或预计算
        ↳ 解耦后失败如何让上游感知？ → 对账 / 状态回查 / 死信告警（不再有同步返回值）
```

> **反向纪律**：异步引入投递语义 / 顺序 / 重复 / 延迟四类新复杂度。**"能同步说清的别异步"**——只为"可延后 + 可解耦 + 有削峰 / 广播 / 故障隔离收益"的工作异步化，不把同步链路硬拆成消息只为"看起来解耦"。

### 3.1 投递语义决策树（按"丢一条的代价"选）

```
这条消息丢了 / 重复了，代价多大？
│
├─ 丢了无所谓（可重采、纯监控埋点） ──────► at-most-once：发后不管，不重试（最省）
│
├─ 不能丢、但能容忍重复（绝大多数业务） ──► at-least-once（默认）＋ 消费端幂等（§3.4）
│        ↳ 这是默认选型：MQ 重试 / 消费重启天然产生重复，幂等是 at-least-once 的孪生纪律
│
└─ 既不能丢也不能重复（精确对账 / 扣费） ─► exactly-once：代价最高，两条实现路径择一
         ├─ 生产侧：Outbox / 事务消息把"写库 + 发消息"绑成原子（§3.3）
         └─ 消费侧：at-least-once + 幂等去重表（消费 + 提交 offset 在同一事务）≈ effectively-once
```

> **exactly-once 多是"at-least-once + 幂等"的工程等价**：真正端到端 exactly-once 跨网络代价极高；生产实践用 **"不丢（Outbox）+ 不重复地生效（消费幂等）"** 拼出 *effectively-once*。先问"能不能容忍重复"——能，就 at-least-once + 幂等，别上重机制。

### 3.2 顺序与分区键决策树（顺序是吞吐的对立面）

```
这批消息的顺序要求是什么？
│
├─ 全局有序（所有消息一条线） ──► 单分区：牺牲并行（吞吐 = 单消费者），仅在真需全局序时用
├─ 按 {key} 局部有序（同实体内有序、跨实体无所谓） ──► 按 {分区键} 哈希分区（默认）
│        ↳ 分区键选"同一实体的事件必落同分区"的字段（如 {entity-id} / {account}）
│        ↳ 分区数 = 消费并行上限：分区键基数要远大于分区数，避免热分区（与 05 热 key 同构）
└─ 无序（互相独立） ──────────► 不绑分区键，最大并行
```

> **顺序与并行是此消彼长**：全局有序 = 单点串行；要并行就接受"只在分区内有序"。**分区键选错 = 要么丢序（同实体散落多分区）、要么热分区（高频实体挤一个分区，加分区也分不走，与 [`05 §3.6`](05_caching_design.md) 热 key 同构）。**

### 3.3 双写一致：Outbox / CDC（消灭"写库成功、发消息失败"的撕裂）

```
一个操作既要写 {库}、又要发 {消息}，怎么保证不只成一半？
│
├─ ❌ 裸双写（先写库再发 MQ） ──► 两步之间崩溃 → 库已改、消息没发 → 永久丢事件（反模式）
│
├─ Outbox / 本地消息表 ──► 把"待发消息"和业务数据写进【同一个本地事务】→ 事务成功即两者都落
│        ↳ 旁路投递器轮询 outbox 表 → 发 MQ → 标记已发（投递器崩溃可重发 → at-least-once）
│        ↳ 事务边界与补偿登记走 04 / docs-spec/21；这是 04 "跨数据源最终一致"的标准实现之一
│
└─ CDC（订阅 binlog / WAL） ──► 不改业务代码，捕获库变更流 → 转成事件投递（零侵入）
         ↳ 适合"把 OLTP 变更同步到检索 / 数仓 / 缓存 / 下游"（联动 02 读副本 / OLTP-OLAP 分离）
         ↳ 代价：引入 CDC 管道与 schema 演进耦合（表结构变更要顾及下游解析）
```

> **Outbox vs CDC**：Outbox 在**应用层**显式造事件（语义清晰、与业务解耦度低）；CDC 在**存储层**捕获变更（零侵入、但事件是"行变更"非"业务语义"，下游要自己翻译）。双写一致的事务模型选型真相源在 [`04 §3.1`](04_transaction_design.md) / [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)，本篇只给"何时该用 Outbox / CDC"的判定。

### 3.4 消费幂等（at-least-once 的孪生纪律 / 强制）

```
消费端怎么保证"重复投递不重复生效"？
│
├─ 写操作天然幂等（SET / 覆盖式更新 / 状态机单向跃迁） ──► 直接消费，无需额外去重
├─ 写操作非幂等（累加 / 插入 / 扣减） ──► 幂等键去重：以 {消息唯一 ID / 业务键} 查去重表 / 唯一索引
│        ↳ "去重写入 + 业务写入 + offset 提交"尽量同一事务，避免"已生效未记录→重复生效"
└─ 可在架构层规避 ──► 读写分离消幂等（04 §3.2）：把"判定"与"生效"拆开，重复判定天然安全
```

> 幂等键 / 去重表的规则正文见 [`04`](04_transaction_design.md) / [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)；本篇强调的是**"消费幂等不是可选项，是 at-least-once 的强制配套"**——选了 §3.1 的默认 at-least-once，就必须同时选一种 §3.4 的幂等手段。

### 3.5 消费者扩展、背压与失败处理

```
消费跟不上 / 消费失败了怎么办？
│
├─ 扩展 ──► 消费并行上限 = 分区数；加消费者超过分区数无效 → 先加分区（且分区键基数要够）
│           └─ 消费池容量按 Little 定律（03 §3.2）；消费池独立成舱（03 §3.4 下游归属维度）
├─ 背压 ──► 攒批落库 + 有界 in-flight；下游慢 → 放慢拉取（暂停 poll），不无界堆内存
│           └─ lag 持续涨是过载信号：触发告警 / 扩分区 / 降生产速率（与 07 load shedding 互补）
└─ 失败 ──► 重试（有限次 + backoff + jitter，预算见 07 §3.5）→ 仍失败投死信队列(DLQ) → 告警 + 人工/补偿
            └─ poison message（永远处理失败的毒丸）必须能进 DLQ，不能无限重试卡住整个分区
```

> 攒批策略 / offset 提交语义 / DLQ 结构的表示真相源在 [`docs-spec/11 §3.2.2 消费流程 / §3.2.3 失败处理`](../docs-spec/11_mq_consumer_spec.md)；消费池排空（关闭前先提交位点）在 [`docs-spec/27 §5`](../docs-spec/27_deployment_runtime_spec.md) / [`03 §3.4`](03_concurrency_design.md)；重试预算与 backoff jitter 在 [`07 §3.5`](07_tail_latency_design.md)。

### 3.6 扇出模式：工作队列 vs 发布订阅

```
一条消息要被几方消费？
│
├─ 一条消息只需一方处理（任务分发） ──► 工作队列（竞争消费，consumer group 内分摊分区）
└─ 一条消息每个订阅方各要一份（事件广播） ──► 发布订阅（每订阅者独立 offset / 独立 consumer group）
         ↳ 新增下游订阅者不影响既有订阅者（解耦）；事件 schema 要前向兼容（§3.7 / 09 契约演进）
```

### 3.7 消息 schema 演进（生产者与消费者解耦的前提）

```
改消息结构会不会打挂存量消费者？
│
├─ 加可选字段 ──► 前向兼容（老消费者忽略新字段），安全
├─ 删 / 改字段语义 ──► 破坏性：必须带版本号灰度（新老并存一个迁移窗口，09 expand-contract 同构）
└─ 序列化格式 ──► 选带 schema 演进能力的编码（如 Avro / Protobuf），别用裸 JSON 硬解析位置
```

> 消息 schema 演进与 [`09 §3`](09_api_contract_design.md) API 契约的 expand-contract、[`11 §3`](11_evolution_migration_design.md) 数据演进**同构**：都是"破坏性变更必须新老并存一个窗口"。协议版本兼容的表示规则见 [`docs-spec/11 §3.2.1 消息结构`](../docs-spec/11_mq_consumer_spec.md)。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 副作用 / 派生工作 | 留同步（简单直接） | 可延后 + 有削峰 / 广播 / 隔离收益 → 解耦异步（§3.0） | 强一致读后写 → **禁异步** |
| 投递语义 | at-least-once + 消费幂等 | 丢了无所谓 → at-most-once；不可重复 → Outbox + 幂等去重 | 不在乎重复 → **别上 exactly-once** |
| 顺序 | 按 {key} 分区有序 | 真需全局序 → 单分区（牺牲并行） | 互相独立 → 不绑分区键（最大并行） |
| 双写一致 | Outbox / 本地消息表 | 零侵入同步存量变更 → CDC | 单数据源写 → 无需 Outbox（本地事务即可） |
| 消费扩展 | 消费者数 ≤ 分区数 | lag 涨 → 先扩分区（保分区键基数）再加消费者 | 分区已足 → 加消费者无效 |
| 消费失败 | 有限重试 + backoff + jitter | 仍失败 → DLQ + 告警 + 补偿 | 毒丸 → 别无限重试（卡死分区） |
| 扇出 | 工作队列（竞争消费） | 多下游各要一份 → 发布订阅（独立 group） | 单方处理 → 别广播 |
| 消息 schema | 加可选字段（前向兼容） | 破坏性变更 → 版本号灰度（新老并存窗口，§3.7） | 别原地改字段语义 |

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 投递语义 / 一致性手段权衡矩阵

| 方案 | 不丢 | 不重复 | 吞吐 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| at-most-once | ✗ | ✓ | 最高 | 最低 | 可丢的监控 / 埋点 |
| at-least-once + 幂等 | ✓ | ✓（消费侧） | 高 | 中 | **默认**：绝大多数业务事件 |
| Outbox / 本地消息表 | ✓ | 重发可能重复→配幂等 | 中 | 中 | 写库 + 发事件的双写一致 |
| CDC 订阅 binlog | ✓ | 重放可能重复→配幂等 | 高 | 高（管道 + schema 耦合） | 零侵入同步到下游 / 数仓 / 缓存 |
| 单分区全局有序 | ✓ | — | 低（串行） | 低 | 真需全局序 |
| 发布订阅广播 | ✓ | — | 高 | 中 | 一事件多下游 |

### 5.2 借纪律不借实现：按"丢一条的代价 + 顺序需求"重选原语（强制）

同一条"把工作可靠地解耦出去"的纪律，**原语随场景变**（[`00 §2.4`](00_design_overview.md)）：可丢监控 → at-most-once；业务事件 → at-least-once + 幂等；扣费对账 → Outbox + 去重表；异构同步 → CDC。把"对账级 exactly-once"硬套到"可丢埋点"是过度设计，把"at-most-once"硬套到"扣费"是欠设计。**可迁移的是"按代价选语义、at-least-once 必配幂等、双写必走 Outbox / CDC、顺序与并行权衡"这套纪律，不是某个 MQ（Kafka / RocketMQ / Pulsar）的具体 API。**

## 6. 反模式速查 + 机械闸门（引用不复制）

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向（详见 docs-spec/11 / 04） |
| --- | --- | --- |
| 裸双写（先写库再发 MQ，中间崩溃丢事件） | `R-MQ-DUAL-WRITE` | Outbox / 本地消息表绑同一事务（§3.3） |
| at-least-once 消费端无幂等（重复生效） | `R-MQ-NO-IDEMPOTENT` | 幂等键去重 / 架构层读写分离（§3.4） |
| 同一实体事件散落多分区（丢序） | （选型审查） | 分区键选"同实体必同分区"的字段（§3.2） |
| 高频实体挤一个热分区（加分区分不走） | （选型审查） | 分区键基数 ≫ 分区数（§3.2，同 05 热 key） |
| 消费失败无限重试（毒丸卡死分区） | `R-MQ-NO-DLQ` | 有限重试 + DLQ + 告警（§3.5） |
| 消费无界堆积（lag 涨到 OOM） | `R-MQ-NO-BACKPRESSURE` | 攒批 + 有界 in-flight + 暂停 poll（§3.5） |
| 消费者数 > 分区数（空转无效扩展） | （选型审查） | 先扩分区再加消费者（§3.5） |
| 能同步说清的硬拆成消息（伪解耦） | （选型审查） | 强一致 / 即时反馈留同步（§3.0） |
| 消息结构原地改字段语义（打挂存量消费者） | （选型审查） | 版本号灰度 + 前向兼容编码（§3.7） |
| 消费者关闭不先提交 offset（重复消费一大批） | `R-SHUTDOWN-LEAK-GOROUTINE` | 排空 + 先提交位点（27 §5 / 03 §3.4） |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）：**裸双写检测**（同一函数体内"DB 写"与"MQ 发送"相邻且不在 outbox 事务里 → 污点提示 `R-MQ-DUAL-WRITE`）；**消费入口缺幂等**（消费 handler 含非幂等写却 grep 不到去重键 → `R-MQ-NO-IDEMPOTENT` 软提醒）；**无界消费缓冲**（`make(chan T)` 无界 / 无背压 → 复用 `R-PERF-UNBOUNDED-QUEUE`）。"投递语义是否选对""分区键基数够不够"等语义类违规走选型审查（软提醒）。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节系统名 / 数值 / 代码为某真实工程实现，**只借纪律不借实现**——落地工程时按自己的"丢一条代价 + 顺序需求"重选语义与原语。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 纯追加流水走 MQ（系统 B 计费流水）

```
纯追加（调用流水，无中间态）：投 Kafka（key=appKey 保序）→ 消费端攒批（200 条或 5s）
                            → 按月分表 CreateInBatches → 重试 3 次指数退避 → 死信队列
投递语义：at-least-once（消费幂等靠"流水唯一 ID 唯一索引"去重，重复插入被唯一键挡下）
分区键：appKey → 同一应用的流水落同分区、按序消费；appKey 基数远大于分区数，无热分区
```
与 [`01 §9.4`](01_io_design.md) 写路径削峰同源——本视角补的是"为什么是 at-least-once + 唯一键幂等、为什么分区键选 appKey"的决策。

### 9.2 Outbox 消灭双写撕裂

```
❌ 裸双写：tx{ UPDATE order } ; commit ; mq.Send(event)   // commit 后崩溃 → 事件永久丢
✅ Outbox：tx{ UPDATE order ; INSERT outbox(event) } ; commit // 业务 + 事件同一事务，要么都成
            旁路投递器：轮询 outbox WHERE sent=0 → mq.Send → UPDATE sent=1（崩溃可重发→消费幂等兜重复）
```

### 9.3 CDC 同步到检索 / 缓存（零侵入）

```
binlog → CDC（如 Debezium / Canal）→ 变更事件流 → 消费端更新 ES 索引 / 失效缓存
适用：把 OLTP 主库变更实时同步到检索引擎 / 数仓 / 缓存，业务代码零改动（联动 02 OLTP-OLAP 分离）
代价：表结构变更要顾及下游解析；事件是"行变更"非业务语义，下游自行翻译
```

### 9.4 消费幂等的三种形态

```
状态机单向跃迁：UPDATE order SET status='paid' WHERE id=? AND status='pending'  // 重复执行影响 0 行，天然幂等
唯一键去重：    INSERT INTO ledger(msg_id, ...) ...  // msg_id 唯一索引，重复投递被挡
读写分离消幂等：Verify（只读判定）+ Report（写生效）拆两步 → 重复 Verify 天然安全（04 §3.2）
```

### 9.5 顺序 / 分区 / 背压速记

```
顺序：同一 {entity-id} 的事件 → hash 到同分区 → 分区内 FIFO；跨实体无序、可并行
扩展：分区数 = 消费并行上限；lag 涨 → 先扩分区（且 entity-id 基数 ≫ 分区数）再加消费者
背压：攒批 + 有界 in-flight；下游慢 → 暂停 poll，不无界堆内存；lag 持续涨 → 告警（与 07 load shedding 互补）
失败：有限重试（backoff + jitter，07 §3.5）→ DLQ → 告警 + 补偿；毒丸必须能进 DLQ，不卡死分区
```
