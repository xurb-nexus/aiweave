# 11 - 演进与在线迁移设计决策规范

> 回答"当 schema / 数据 / 行为要在**带负载、不停机**的前提下演进时，怎么判形态、选迁移路径"。
>
> **核心命题**：前面的视角多在"一个时刻的截面"上做对设计；本视角治的是**时间轴上的安全变更**——线上有流量、有存量数据、有存量调用方时，怎么把"改表结构 / 迁数据 / 换实现"做成零停机、可灰度、可回滚。**变更期的系统同时跑着新旧两套,这正是它独立成视角的根因。**
>
> **边界先声明**：字段演进安全路径（默认值 / 类型兼容 / 废弃登记）、字段语义变更记录等**表示规则的真相源在 [`docs-spec/05 §8`](../docs-spec/05_schema_design_spec.md)**；增量同步的全链路文档变更契约在 [`docs-spec/19`](../docs-spec/19_incremental_sync_spec.md)；安全重构方法论 / 灰度切流量在 `mvp_rebuild_path.md §11`；迁移期一致性窗口在 [`04`](04_transaction_design.md) + [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)，本篇只引用不复制。本篇新增决策资产：**扩展-收缩（expand-contract / 并行变更）判定、在线 DDL 选型、限流回填（backfill）、双写双读灰度切换、行为变更影子比对、迁移回滚闸门**。本视角无独立 docs-spec，按双源真相**分布落地**到既有表示真相源（同 [`07`](07_tail_latency_design.md)）。

---

## 1. 设计目标

- **零停机演进**：schema / 数据 / 实现的变更不靠"停服窗口"完成；线上持续服务，变更在后台分阶段推进。
- **任一时刻新旧并存且自洽**：迁移过程中系统同时跑新旧两套，**任一中间状态都必须正确**（不能依赖"瞬间全切换"）——这是 expand-contract 的第一性约束。
- **可灰度可回滚**：每一步都能小流量验证、出问题能退回上一步；破坏性变更前，旧路径始终保留到确认新路径稳。
- **回填不打挂线上**：存量数据回填是后台限流任务,绝不与在线流量抢资源、绝不一条大事务扫全表。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 改既有表 | "给 `{大表}` 加 / 改 / 删字段、改类型、加索引" | 在线 DDL + 字段演进 |
| 改字段语义 | "`{字段}` 的含义 / 单位 / 取值要变" | expand-contract |
| 拆 / 并表 | "把 `{表}` 拆分 / 合并 / 换主键 / 换分片键" | 双写双读迁移 |
| 存量回填 | "新字段 / 新表要把存量数据补齐" | 限流 backfill |
| 换存储 / 换实现 | "把 `{逻辑}` 从旧实现切到新实现 / 换底层存储" | 影子比对 + 灰度 |
| 换缓存结构 | "缓存 key 结构 / 编码要变,存量 key 怎么办" | 世代式切换（[`05 §3.7`](05_caching_design.md)） |
| 带不兼容变更发布 | "这次发布含不向后兼容的 schema / 协议变更" | 回滚闸门 |
| 大表加索引 | "给亿级表加索引,怕锁表" | 在线 DDL |

> 识别口诀：**"改既有线上结构 / 语义"→ 默认 expand-contract,绝不原地改;"补存量"→ 限流后台回填;"换实现"→ 影子比对 + 灰度切流量。**

## 3. 决策树（强制 / 本篇核心）

### 3.0 先判：这是兼容变更还是破坏性变更？

```
这次变更,老代码 / 老数据 / 老调用方还能正常工作吗?
│
├─ 能（纯增量:加可空字段 / 加表 / 加可选参数 / 加索引） ──► 兼容变更:可直接上(仍走在线 DDL,§3.2)
└─ 不能（改语义 / 删 / 改类型 / 改主键 / 换结构） ──────► 破坏性变更:**禁止原地改**,走 expand-contract(§3.1)
```

> **第一性纪律:破坏性变更永不"原地一步到位"**。原地改字段语义 / 删列 / 改类型,会在"代码已发新版、数据还是旧的"或反之的中间窗口里崩。所有破坏性变更都拆成"先扩展、再迁移、后收缩"三段,让任一中间态都自洽。

### 3.1 扩展-收缩（expand-contract / 并行变更）决策树

```
破坏性变更怎么拆成"任一中间态都自洽"的三段?
│
① expand（扩展） ──► 新增"新字段 / 新表 / 新接口"与旧并存;代码同时能读新旧、写时双写新旧
│                     ↳ 此阶段发布:老实例只认旧、新实例双写 → 新旧实例混跑也正确
│
② migrate（迁移） ──► 后台限流回填存量(§3.3) → 把读切到新(双读校验 → 灰度放量,§3.4)
│                     ↳ 监控新旧一致性;读流量逐步从旧迁到新(1% → 10% → 100%)
│
└─③ contract（收缩） ──► 确认无人读旧、新路径稳定 → 停双写 → 删旧字段 / 旧表 / 旧接口
                         ↳ 至少跨一个发版周期(留回滚余地);删除是不可逆的,最后做
```

> **expand-contract 是本支柱"破坏性变更必须新老并存一个窗口"纪律的数据层实例**——与 [`08 §3.7`](08_messaging_design.md) 消息 schema 演进、[`09 §3.3`](09_api_contract_design.md) API 版本兼容**完全同构**:三者都是"扩展并存 → 迁移灰度 → 收缩删旧"。字段演进安全路径(加字段必有默认值、类型兼容方向、废弃登记)的规则正文见 [`docs-spec/05 §8`](../docs-spec/05_schema_design_spec.md);安全重构灰度(Shadow read → 双写 → 1%/10%/50%/100%)见 `mvp_rebuild_path.md §11.5.4`。

### 3.2 在线 DDL 选型（大表改结构不锁表）

```
要给线上(大)表改结构,怎么不阻塞读写?
│
├─ 小表 / 低峰 / DBMS 原生支持 instant DDL ──► 原生 DDL(部分引擎加列 / 改默认值是 instant,毫秒级)
├─ 大表加索引 / 改列(原生会锁表或长时间持锁) ──► 在线 DDL 工具(影子表 + 触发器 / binlog 追平 + 原子改名)
│        ↳ 全程不长时间持锁;限速避免拖垮主库;切换瞬间原子 rename
├─ 加字段 ──────────────────────────────────► 必须有默认值(否则旧行读取 / 旧代码插入失败,05 §8)
└─ 删 / 改类型 ──────────────────────────────► 不直接做:走 expand-contract(§3.1),先停用再物理删
```

> 在线 DDL 的代价是"影子表期间双倍写放大 + binlog 追平延迟",须在低峰跑并限速。字段类型变更的兼容方向(`INT→BIGINT` 可、`BIGINT→INT` 不可)、删列废弃登记规则见 [`docs-spec/05 §8`](../docs-spec/05_schema_design_spec.md)。

### 3.3 限流回填（backfill）：后台、分批、不抢在线资源

```
存量数据要补新字段 / 灌新表,怎么填?
│
├─ ❌ 一条 UPDATE 扫全表 ──► 长事务持锁 + 顶爆复制延迟 + 锁等待雪崩(反模式)
├─ 分批回填 ──────────────► keyset 游标分块(02 §3.6)逐批处理,每批小事务提交
│        ↳ 限速:每批之间 sleep / 令牌桶,把回填 QPS 压在在线流量之下(夜间更宽)
│        ↳ 幂等可重入:记回填位点,中断可续(与 10 §3.2 同构);重复执行不重复写
└─ 监控反压 ──────────────► 盯主库负载 / 复制延迟,超阈值自动降速 / 暂停(与 07 自适应同构)
```

> 回填是典型的"大写事务必须分批"——与 [`04 §3.3`](04_transaction_design.md) group-commit(摊 fsync)互补:那是把小事务*合*,这是把巨型写*拆*;都为避开"一次性大事务"。分批游标走 [`02 §3.6`](02_data_model_design.md) keyset;限流降速思路与 [`07 §3.3`](07_tail_latency_design.md) 自适应同源。

### 3.4 行为变更:双读影子比对 + 灰度切流量

```
换实现 / 换存储,怎么确认新路径与旧等价再切?
│
① 影子读(shadow) ──► 线上同时走新旧两条路径,只用旧结果对外,后台比对新旧差异(不影响用户)
│                     ↳ 差异收敛到 0 → 才进下一步;差异是 bug 清单
② 灰度放量 ──────► 读流量按比例切新:1% → 10% → 50% → 100%,每档观测错误率 / 延迟
└─③ 旧路径退役 ──► 新路径全量稳定一个周期 → 删旧实现(expand-contract 的 contract 段)
```

> 影子比对 + 灰度的表示规则见 `mvp_rebuild_path.md §11.5.4`(行为变更必按 Shadow read → 双写 → 1%/10%/50%/100% 切流量);增量变更的全链路文档同步契约见 [`docs-spec/19`](../docs-spec/19_incremental_sync_spec.md)。

### 3.5 迁移回滚闸门（不兼容发布的安全网 / 强制）

```
含不兼容变更的发布要回滚,安全吗?
│
├─ 回滚前置条件 ──► 软兼容必须已就位:旧代码能读新写的数据 / 新代码能读旧数据(05 §8 双向兼容)
├─ 数据已收缩(删列/删表)则不可回滚 ──► contract 段一旦执行,旧版本无法启动 → 收缩前是最后回滚点
└─ 灰度未全量则可秒退 ──► 灰度阶段保留旧路径,出问题切回旧(流量比例归零),不动数据
```

> **回滚安全是 expand-contract 的副产品**:只要还在 expand / migrate 阶段(旧路径未删),回滚就是"流量切回旧";一旦 contract(删旧),回滚需数据反向迁移,代价陡增。所以"删除"永远是最后一步、跨发版周期后做。含不兼容 schema 变更的回滚前置(确认 `database_design.md §8` 软兼容就位)见部署规则。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 改既有结构 | 先判兼容 vs 破坏(§3.0) | 破坏性 → expand-contract 三段(§3.1) | 纯加可空字段 → 直接上,别套三段 |
| 大表改结构 | 原生 DDL(小表 / instant) | 大表加索引 / 改列 → 在线 DDL 工具(§3.2) | 小表低峰 → 别上重型在线 DDL |
| 加字段 | 带默认值直接加 | —(无默认值禁止,05 §8) | — |
| 存量补数 | 分批限流回填(§3.3) | 主库压力大 → 监控反压自动降速 | 小表 → 一批搞定,别过度分批 |
| 换实现 / 换存储 | 影子读比对 → 灰度(§3.4) | 差异收敛后逐档放量 | 等价小改 → 直接灰度,别都上影子 |
| 不兼容发布 | 保旧路径 + 灰度可秒退 | 已 contract 删旧 → 不可回滚(收缩前是最后回滚点) | 兼容发布 → 无需回滚闸门 |
| 缓存结构变更 | 世代式切换(05 §3.7) | key 编码变 → 新前缀并存 + 旧靠 TTL 自然回收 | — |

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 迁移路径权衡矩阵

| 路径 | 停机 | 回滚成本 | 一致性风险 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| 停服迁移 | 有 | 低 | 低 | 低 | 可接受停机窗口的小系统(多数不可接受) |
| expand-contract | 无 | 低(收缩前) | 低(新旧并存自洽) | 中 | **默认**:破坏性 schema / 数据变更 |
| 在线 DDL 工具 | 无 | 中 | 低 | 中 | 大表加索引 / 改列 |
| 双写双读 + 灰度 | 无 | 低(灰度期) | 中(需比对收敛) | 高 | 换存储 / 换主键 / 拆表 |
| 影子读比对 | 无 | 低(不对外) | 无(只比对) | 中 | 换实现等价性验证 |

### 5.2 借纪律不借实现:按"兼容性 + 数据量"重选路径（强制）

同一条"零停机安全演进"的纪律,**路径随变更形态变**（[`00 §2.4`](00_design_overview.md)）:加可空字段 → 直接上;改语义 → expand-contract;大表加索引 → 在线 DDL;换实现 → 影子比对 + 灰度。把"双写双读 + 影子比对"硬套到"加一个可空字段"是过度设计,把"原地改字段类型"留给亿级线上表是欠设计(锁表 + 旧数据崩)。**可迁移的是"破坏性必走 expand-contract、回填必限流分批、删除永远最后做、收缩前是最后回滚点"这套纪律,不是某个在线 DDL 工具(gh-ost / pt-osc)或某个迁移框架。**

## 6. 反模式速查 + 机械闸门（引用不复制）

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向（详见 docs-spec/05 / mvp_rebuild_path） |
| --- | --- | --- |
| 原地改字段语义 / 类型 / 删列(打挂新旧) | `R-EVO-INPLACE-BREAKING` | expand-contract 三段(§3.1) |
| 加字段无默认值(旧行 / 旧代码崩) | `R-SCHEMA-NO-DEFAULT`(05 §8) | 加字段必有默认值(§3.2) |
| 一条 UPDATE 扫全表回填(长事务 + 复制延迟) | `R-EVO-BACKFILL-BIGTXN` | keyset 分批 + 限流(§3.3) |
| 回填不限速(顶爆主库 / 复制延迟) | `R-EVO-BACKFILL-NOTHROTTLE` | 监控反压自动降速(§3.3) |
| 大表原生 DDL 锁表(阻塞在线读写) | `R-EVO-DDL-LOCK` | 在线 DDL 工具(影子表 + 原子改名,§3.2) |
| 换实现不影子比对直接切(无等价验证) | （选型审查） | shadow read 比对收敛 → 灰度(§3.4) |
| 不兼容发布无灰度(全量一把切) | `R-REFACTOR-NO-CANARY` | 1%/10%/50%/100% 灰度切流量(§3.4) |
| 先删旧再迁(收缩早于迁移,无回滚点) | `R-EVO-CONTRACT-EARLY` | 删除永远最后做,跨发版周期(§3.1/§3.5) |
| 回滚时旧版本读不了新数据(软兼容缺位) | （选型审查） | 回滚前确认双向兼容就位(§3.5) |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）:**加字段无默认值**(DDL `ADD COLUMN` 无 `DEFAULT` → 复用 `R-SCHEMA-NO-DEFAULT`);**全表回填大事务**(`UPDATE`/`DELETE` 无 `WHERE` 限定 / 无分批 → `R-EVO-BACKFILL-BIGTXN`);**破坏性 DDL 直改**(`DROP COLUMN`/`MODIFY` 类型缩窄未先在 §8 登记废弃 → `R-EVO-INPLACE-BREAKING` 软提醒)。"是否该走 expand-contract""灰度档位齐不齐""回滚点判对没"等语义类违规走选型审查(软提醒)。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意:规范本体（§1-§8）已用占位符表达;本节系统名 / 数值 / 代码为某真实工程实现,**只借纪律不借实现**——落地工程时按自己的兼容性与数据量重选路径。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 改字段语义走 expand-contract（三段并存）

```
需求:把 amount 从"分"改成"元"(语义变,破坏性)
① expand:加 amount_yuan 列(默认值),双写 amount(分) + amount_yuan(元);发布(新旧实例混跑都对)
② migrate:后台限流回填存量 amount_yuan = amount/100;读切到 amount_yuan(影子比对 → 灰度放量)
③ contract:确认无人读 amount → 停双写 → 跨一个发版周期后 DROP amount
绝不:原地把 amount 单位改成元(旧代码按"分"解析即错账)
```

### 9.2 大表加索引走在线 DDL

```
亿级表加复合索引:
❌ ALTER TABLE big ADD INDEX ...        // 原生:长时间持锁,阻塞在线写
✅ 在线 DDL 工具:建影子表 + 触发器/binlog 追平 + 低峰限速 + 切换瞬间原子 rename
监控:复制延迟超阈值自动暂停,不与在线流量抢
```

### 9.3 限流分批回填（不抢在线资源）

```go
// keyset 游标分块 + 限速,把回填压在在线流量之下
var lastID uint64
for {
    rows := SELECT id, ... FROM t WHERE id > lastID ORDER BY id LIMIT 1000   // 02 §3.6 keyset
    if len(rows) == 0 { break }
    UpsertBatch(rows.fill())                                                 // 小事务,幂等
    lastID = rows[len(rows)-1].id                                           // 记位点,中断可续(10 §3.2)
    if replicaLagHigh() { sleep(backoff) }                                  // 反压降速(07 同构)
    sleep(rateLimitInterval)                                                 // 夜间更宽
}
```

### 9.4 换实现:影子读 → 灰度

```
旧实现 oldCalc、新实现 newCalc:
① shadow:线上两条都跑,对外用 oldCalc,后台 assert(newCalc == oldCalc) 记差异 → 差异收敛到 0
② canary:读切 newCalc 1% → 10% → 50% → 100%,每档盯错误率 / P99(mvp_rebuild_path §11.5.4)
③ retire:newCalc 全量稳一个周期 → 删 oldCalc
```

### 9.5 回滚点判定

```
expand / migrate 阶段(旧路径在) → 回滚 = 流量切回旧,数据不动,秒退
contract 阶段(已 DROP 旧列/旧表) → 不可秒退:旧版本读不了新结构 → 需数据反向迁移
∴ 删除永远最后做、跨发版周期;收缩前的最后一刻 = 最后回滚点
```
