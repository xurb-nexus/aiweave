# 09 - API 契约 / 接口粒度设计决策规范

> 回答"面对一个需求，怎么把对外接口的**粒度、批量化、分页契约、版本兼容、协议**选对"。这是**比 IO 还上游**的视角——一个 chatty 的接口会**在调用方强制制造 N+1**，[`01`](01_io_design.md) 在服务端拼命消灭的 N+1，常是接口粒度逼出来的。
>
> **边界先声明**：请求 / 响应 JSON 与 Go struct、字段校验规则、Controller 伪码、错误码映射、BFF 集成等**表示规则的真相源在 [`docs-spec/07`](../docs-spec/07_api_interfaces_spec.md)**；跨接口业务流程在 [`docs-spec/08`](../docs-spec/08_api_flow_spec.md)；跨服务 SLA / 超时 / 版本号管理在 [`docs-spec/24`](../docs-spec/24_cross_service_contract_spec.md)；状态码语义在 [`docs-spec/14`](../docs-spec/14_status_codes_spec.md)，本篇只引用不复制。本篇新增决策资产：**接口粒度（chatty vs chunky）判定、批量 / 聚合端点设计、分页契约选型、版本与前后向兼容（expand-contract）、字段投影 / 部分响应、幂等键、协议选型（请求-响应 / 流式 / 长连接）**。

---

## 1. 设计目标

- **粒度匹配调用场景**：接口的"一次往返拿到多少"匹配调用方的真实用法——不让调用方为拼一个页面循环调 N 次（chatty 逼出客户端 N+1），也不返一个谁都用不全的巨无霸（over-fetch）。
- **契约可演进**：接口能加字段 / 加能力而不打挂存量调用方；破坏性变更有版本与迁移窗口，**永不原地改字段语义**。
- **批量是一等公民**：对"一批"的操作提供批量端点，把"消灭 N+1"从服务端纪律前移成**接口契约**——调用方天然有批量接口可用，不必循环单调。
- **协议匹配交互模式**：请求-响应 / 服务端推送 / 双向流，按交互形态选，不用轮询硬模拟推送。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 新增对外接口 | "对 `{audience}` 暴露一个新接口" | 粒度 / 契约 |
| 一屏多份数据 | "前端一个页面要 `{A}`、`{B}`、`{C}` 多份数据" | 聚合端点 / BFF |
| 调用方循环调 | "调用方要对一批 `{id}` 各调一次接口" | 批量端点（防客户端 N+1） |
| 列表 / 翻页 | "返回一个可翻页 / 可滚动的 `{列表}`" | 分页契约 |
| 接口要改 | "给既有接口加 / 改 / 删字段 / 改语义" | 版本与兼容 |
| 大响应 / 多余字段 | "响应很大、但调用方多数时候只要几个字段" | 字段投影 / 部分响应 |
| 写接口可能重试 | "客户端 / 网关可能重复提交同一个写" | 幂等键 |
| 实时 / 推送 | "要实时下发 `{变化}` / 长任务进度 / 双向通信" | 协议选型 |
| 跨服务调用 | "本服务作为下游被 `{上游}` 调用 / 调用 `{下游}`" | 跨服务合约（[`docs-spec/24`](../docs-spec/24_cross_service_contract_spec.md)） |

> 识别口诀：**"调用方要循环调 / 要拼多份数据"→ 粒度问题，做批量 / 聚合端点；"接口要改"→ 先问破坏性，破坏性必上版本；"要推送"→ 别用轮询硬模拟。**

## 3. 决策树（强制 / 本篇核心）

### 3.1 接口粒度决策树（chatty vs chunky，直接决定调用方 N+1）

```
调用方拿这个接口的数据，会怎么用？
│
├─ 要对一批 {id} 逐个拿 ──────► 提供批量端点（一次传一批 → 一次返一批），杜绝调用方循环单调
│        ↳ 这是"消灭 N+1"前移到契约层：服务端 01 的批量纪律，要先有批量接口承接
│        ↳ 批量上限设硬阈值（如 ≤ {batch-limit}），超限分页 / 拒绝，防单请求过载
│
├─ 一屏 / 一次交互要多份关联数据 ──► 聚合端点 / BFF：一次往返返聚合好的视图，调用方不自己编排多调用
│        ↳ 服务端用聚合器并行取各份（01 §3）；对外只暴露"一次拿全"的粗粒度接口
│
├─ 各调用方要的字段差异大 ──────► 字段投影 / 部分响应（§3.4）：让调用方声明要哪些字段，免 over-fetch
│
└─ 单实体单次操作 ──────────────► 细粒度接口即可（别为单操作硬造批量 / 聚合，过度设计）
```

> **粒度的反直觉点**：服务端 [`01`](01_io_design.md) 消灭 N+1 是治标——若接口本身只支持"一次一个"，调用方被迫循环调 N 次，N+1 跑到了**网络上、跨进程**，更贵且服务端管不着。**接口粒度是 N+1 的"上游闸门"**：先有批量 / 聚合端点，服务端的批量纪律才有意义。

### 3.2 分页契约决策树（与 02 keyset 同一根，落在契约侧）

```
列表接口怎么定分页契约？
│
├─ 浅翻页（只看前几页 / 总页数有意义） ──► offset/limit 契约（页码 + 页大小 + total）
├─ 深翻 / 无界下滑 / 导出 ──────────────► 游标契约（返回 next_cursor，调用方回传，不暴露 offset）
│        ↳ 契约层只给 cursor 不可猜（编码上页末游标）；底层走 keyset 索引定位（02 §3.6）
│        ↳ 不可随机跳页是契约的一部分，要在接口文档显式声明（前端按"加载更多"设计）
└─ 全量同步 ──────────────────────────► 分块流式契约（cursor + 固定 chunk）或变更增量（CDC / since-token，08）
```

> 分页契约（对外形态）与 [`02 §3.6`](02_data_model_design.md) keyset 游标（底层索引实现）是**同一决策的两侧**：02 管"DB 怎么 O(log N) 翻"，本篇管"接口怎么把游标暴露给调用方"。深分页必须在契约层就是 cursor 形态，否则底层再优也被 offset 契约拖死。

### 3.3 版本与前后向兼容：expand-contract（永不原地改语义 / 强制）

```
要改一个已被调用的接口，怎么不打挂存量调用方？
│
├─ 加字段 / 加可选参数 ──► 后向兼容（老调用方忽略新字段）：直接加，不升版本
├─ 改字段语义 / 删字段 / 改类型 ──► 破坏性：走 expand-contract（扩展-收缩）三步，绝不原地改
│        ① expand：新增"新字段 / 新接口"与旧并存，双写双供
│        ② migrate：推动调用方迁到新字段 / 新接口（一个迁移窗口）
│        ③ contract：确认无人用旧字段后再删（至少跨一个发版周期）
└─ 整体不兼容重构 ──► 升主版本号（/v2），新老版本并行一个废弃窗口，旧版标 deprecation
```

> **expand-contract 是本支柱"破坏性变更必须新老并存一个窗口"纪律的契约层实例**——与 [`08 §3.7`](08_messaging_design.md) 消息 schema 演进、[`11 §3`](11_evolution_migration_design.md) 数据库字段演进**同构**。版本号升级判定 / 废弃登记的表示规则见 [`docs-spec/24 §5`](../docs-spec/24_cross_service_contract_spec.md)；字段演进安全路径见 [`docs-spec/05 §8`](../docs-spec/05_schema_design_spec.md)。

### 3.4 字段投影 / 部分响应（治 over-fetch）

```
响应大、但不同调用方只要其中一部分？
│
├─ 字段差异小 / 调用方少 ──► 固定多视图（如 Brief / Full 两档），别过度灵活
├─ 字段差异大 / 调用方多 ──► 字段投影：调用方声明 fields=a,b,c，服务端按需裁剪（省载荷 + 省序列化）
└─ 关系图复杂、按需取关联 ──► 查询式接口（如 GraphQL 风格），代价是服务端要防"深查询放大"
```

> 字段投影同时省**网络载荷**与**服务端序列化 CPU**（与 [`05 §3.3`](05_caching_design.md) 序列化格式选型互补）。但灵活度有代价：查询式接口要防调用方构造放大查询（深嵌套 → 服务端 N+1），需配深度 / 复杂度上限。

### 3.5 写接口幂等键（让重试安全成为契约）

```
写接口（创建 / 扣减 / 提交）可能被重复调用吗？
│
├─ 天然幂等（覆盖式 PUT / 状态机单向跃迁） ──► 无需额外幂等键
└─ 非幂等（POST 创建 / 累加 / 扣费） ──► 契约层要求调用方带【幂等键】（如 Idempotency-Key 头）
         ↳ 服务端以幂等键去重：同键重复请求返同一结果（不重复生效），见 04 / 08 §3.4
         ↳ 把"重试安全"写进接口契约 → 调用方 / 网关可放心重试（与 07 重试预算配套）
```

### 3.6 协议选型（匹配交互模式，别用轮询硬模拟推送）

```
这个接口的交互形态是什么？
│
├─ 一问一答（绝大多数） ──────► 请求-响应（REST / RPC）：默认
├─ 内部高频 / 强类型 / 多语言 ─► RPC（如 gRPC，二进制 + IDL + 多路复用），省序列化与连接开销
├─ 服务端单向推送（进度 / 通知 / 行情） ──► 服务端推送（SSE / 长轮询），别用前端高频轮询硬刷
├─ 双向实时（聊天 / 协同 / 信令） ──────► 全双工长连接（WebSocket）
└─ 大文件 / 流式结果 ──────────────────► 流式传输（分块 / streaming），别一次性 load 全量进内存
```

> 协议选型与传输层优化（HTTP/2 多路复用、gRPC streaming）联动 [`01`](01_io_design.md) 的"消灭往返"。轮询模拟推送是高频空往返的典型反模式——交互是"推送"就用推送协议。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 接口粒度 | 细粒度单实体接口 | 调用方循环调 → 批量端点；一屏多份 → 聚合 / BFF | 单操作 → 别硬造批量 / 聚合 |
| 批量端点 | 无（单条） | 调用方对一批逐个调 → 批量端点（带硬上限） | 上限要封顶，防单请求过载 |
| 分页 | offset/limit | 深翻 / 无界 → 游标契约（不暴露 offset，§3.2） | 浅翻 → offset 够用，别强上游标 |
| 接口演进 | 加可选字段（后向兼容） | 破坏性变更 → expand-contract 三步 / 升主版本（§3.3） | 加字段 → 别升版本（兼容即可） |
| 响应字段 | 固定全量 | over-fetch 明显 → 多视图 / 字段投影（§3.4） | 字段差异小 → 别上灵活投影 |
| 写接口 | 不要求幂等键 | 非幂等 + 可能重试 → 契约要求幂等键（§3.5） | 天然幂等 → 无需幂等键 |
| 协议 | 请求-响应 REST | 推送 → SSE / 长连接；内部高频 → RPC；流式 → streaming | 一问一答 → 别上长连接 |

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 粒度 / 协议权衡矩阵

| 方案 | 往返次数（调用方视角） | 载荷 | 演进灵活度 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| 细粒度单实体 | N（易逼出客户端 N+1） | 小 | 高 | 低 | 单操作 / CRUD |
| 批量端点 | 1（一批一次） | 中 | 中 | 中 | 调用方批处理 |
| 聚合端点 / BFF | 1（一屏一次） | 大 | 中 | 中高（服务端编排） | 多份关联数据拼视图 |
| 字段投影 / 查询式 | 1（按需取） | 可控 | 最高 | 高（防放大查询） | 字段差异大 / 关系复杂 |
| 流式 / 推送 | 长连接 | 流 | 中 | 中高 | 实时 / 大结果 |

### 5.2 借纪律不借实现：按调用场景重选契约（强制）

同一条"接口粒度匹配用法、契约可演进"的纪律，**形态随调用场景变**（[`00 §2.4`](00_design_overview.md)）：批处理调用方 → 批量端点；拼页面 → 聚合 / BFF；字段差异大 → 投影；实时 → 推送协议。把"GraphQL 查询式"硬套到"内部两个服务一问一答"是过度设计，把"细粒度 REST"硬留给"前端拼十份数据的页面"是逼出客户端 N+1。**可迁移的是"粒度对齐用法、批量是契约一等公民、破坏性变更走 expand-contract、协议匹配交互"这套纪律，不是 REST / gRPC / GraphQL 中的某一个。**

## 6. 反模式速查 + 机械闸门（引用不复制）

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向（详见 docs-spec/07 / 24） |
| --- | --- | --- |
| 只给单条接口、逼调用方循环调（客户端 N+1） | `R-API-CHATTY-NPLUS1` | 批量 / 聚合端点（§3.1） |
| 批量端点无上限（单请求拉爆） | `R-API-BATCH-NOLIMIT` | 批量硬阈值，超限分页 / 拒绝（§3.1） |
| 深分页对外暴露 offset（底层被迫深扫描） | `R-API-OFFSET-DEEP` | 游标契约 next_cursor（§3.2 / 02 §3.6） |
| 原地改字段语义 / 删字段（打挂存量调用方） | `R-API-BREAKING-INPLACE` | expand-contract 三步 / 升主版本（§3.3） |
| 破坏性变更不升版本号 | `R-XSVC-VERSION`（24 §5） | 升主版本 + 废弃窗口（§3.3） |
| 巨无霸响应（谁都用不全的 over-fetch） | （选型审查） | 多视图 / 字段投影（§3.4） |
| 非幂等写接口无幂等键（重试重复生效） | `R-TXN-NO-IDEMPOTENT` | 契约要求幂等键（§3.5 / 04） |
| 前端高频轮询模拟推送（高频空往返） | （选型审查） | SSE / 长连接推送（§3.6） |
| 大结果一次性 load 进内存返回 | （选型审查） | 流式 / 分块（§3.6 / 02 §3.6） |
| 接口返回未加密真实主键（可顺序枚举） | `R-SEC-ID-PLAINTEXT` | 出口确定性加密 ID（05 §9.5） |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）：**批量端点缺上限**（批量入参 slice 直接进查询、grep 不到长度校验 → `R-API-BATCH-NOLIMIT`）；**深分页暴露 offset**（对外参数含 offset/page 且无 cursor → `R-API-OFFSET-DEEP` 软提醒）；**对外 ID 明文**（出口 struct 直接带真实主键字段 → 复用 `R-SEC-ID-PLAINTEXT`）。"粒度是否匹配用法""版本该不该升"等语义类违规走选型审查（软提醒）。

### 9.1 批量端点把 N+1 挡在契约外（系统 A 读网关）

```
❌ 细粒度：GET /entity/{id}  → 调用方拼一屏 16 个 → 循环调 16 次（客户端 N+1，跨网络更贵）
✅ 批量：  POST /entity:batchGet { ids: [...] }  → 一次返一批（上限 ids ≤ 100，超限拒绝）
服务端：批量端点内部走聚合器（01 §9.2）"1 次 MGET + 并行 DB"，对外只暴露"一次拿全"的粗粒度接口
实测：某 KYC 页面从 16 次往返 → 1 次批量端点，端到端 RTT 降 80%+
```

### 9.2 聚合端点 / BFF（一屏一次）

```
前端首页要：用户基本信息 + 配额余量 + 最近流水 + 公告  → 4 份异构数据
❌ 前端各调一次（4 往返、前端编排、易踩串行）
✅ BFF 聚合端点 GET /home:aggregate → 服务端用聚合器并行取 4 份（01 §3）→ 一次返聚合视图
```

### 9.3 游标分页契约（对外不暴露 offset）

```json
// 请求：GET /list?limit=20&cursor={opaque}      // 首页 cursor 省略
// 响应：

// 文档显式声明：不可随机跳页，前端按"加载更多"设计
```

### 9.4 expand-contract 改字段（不打挂存量）

```
需求：把 status: int（0/1/2）改成 status: string（"pending"/"paid"/...）
① expand：响应同时返 status（旧 int，标 deprecated）+ status_text（新 string），双供一个窗口
② migrate：通知调用方迁到 status_text；监控旧字段读取量降为 0
③ contract：跨一个发版周期后删 status（旧 int）
绝不原地把 status 的 int 改成 string（存量调用方解析即崩）
```

### 9.5 幂等键契约（重试安全写进合约）

```
POST /order  Header: Idempotency-Key: {client-uuid}
服务端：以 Idempotency-Key 查去重表 → 命中返首次结果（不重复创建）/ 未命中创建并落键（04 / 08 §3.4）
契约声明"本接口幂等"→ 网关 / 客户端可放心重试（与 07 §3.5 重试预算配套）
```

### 9.6 协议选型速记

```
一问一答          → REST（默认，可调试、跨语言）
内部高频强类型     → gRPC（二进制 + IDL + HTTP/2 多路复用，省序列化与连接）
长任务进度 / 通知  → SSE / 长轮询（服务端单向推送，别前端高频轮询）
双向实时           → WebSocket（聊天 / 协同 / 信令）
大文件 / 流式结果  → streaming 分块（别一次性 load 全量进内存，02 §3.6 流式同构）
```
