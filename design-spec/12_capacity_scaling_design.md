# 12 - 容量规划与弹性伸缩设计决策规范

> 回答"怎么从 SLA + 流量**反推**出需要多少容量(副本 / 连接 / 池),留多少余量,以及伸缩用什么信号、怎么不抖动"。
>
> **核心命题**：[`07 §9.6`](07_tail_latency_design.md) 已把两条定律(扇出序统计、排队 `1/(1−ρ)`)写透,但那是**诊断**长尾;本视角把同一组定律**正向**用起来——`ρ≈0.8 的膝盖`不只是过载警报,更是**容量规划的设计输入**:把利用率压在膝盖以下需要多少资源,就是该备的容量。把 [`03`](03_concurrency_design.md) 的 Little 池容量 + [`07`](07_tail_latency_design.md) 的排队律,从"单点取值"升为"全系统容量配比 + 弹性策略"。
>
> **边界先声明**：全局 SLA / 热路径 / 内存预算 / 背压 / 容量目标等**规则真相源在 [`docs-spec/22`](../docs-spec/22_performance_contract_spec.md)**;部署配额(requests/limits) / 运行时旋钮(GOMAXPROCS/GOMEMLIMIT) / 弹性伸缩编排在 [`docs-spec/27`](../docs-spec/27_deployment_runtime_spec.md);连接池基线在 [`06 §9.2`](06_resilience_design.md);可观测性指标在 [`docs-spec/23`](../docs-spec/23_observability_spec.md),本篇只引用不复制。本视角无独立 docs-spec,按双源真相**分布落地**到既有表示真相源(同 [`07`](07_tail_latency_design.md))。本篇新增决策资产:**从 SLA 反推容量、余量(headroom)规划、资源配比(副本 / 连接 / 池一致性)、伸缩信号选型、伸缩防抖、有状态 vs 无状态伸缩**。

---

## 1. 设计目标

- **容量有据可算**：副本数 / 连接数 / 池容量从"流量 + 单请求成本 + SLA"反推,不靠拍脑袋;每个数字能回答"为什么是这个值"。
- **留够余量**：稳态利用率压在排队膝盖(`ρ≈0.8`)以下,给突发 / 故障转移 / 伸缩滞后留追赶余力——`1/(1−ρ)` 告诉你:省到 `ρ=0.95` 时延迟已 20×。
- **资源配比自洽**：副本数、每副本连接数、下游连接上限、池容量必须**一致**——不能"加副本却把下游连接打爆",也不能"池很大但下游承载不住"。
- **伸缩稳、不抖**：扩容快(保护 SLA)、缩容慢(防抖);用对信号(反映饱和的指标),配冷却期,不在阈值附近反复横跳。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 定容量 | "这个服务要部署几个副本 / 连接池开多大" | 从 SLA 反推 |
| 有 SLA + 流量预估 | "峰值 `{QPS}`、要保 P99 `{ms}`" | 容量计算 + 余量 |
| 流量起伏大 | "白天高峰夜间低谷 / 大促尖峰" | 弹性伸缩 |
| 加副本不见效 | "加了实例吞吐没涨 / 下游反而被打爆" | 资源配比 / 瓶颈定位 |
| 伸缩抖动 | "副本数反复扩缩、震荡" | 伸缩防抖 |
| 冷启慢 | "扩容新实例头几秒 P99 飙高" | 伸缩滞后 + 预热(联动 [`07`](07_tail_latency_design.md)) |
| 有状态扩容 | "带本地状态 / 分片 / 长连接的服务怎么扩" | 有状态伸缩 |
| 容量评审 | "大促前要做容量评估 / 压测定容" | 全链路容量 |

> 识别口诀：**"要定几个副本 / 池多大"→ 从 SLA + 流量反推;"加机器不见效"→ 查资源配比瓶颈;"伸缩抖 / 冷启飙"→ 防抖 + 预热。**

## 3. 决策树（强制 / 本篇核心）

### 3.1 从 SLA + 流量反推容量（容量规划的主干）

```
要多少容量? 从"流量 × 单请求成本 / SLA"反推
│
├─ 算并发需求 N ──► Little 定律:N = 到达率 λ × 单请求驻留 W(03 §3.2)
│        例:λ=2000/s、W=20ms → 稳态 N≈40 个请求同时在飞
│
├─ 留余量(headroom) ──► 把稳态利用率压在膝盖 ρ≤0.8 以下 → 实备容量 = N / ρ ≈ N / 0.8
│        ↳ 依据:排队律 1/(1−ρ),ρ=0.8 延迟 5×、ρ=0.95 已 20×(07 §9.6);余量给突发 + 故障转移 + 伸缩滞后
│
├─ 折算资源 ──► 副本数 = ⌈实备容量 / 单副本可承载⌉;每副本连接 / 池按 03 Little + 06 基线
│
└─ 校验下游 ──► 副本数 × 每副本下游连接 ≤ 下游可承载 → 否则加副本即打爆下游(回 §3.3 配比)
```

> **余量不是浪费,是排队律的必然**:`ρ` 趋 1 时延迟趋无穷,膝盖≈0.8。把利用率顶到 95% 省下的机器,会以 20× 的尾延迟还回来,且突发 / 故障转移时无追赶余力 → 雪崩。SLA / 容量目标的表示真相源在 [`docs-spec/22 §1（SLA）/ §6（背压与容量保护）`](../docs-spec/22_performance_contract_spec.md)。

### 3.2 余量分级（不同角色留不同余量）

```
该留多少余量? 按"突发性 + 故障域"分级
│
├─ 无状态在线服务 ──► ρ≤0.7~0.8;留 1 个故障域(N+1 冗余:挂一个副本剩下的扛得住)
├─ 共享下游(DB / 缓存) ──► 更保守:它被所有副本共享,打爆即全局故障 → 连接上限留更多余量
├─ 突发尖峰(大促 / 定时洪峰) ──► 按峰值定容或预扩(伸缩滞后赶不上瞬时尖峰,§3.4)
└─ 批 / 离线 ──────────────► 可吃高利用率(ρ→0.9+):无 SLA 压力、可排队、可慢
```

> 在线服务留余量、离线吃满利用率——这是"读写分治"在容量维度的同构:在线求低延迟(留余量),离线求高吞吐(吃满)。

### 3.3 资源配比一致性（加副本不打爆下游 / 强制）

```
副本 / 连接 / 池 / 下游必须配比自洽,否则加机器适得其反:
│
├─ 副本数 × 每副本 DB 连接 ≤ DB 最大连接 ──► 否则加副本 → DB 连接耗尽 → 全局拒绝
│        ↳ 加副本时必须同步下调每副本连接 / 上连接池代理(连接复用),否则连接数随副本线性爆
├─ 池容量 ≤ 下游可承载 ──────────────────► 池比下游能扛的还大 = 把下游打满(03 §3.4 下游归属舱)
├─ 缓存 / 分片容量随分片线性 ──────────────► 前提无大 key / 热 key / 倾斜(05 §3.6),否则卡在最热分片
└─ 瓶颈在哪加哪 ──────────────────────────► 先定位瓶颈资源(CPU / 连接 / 下游 / 锁),别盲目加副本
```

> **加副本不见效的头号原因是配比失衡**:无状态层加副本,但所有副本共享的 DB / 缓存 / 下游没扩,瓶颈纹丝不动甚至更糟(连接被打爆)。**容量规划是全链路的**:扩一层必须校验它依赖的下一层能否承接。连接池基线见 [`06 §9.2`](06_resilience_design.md),下游承载见 [`03 §3.4`](03_concurrency_design.md) / [`docs-spec/24`](../docs-spec/24_cross_service_contract_spec.md)。

### 3.4 弹性伸缩:信号、方向不对称、防抖

```
要不要自动伸缩? 用什么信号? 怎么不抖?
│
├─ 该不该上自动伸缩 ──► 流量起伏大(峰谷比高)→ 上;流量平稳 → 固定容量 + 余量即可(别为伸缩引复杂度)
│
├─ 伸缩信号选型 ──► 选"反映饱和"的信号,别只看 CPU
│        ├─ CPU-bound → CPU 利用率
│        ├─ IO-bound(多数) → 在途并发 / 队列深度 / P99(CPU 可能不高但已排队,07 §9.6)
│        └─ 消费型 → MQ lag(08 §3.5)
│
├─ 方向不对称 ──► 扩容快(立即响应,保 SLA)、缩容慢(长冷却,防抖);缩容步长小、扩容步长大
│
└─ 防抖 ──► 冷却期(scale 后静默一段)+ 阈值滞后(扩/缩用不同阈值,留死区)+ 冷启预热(07 §3.0)
            ↳ 否则:扩→负载降→缩→负载升→扩…在阈值附近震荡(flapping)
```

> **伸缩的第一性约束是"滞后"**:从触发扩容到新实例就绪有分钟级延迟(拉起 + 预热),赶不上秒级尖峰 → 尖峰要靠余量 / 预扩,不能指望实时伸缩。冷启预热(readiness 晚于预热)见 [`07 §3.0`](07_tail_latency_design.md) / [`docs-spec/27 §4`](../docs-spec/27_deployment_runtime_spec.md);伸缩与 load shedding 互补——伸缩是中期加容量,load shedding(07)是瞬时扛不住时有序丢。

### 3.5 有状态 vs 无状态伸缩

```
这个服务带状态吗? 决定能不能随便加副本
│
├─ 无状态 ──────────► 随便水平扩(加副本即加容量);状态外置到 DB / 缓存(默认追求的形态)
├─ 带本地状态 / 分片 ─► 扩容要迁分片 / rebalance(代价高);用一致性哈希减小迁移面(05 §3.6,禁裸 mod N)
├─ 长连接 / 会话粘连 ─► 扩容新副本不自动分担存量长连接 → 需连接再平衡 / 优雅迁移
└─ 垂直 vs 水平 ─────► 单点瓶颈(单写库)先垂直(加配置),到顶再想拆分(分库分表,02);无状态优先水平
```

> **无状态是可伸缩性的前提**:把状态外置(DB / 缓存 / MQ),计算层才能随意水平扩。带状态的扩容必涉及数据迁移 / rebalance,代价远高——这也是 [`02`](02_data_model_design.md) 分库分表、[`05`](05_caching_design.md) 一致性哈希要解决的问题。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 定容量 | Little 反推 N + 余量(N/0.8) | 有压测数据 → 按实测校准 | 别按峰值的峰值过度备(成本) |
| 余量 | 在线 ρ≤0.8 + N+1 冗余 | 突发尖峰 → 预扩 / 按峰定容 | 离线批 → 吃满 ρ→0.9+(别留余量浪费) |
| 资源配比 | 副本 × 连接 ≤ 下游承载 | 连接随副本爆 → 连接池代理 / 下调每副本连接 | 别加副本不校验下游 |
| 自动伸缩 | 平稳流量 → 固定容量 | 峰谷比高 → 自动伸缩(选饱和信号,§3.4) | 平稳流量 → **别上自动伸缩**(徒增抖动) |
| 伸缩信号 | IO-bound 看并发 / 队列 / P99 | CPU-bound → CPU;消费 → lag | 别只看 CPU(IO 型 CPU 不高也已排队) |
| 伸缩方向 | 扩快缩慢 + 冷却 + 阈值滞后 | 仍抖 → 加大死区 / 延长冷却 | — |
| 伸缩 | 无状态水平扩 | 带状态 → 一致性哈希 rebalance | 单点 → 先垂直再拆 |

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 容量 / 伸缩策略权衡矩阵

| 策略 | 成本 | 抗突发 | SLA 保障 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| 固定容量 + 余量 | 中(留余量) | 中(靠余量) | 好 | 低 | **默认**:流量平稳 |
| 按峰值定容 | 高(峰谷浪费) | 好 | 最好 | 低 | 尖峰不可丢、伸缩赶不上 |
| 自动水平伸缩 | 低(随需) | 中(有滞后) | 中(冷启窗口) | 高 | 峰谷比高、无状态 |
| 预扩(定时 / 预测) | 中 | 好 | 好 | 中 | 可预测的定时洪峰(大促) |
| 垂直扩容 | 中 | 低(单点上限) | 中 | 低 | 有状态单点、暂不拆分 |

### 5.2 借纪律不借实现:按"流量形态 + 状态性"重选策略（强制）

同一条"容量够用且不浪费"的纪律,**策略随形态变**（[`00 §2.4`](00_design_overview.md)）:平稳流量 → 固定容量 + 余量;高峰谷比无状态 → 自动伸缩;可预测洪峰 → 预扩;有状态单点 → 垂直 / 拆分。把"自动伸缩"硬套到"平稳流量的有状态服务"是过度设计(抖动 + 迁移代价),把"固定小容量"留给"大促尖峰"是欠设计(雪崩)。**可迁移的是"从 SLA 反推、压膝盖以下留余量、配比全链路自洽、扩快缩慢防抖"这套纪律,不是某个 HPA 配置或某个云的弹性产品。**

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向（详见 docs-spec/22 / 27） |

| 有状态服务当无状态随意扩(丢状态 / 裂分片) | （选型审查） | 一致性哈希 rebalance / 先垂直(§3.5) |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）:**缺配额**(编排清单无 requests/limits → 复用 `R-DEPLOY-NO-LIMITS`);**冷启无预热**(readiness 置位早于缓存 / 连接池预热 → 复用 `R-TAIL-COLDSTART`)。"余量留够没""配比自洽否""伸缩信号选对没""flapping 防住没"等语义类违规走选型审查 / 压测验证(软提醒)——容量类决策多需实测校准,机械闸门只兜最硬的配额 / 预热缺失。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意:规范本体（§1-§8）已用占位符表达;本节数值 / 公式为示意推导,**只借纪律不借实现**——落地工程时按自己的流量与压测数据重算。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 从 SLA 反推容量（含余量）

```
输入:峰值 λ=2000 QPS、单请求 W=20ms、P99 SLA=100ms、单副本可承载 ~600 QPS
① Little:稳态在飞 N = λ×W = 2000×0.02 = 40 个请求
② 余量:压 ρ≤0.8 → 实备吞吐 = 2000/0.8 = 2500 QPS 的承载力
③ 副本:⌈2500/600⌉ = 5 副本(再 +1 故障域冗余 = 6)
④ 校验下游:6 副本 × 每副本 20 DB 连接 = 120 ≤ DB maxOpen 200 ✓(否则降每副本连接)
```

### 9.2 余量为什么是 ρ≤0.8（排队律的账）

```
ρ=0.50 → 延迟 2×    ρ=0.80 → 5×(膝盖)    ρ=0.95 → 20×    ρ=0.99 → 100×→∞   (07 §9.6)
顶到 95% 省下的机器,以 20× 尾延迟 + 突发无余力还回来 → 在线服务恒压膝盖以下
离线批无 SLA → 可吃 ρ→0.9+,反而追求高利用率(读写分治在容量维度的同构)
```

### 9.3 加副本打爆下游的反例

```
现象:无状态层 5→10 副本,QPS 没涨反而大面积 DB 报错
根因:每副本 maxOpen=50 → 10×50=500 > DB maxConnections=400 → 连接耗尽全局拒绝
修:加副本同步下调每副本连接(10×40=400)或上连接池代理(连接复用),并核 DB 才是真瓶颈(§3.3)
```

### 9.4 伸缩信号与防抖

```
IO-bound 网关:HPA 不用 CPU(CPU 30% 但在途并发已满、P99 飙) → 用"在途并发 / P99"做信号
方向:扩容阈值 70%、缩容阈值 40%(留 30% 死区);扩 cooldown 30s、缩 cooldown 300s(扩快缩慢)
尖峰:大促按预测预扩(伸缩分钟级滞后赶不上秒级尖峰);冷启 readiness 晚于预热(07,防扩容即飙 P99)
```
