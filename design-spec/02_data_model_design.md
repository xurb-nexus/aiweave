# 02 - 数据建模设计决策规范

> 回答"面对一个需求，怎么把实体落成表结构、索引、分库分表、数据访问层与组装层"。目标：**让数据库在热路径上尽可能少出现、出现时尽可能批量与并行**。
>
> **边界先声明**：DDL 字段精度、分库设计、Redis Key 注册表、GORM Model 规范等**表示规则的真相源在 [`docs-spec/05`](../docs-spec/05_schema_design_spec.md)**；批量 / 聚合编排在 [`01`](01_io_design.md) + [`docs-spec/25`](../docs-spec/25_io_aggregation_spec.md)，本篇只引用不复制。本篇新增决策资产：数据访问层范式、分库分表决策树、shard 分组并行、组装层 map 入参纪律。

---

## 1. 设计目标

- **正确 / 可扩展 / 可索引**：实体关系无歧义、演进路径预留、高频查询路径有支撑索引。
- **数据层可无副作用复用**：数据访问层是纯取数单元，被单查 / 预取 / 批量 / 多入口复用而行为不漂移。
- **结构上杜绝 N+1**：把"必须先批量查"编译进类型签名（组装层 map 入参），而非靠人记。

## 2. 输入信号识别

| 迹象 | 典型描述 |
| --- | --- |

| 读吞吐扩展 | "读 QPS 极高 / 主库读压力大 / 要不要上读副本" |

## 3. 决策树（强制）

### 3.1 数据访问层范式（每表 Repository 的标准形态）

```
一个表的 Repository 怎么分层？
│
├─ 私有纯查询 getFromDB ──► 分表路由 → 查询（仅索引列/主键 + 软删过滤）→ NotFound 转 (nil,nil)
│                            （无副作用真相源，被下面三条路径复用，SQL/分表逻辑只写一处）
├─ 公有单条缓存编排 ──────► 拼 key → 缓存 Get → 命中返回 / miss 调 getFromDB → 序列化 → 回写
├─ 公有预取注册 Preload ──► 注册到批量预取编排器，跨 Repo 合并 MGET（01 §5.2）
├─ 公有批量 {...List} ────► MGET → 未命中按 shard 分组并行 IN → 内存分组 → 回写（铁律0/三段式）
└─ 公有批量写 {...UpsertList} ─► 批量 Upsert（`ON DUPLICATE KEY`/`ON CONFLICT`）一次往返；大批量装载用 `LOAD DATA`（写极致见 01 §3.2）
```

> 私有纯查询是被复用的核心：缓存编排与 DB 查询解耦，编排器才能在不碰 SQL 的前提下接管缓存合并。

### 3.2 分库分表决策树

```
数据量 / 读写模式决定怎么切？
│
├─ 分库：按"业务域 + 量级 + 读写模式"切（不是按表名）──► 故障域物理隔离
│        └─ 每库按读写模式设不同持久化 / 复制级别（强一致核心库 vs 放宽 flush 的写密集库）
│
├─ 分表：按量级决定
│        ├─ ≤ 十万 / 百万 ──────► 不分表（别过度设计）
│        ├─ 海量结构化 ────────► id % N 水平分（N 按量级选）
│        └─ 纯追加流水 ────────► 按月分表 + 定时预建下月表 + 历史月表归档 DROP
│
└─ 海量小映射 ──► Hash 分桶（CRC32 取模控桶体积）替代海量独立 String key（存储见 05 §缓存）
```

> **逻辑表名 / 物理表名解耦（强制）**：分表算法收敛到唯一函数（唯一真相源）；**缓存 key 用逻辑表名，物理表名只在查询期计算** → 缓存层不知分表存在，支持测试态一键降表。

### 3.3 批量查询三段式 + shard 分组并行（铁律0 的标准实现）

```
1. MGET 批量缓存            → 一次拿全部命中
2. 未命中先按 shard 分组（只查有请求的物理表）→ 单 shard 直查 / 多 shard 并行 IN（非 N 次）
3. 内存按 key 分组 + 回写未命中（显式复用单查的 key 与序列化形式，避免双写不一致）
```

### 3.4 组装层（Assembler）map 入参纪律

```
DB 模型 → API 响应的组装：
  └─► 无副作用纯函数 + 关联数据强制以【已查好的 map】入参
       ↳ 结构上杜绝 Assembler 内 N+1；可单测（造假 map → 断言）；查询编排集中在上层
```

### 3.5 索引设计：覆盖索引 + 索引下推（读 IO 终极）

```
高频查询的索引怎么建？
│
├─ 默认 ──────────► 按查询路径建单列/复合索引（最左前缀；列序：等值列在前、范围列在后）
├─ 读 IO 极致 ────► 覆盖索引：查询所需列全放进索引 → index-only scan【不回表】
│                    ↳ 配合索引下推(ICP)：WHERE 非前缀条件在索引层先过滤再回表，少回表行数
├─ 长列 ──────────► 前缀索引 `{col}({N})`：只索前 N 字符，以选择性换索引体积
└─ 写放大约束 ────► 不过度索引：每个二级索引 = 写放大 + 额外空间，按查询路径建、定期审无用索引
```

> 覆盖索引是"消灭往返"在 DB 层的延伸——缓存 miss 回源 / 直查走覆盖索引免回表（读路径见 [`01 §3.1`](01_io_design.md)）。索引表示规则（DDL / 索引定义）真相源在 [`docs-spec/05`](../docs-spec/05_schema_design_spec.md)。

### 3.6 分页与大结果集：游标分页 + 流式（消灭 OFFSET 深扫描）

```
怎么翻页 / 取大结果集？
│
├─ 浅分页（只看前几页） ──► OFFSET / LIMIT 可接受（offset 小）
├─ 深分页 / 无界下翻 ────► keyset（游标）分页：`WHERE {有序唯一键} {>/<} {游标} ORDER BY … LIMIT n`
│                          ↳ 走索引定位，O(log N)；消灭 OFFSET 的"扫 N 行再丢弃"（O(N)，随页深线性劣化）
│                          ↳ 游标 = 上页末行的（有序键[+ 唯一 tiebreaker]）；不可随机跳页，但可无限下翻
└─ 全量遍历 / 导出 ──────► 流式 / 分块拉取（keyset 游标 + 固定 chunk）→ 封顶单请求内存，不一次性 load 全表
```

> keyset 依赖"有序唯一键 + 支撑索引"（覆盖索引更佳，§3.5）；排序列非唯一时追加唯一 tiebreaker（如主键）保稳定游标。索引支撑登记真相源见 [`docs-spec/05 §7`](../docs-spec/05_schema_design_spec.md)；往返 / 扫描行数回归见 [`docs-spec/25 §7`](../docs-spec/25_io_aggregation_spec.md)。

### 3.7 读副本路由与副本延迟（横向读扩展）

```
读吞吐不够，要不要上读副本？
│
├─ 热点读（少数 key 高频） ──────► 先走缓存（05 多级 / L1），副本治不了热点 key
├─ 海量非热点读（吞吐型） ──────► 读写分离：写主库、读路由只读副本
│        ├─ 可容忍秒级陈旧（列表 / 报表 / 检索） ──► 默认路由副本
│        └─ read-your-writes（刚写就读，如下单后查单） ──► 该请求回主 / 会话粘主 / 等版本
└─ 副本延迟监控 ──► 延迟超阈值 → 该副本摘出读池（退主库 / 他副本），不读过旧数据
```

> **副本延迟是读写分离的第一性代价**：异步复制下副本恒落后主库一个延迟窗口。纪律是"**默认放宽到副本、对'读自己刚写'显式回主**"——别一刀切全读副本（踩 read-your-writes），也别因噎废食全读主（放弃扩展）。延迟窗口取舍与 [`04 §3.2`](04_transaction_design.md) 弱一致兜底同构；缓存（05）挡热点、副本扩吞吐，二者互补不互替。复制拓扑 / 只读连接的表示真相源在 [`docs-spec/05`](../docs-spec/05_schema_design_spec.md)。

### 3.8 存储引擎 / 访问模式匹配 + OLTP/OLAP 分离（RUM 放大权衡）

```
存储引擎 / 存储选型怎么匹配访问模式？（先认清"读放大 / 写放大 / 空间放大"三选二）
│
├─ 读多写少、点查 / 范围扫 ──► B-tree 系（多数 OLTP 默认）：读放大低，写就地更新有写放大
├─ 写多读少、高写入吞吐 ────► LSM-tree 系：写放大低（顺序追加 + 后台 compaction），读放大稍高
├─ 分析 / 聚合 / 宽表扫少列 ─► 列存：扫指定列、高压缩比；点写 / 点查不适合（与 OLTP 分离）
└─ 全文 / 模糊检索 ─────────► 外置检索引擎（倒排索引），别在 OLTP 库上 LIKE 全表扫
```

```
分析查询拖垮在线库怎么办？──► OLTP / OLAP 分离
  └─► OLTP 主库只服务在线点查 / 短事务 → 经 CDC（08 §3.3）把变更流同步到 OLAP 列存 / 数仓
       ↳ 分析 / 报表 / 大聚合走 OLAP，不在 OLTP 库上跑（避免大查询挤占在线连接与 buffer pool）
       ↳ 与读副本（§3.7）互补：副本扩"同构 OLTP 读吞吐"，OLAP 扩"异构分析负载"
```

> **RUM 猜想（Read / Write / Space amplification 三选二）**：没有同时把读放大、写放大、空间放大都做到最优的结构——B-tree 优读放大、LSM 优写放大、列存 + 压缩优空间但牺牲点写。**按"读写比 + 查询形态"选引擎，是 §3.5 索引"写放大约束"在存储引擎层的放大**。引擎选型 / CDC 拓扑的表示落 [`docs-spec/05`](../docs-spec/05_schema_design_spec.md)；CDC 同步决策见 [`08 §3.3`](08_messaging_design.md)。OLAP 分析负载本身不在本支柱在线高性能范畴，本节只给"何时该把它从 OLTP 剥离"的判定。

## 4. 默认选型与升级路径

| 场景 | 默认选型 | 升级触发 → 升级到 |
| --- | --- | --- |
| 关系建模 | 范式化 + 外键约束 | 读放大严重 → 受控冗余 / 物化 |
| 检索 | 单列 / 复合索引（最左前缀） | 读 IO 紧 → 覆盖索引(免回表) + ICP；长列 → 前缀索引；全文 → 检索引擎外置 |
| 容量 | 单表不分 | 海量 → id%N；纯追加 → 按月分表 |
| 库划分 | 单库 | 写风暴 / 强一致混杂 → 按读写模式分库 + 持久化分级 |
| 海量小映射 | 独立行 | 数亿条 → Hash 分桶 |
| 分页 / 大结果集 | OFFSET / LIMIT | 深分页（offset 大）→ keyset 游标分页；全量导出 → 流式分块拉取 |
| 读吞吐 | 缓存挡热点（05） | 海量非热点读吞吐不足 → 读写分离 + 读副本路由（read-your-writes 回主，§3.7） |
| 存储引擎 | B-tree（OLTP 默认） | 写多 → LSM 系；分析宽表 → 列存；全文 → 外置检索（§3.8） |
| 分析查询 | 在 OLTP 库直跑 | 拖垮在线 → CDC 同步 OLAP 列存 / 数仓分离（§3.8） |

## 5. 权衡矩阵

| 方案 | 一致性 | 读性能 | 写成本 | 演进 / 复杂度 | 适用 |
| --- | --- | --- | --- | --- | --- |
| 范式化 | 高 | 中 | 低 | 中 | 默认 |
| 受控冗余 | 最终 | 高 | 中 | 高（需同步） | 读放大热点 |
| id%N 水平分表 | — | 高（并行 IN） | 中 | 中（需路由函数） | 海量结构化 |
| 按月分表 | — | 高 | 低（纯追加） | 低（预建+归档） | 流水 |
| Hash 分桶映射 | — | 高 | 低 | 中 | 海量小映射省内存 |
| keyset 游标分页 | — | 高（恒定，走索引） | — | 中（需有序键 + 游标） | 深分页 / 无界下翻 |
| 读副本路由 | 最终（副本延迟窗口） | 高（扩读吞吐） | 低（写仍走主） | 中（路由 + 延迟监控） | 海量非热点读 |

## 6. 反模式速查

| 反模式 | 正确方向（详见 docs-spec/05 / 01） |
| --- | --- |
| 数据层缺批量方法 → 上层退化循环单查 | 先补 `{...List}` 批量方法再调用（铁律0） |
| 缓存 key 用物理表名 | 用逻辑表名，物理名只在查询期算 |
| 批量查广播到全部分表 | 按 shard 分组只查有请求的表 + 并行 |
| Assembler 内部做 DB 查询 | map 入参纯函数，上层先批量查 |
| 高频查询无支撑索引 / 大表上线才想分库 | 建模期评估容量预留分库键 + 建索引 |
| `SELECT *` / 取多余列 → 走不了覆盖索引 + 多余载荷 | 只取所需列，高频查询走覆盖索引（index-only scan） |
| 过度索引（每个二级索引都是写放大 + 空间） | 按查询路径建索引，定期审无用索引 |
| select 判有无再分别 insert/update | 批量 Upsert（`ON DUPLICATE KEY`/`ON CONFLICT`） |
| 对外暴露真实主键 | 加密 ID（[`05 §9.5`](05_caching_design.md) 确定性加密对外 ID），真实 PK 仅内部用 |
| 深分页用 `LIMIT {大 offset}, n`（扫 N 行丢弃，随页深线性劣化） | keyset 游标分页（`WHERE 有序键 > 游标`） |
| 一次性 load 大结果集进内存（导出 / 遍历） | 流式 / 分块拉取（游标 + 固定 chunk） |
| 全部读无脑路由副本（含"读自己刚写"） | read-your-writes 请求回主 / 粘主 / 等版本（§3.7） |
| 上了读副本却不监控复制延迟 | 副本延迟超阈值即摘出读池，不读过旧数据 |
| 在 OLTP 库直跑大分析 / 报表聚合（挤占在线连接 + buffer pool） | CDC 同步到 OLAP 列存 / 数仓，分析负载分离（§3.8） |
| 写多场景硬用就地更新引擎致写放大爆 | 按读写比评估 LSM 系（§3.8 RUM 三选二） |
| OLTP 库上 `LIKE` 全表扫做全文检索 | 外置倒排检索引擎（§3.8） |

> 字段演进安全路径、DDL 精度、Redis Key 注册表规则正文见 [`docs-spec/05`](../docs-spec/05_schema_design_spec.md)；在线演进 / 迁移 / 回填的决策见 [`11`](11_evolution_migration_design.md)。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节系统名 / 数值 / 代码为某真实工程实现，**只借纪律不借实现**。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 分库按读写模式分级（系统 B）

| 库 | 量级 | 读写模式 | `innodb_flush_log_at_trx_commit` | 复制 |
| --- | --- | --- | --- | --- |
| 核心库 | <50 万 | 读为主 | 1（强一致） | 半同步 |
| 交易库 | <100 万 | 读写均衡、强一致事务 | 1 | 半同步 |
| 日志库 | 百万级 | 写为主、容忍延迟 | 2（允许丢 1s） | 异步 |
| 流水库 | 亿级/月 | 极高写、纯追加 | 2 | 异步 |

把"亿级写风暴"和"强一致核心"放进不同物理库，写风暴不会拖垮鉴权/交易。

### 9.2 逻辑/物理表名解耦函数

```go
ModTable(base, shardKey, count, width) = base + "_" + fmt("%0*d", width, shardKey % count)
// 缓存 key 用 base（逻辑名）；物理名只在查询期算 → 测试态 N 表降为 2 表，缓存零改动
```

海量结构化表 `id % N`（N=8/16/32/96 按量级选）；纯追加流水按 `_YYYYMM` 月分表。

### 9.3 shard 分组并行（复制循环变量 + 内联兜底提交器）

```go
shardGroups := groupByShard(missedIDs)          // 不广播全 16/32 表
if len(shardGroups) == 1 { queryShard(...) }     // 单分表不入池
else {
  for physical, ids := range shardGroups {
    physical, ids := physical, ids               // 必须复制循环变量
    wg.Add(1)
    gopool.Go(pool, func() { defer wg.Done(); queryShard(physical, ids, &result, &mu) }) // 经内联兜底（03）
  }
  wg.Wait()
}
```

### 9.5 海量小映射 Hash 分桶

```
bucket = crc32.ChecksumIEEE([]byte(field)) % 16_000_000   // 1600 万桶，单桶维持 listpack
key="cname_"+bucket; field=原始名称; value={"1":"id1",...}（一名可多 id，上限 32，存十进制串规避 2^53 精度丢失）
```
为何 Hash 不用海量独立 String：N 条映射共享一个 key 的开销远小于 N 个独立 String（省 robj + dictEntry + TTL 元数据），key 维度统一 EXPIRE。存储细节见 [`05 §参考示例`](05_caching_design.md)。

### 9.6 覆盖索引 + ICP

```sql
-- 覆盖索引：查询所需列 (b,c) 全在索引 (a,b,c) 里 → 不回表
CREATE INDEX idx_cover ON t(a, b, c);
SELECT b, c FROM t WHERE a = ?;            -- EXPLAIN: Using index（index-only scan，不回表）
-- 索引下推 ICP：非前缀条件下推到索引层先过滤，减少回表行数
SELECT * FROM t WHERE a = ? AND c LIKE ?;  -- EXPLAIN: Using index condition
-- 前缀索引（长列）：以选择性换索引体积
CREATE INDEX idx_prefix ON t(longcol(20));
```

### 9.7 keyset 游标分页 vs OFFSET 深分页

```sql
-- ❌ 深分页：扫描并丢弃前 100000 行（随 offset 线性劣化）
SELECT … FROM t WHERE status = 1 ORDER BY id DESC LIMIT 100000, 20;
-- ✅ keyset：游标定位走索引，与页深无关
SELECT … FROM t WHERE status = 1 AND id < {last_id} ORDER BY id DESC LIMIT 20;   -- 游标 = 上页末行 id
-- 排序列非唯一 → 加唯一 tiebreaker 保稳定游标：
--   WHERE (sort_col, id) < ({last_sort}, {last_id}) ORDER BY sort_col DESC, id DESC LIMIT 20
-- 全量遍历 / 导出：同一 keyset 游标循环拉取固定 chunk（如每批 1000），封顶内存
```
