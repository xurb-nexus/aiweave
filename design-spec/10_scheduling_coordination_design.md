# 10 - 定时任务 / 分布式协调设计决策规范

> 回答"面对一个后台 / 周期 / 协调类需求，怎么判它该在**每个副本各跑一份**还是**全局只跑一份**、需不需要 leader 选举 / 分布式锁、怎么把调度做成幂等可重入"。
>
> **核心命题**：前面的视角多以"单进程"为隐含前提（[`03`](03_concurrency_design.md) 的"单后台协程顺序刷新"在单进程下完美）；**一旦多副本部署，"单后台协程"就是 N 个副本各跑一份**——这是个正确性问题，不是性能问题。本视角把"多副本下的单例 / 协调"从 ad-hoc（这里用 SPOP、那里用 cobra）升为系统化的决策透镜。
>
> **边界先声明**：三种执行模式（cobra 外部调度 / cycle 内部间隔 / crontab 定点）、任务清单总表、注册代码模板、异常恢复与容错、SPOP / 分布式锁 / 单进程外部调度三种并行策略等**表示规则的真相源在 [`docs-spec/10`](../docs-spec/10_scheduled_tasks_spec.md)**；goroutine 生命周期 / 关闭排空在 [`docs-spec/20`](../docs-spec/20_concurrency_safety_spec.md) + [`docs-spec/27 §5`](../docs-spec/27_deployment_runtime_spec.md)，本篇只引用不复制。本篇新增决策资产：**单例 vs 每副本判定、协调原语选型（幂等 / 原子认领 / 分布式锁 / leader 选举 / 外部调度）、幂等可重入调度、fencing 防脑裂、调度与一致性 / 缓存的联动**。

---

## 1. 设计目标

- **多副本语义明确**：每个后台 / 周期任务都显式回答"该在所有副本并行跑，还是全局只跑一份"——绝不让"单进程下写的后台协程"在多副本里悄悄跑成 N 份、产生重复副作用。
- **协调代价最小**：能用幂等 / 原子认领规避协调的，绝不上分布式锁 / leader 选举（协调有可用性与脑裂代价）；真需单例时，选最轻的够用原语。
- **可重入 / 可恢复**：任务崩溃重启、调度重复触发都安全——靠幂等与断点续作，而非"假设只跑一次"。
- **不脑裂**：用锁 / leader 时，租约过期 + fencing token 兜"自以为还是 leader 的僵尸"，避免两个实例同时认为自己独占。

## 2. 输入信号识别（需求落入本视角的迹象）

| 迹象 | 典型描述 | 多半属于 |
| --- | --- | --- |
| 周期任务 | "每隔 `{N}` / 每天 `{time}` 跑一次 `{任务}`" | 定时调度 |
| flush / 巡检 | "周期把 `{脏数据}` flush 落库 / 对账 / 清理" | 单例 or 幂等认领 |
| 全局唯一动作 | "每天**只**发一次 `{报表 / 通知}`，多发就重复打扰" | 单例（需协调） |
| 多副本部署 | "本服务跑 `{N}` 个副本 / 实例" | 单例 vs 每副本判定 |
| 预建 / 迁移 | "定时预建下月表 / 触发数据迁移 / 预热" | 单例幂等 |
| 抢占处理 | "多实例竞争处理同一批 `{待办}`，别重复处理" | 原子认领 / 锁 |
| 长任务选主 | "一个常驻任务只能有一个实例在跑（如 compaction）" | leader 选举 |
| 崩溃恢复 | "任务跑一半崩了，重启要能接着 / 重来不出错" | 幂等可重入 |

> 识别口诀：**"每个副本都跑也无害（各刷各的本地 / 靠原子认领去重）"→ 每副本；"多跑一份就重复副作用"→ 单例，再选协调原语；"任务会崩 / 会重复触发"→ 必须幂等可重入。**

## 3. 决策树（强制 / 本篇核心）

### 3.0 先判：这个任务该"每副本各跑"还是"全局单例"？

这是本视角的**第一性分叉**——判错方向，要么重复副作用、要么单点不可用：

```
这个后台 / 周期任务，在 N 个副本上同时跑会怎样？
│
├─ 无害（各刷各的进程内状态 / 靠原子认领天然去重） ──► 每副本各跑（无需协调，最省）
│        例：本地镜像刷新（各副本刷自己的 05 §9.4）；SPOP 认领 flush（原子弹出不重复）
│
└─ 有害（多跑一份 = 重复副作用：重复发通知 / 重复扣费 / 重复迁移） ──► 全局单例（进入 §3.1 选协调原语）
         ↳ 先问能否改造成"每副本安全"：幂等化 / 原子认领 → 能则回到上一支（绕开协调）
         ↳ 不能绕 → §3.1 选最轻的单例协调原语
```

> **反向纪律**：协调（锁 / leader）引入可用性折损与脑裂风险。**"能靠幂等 / 原子认领规避协调的，绝不上分布式锁"**——很多"看似要单例"的任务，改成"每副本跑 + 原子认领去重"后根本不需要协调（[`01 §9.4`](01_io_design.md) 的 SPOP flush 即此典范）。

### 3.1 协调原语选型（从轻到重，每步要有升级理由）

```
确需"全局只生效一份"，选哪种协调？
│
├─ 任务可拆成"认领一条做一条" ──► 原子认领：抢占式领取（SPOP / SELECT FOR UPDATE SKIP LOCKED / 原子状态跃迁）
│        ↳ 多副本并行各领各的、不重复，天然无需独占锁 —— 最优先（把"单例"消解成"原子认领"）
│
├─ 一批动作必须整体只跑一次（不可拆） ──► 分布式锁（带租约 TTL + fencing token）
│        ↳ 抢到锁的副本跑、抢不到的跳过；锁带 TTL 防持锁者崩溃后死锁；fencing 防租约过期的僵尸写
│
├─ 一个常驻角色长期只能一个实例担当 ──► leader 选举（租约续约；leader 挂了重新选）
│        ↳ 适合"长生命周期单例"（compaction / 单点调度器）；比"每次抢锁"省抢占开销
│
└─ 调度本身就该外部触发一次 ──► 外部调度器（如 K8s CronJob / 集中调度平台）触发单实例运行
         ↳ 把"单例"外移给编排层：每次拉起一个一次性 Pod 跑完退出，进程内不常驻（cobra 模式同源）
```

> **四种原语的取舍**：原子认领**无独占**（并行 + 去重，可用性最好）；分布式锁**短临界独占**（一批动作原子）；leader 选举**长角色独占**（省反复抢占）；外部调度**把单例外包给编排层**（进程内最简）。优先级恒为 **原子认领 > 外部调度 > 分布式锁 > leader 选举**（越往右协调越重、脑裂面越大）。三种并行策略（SPOP / 分布式锁 / 单进程外部调度）的表示规则见 [`docs-spec/10 §7`](../docs-spec/10_scheduled_tasks_spec.md)。

### 3.2 幂等可重入调度（崩溃重启 / 重复触发都安全 / 强制）

```
任务会崩溃重启、会被重复触发，怎么保证不出错？
│
├─ 断点续作 ──► 记录进度位点（处理到哪）；重启从位点续，不从头重跑（大任务必须）
├─ 幂等生效 ──► 每步以业务键去重 / 状态机单向跃迁；重复执行影响 0 行（与 08 §3.4 消费幂等同构）
└─ 边界对账 ──► 弱一致任务（flush / 同步）用不变量校验兜重复 / 遗漏（与 04 §3.3 边界兜底同构）
```

> 任务崩溃恢复 / 锁未释放兜底 / 数据不一致修复的表示规则见 [`docs-spec/10 §5`](../docs-spec/10_scheduled_tasks_spec.md)。**"假设只跑一次"是后台任务的头号 bug 源**——调度器重试、副本重启、网络分区都会让任务重复触发，幂等可重入是默认要求不是可选项。

### 3.3 防脑裂：租约 + fencing token（用锁 / leader 时强制）

```
持锁 / leader 的实例"卡了一下"（GC / 网络分区），租约过期被别人抢走，它醒来还以为自己独占？
│
├─ 锁 / leader 必须有租约 TTL ──► 持有者须周期续约；续不上即自动失去（不能无限持有）
├─ 写操作带 fencing token ──────► 每次获得锁附带单调递增 token；下游写时校验 token，旧 token 的写被拒
└─ 临界区内定期自检仍持有 ──────► 长临界区中途丢锁要能感知并中止，不闷头写到底
```

> **fencing 是脑裂的最后防线**：单靠 TTL 不够——持有者可能在 TTL 过期后、自己还没察觉时继续写。下游用单调 token 拒掉"过期持有者的迟到写"，才真正防住双主。leader 选举 / 租约的表示规则见 [`docs-spec/10 §7.2`](../docs-spec/10_scheduled_tasks_spec.md)。

### 3.4 与一致性 / 缓存 / 部署的联动

```
├─ 与事务（04） ──► flush 类任务用"弱一致 + 边界兜底 + 监控"，SPOP 认领天然多实例安全（04 §3.2 / 9.3）
├─ 与缓存（05） ──► 缓存巡检 / 世代式失效的离线 SCAN 限速、预热错峰由单例 / 每副本任务承载（05 §3.7/§3.8）
├─ 与消息（08） ──► 消费者本质是"每副本竞争消费 + 分区独占"——consumer group 即一种内建协调，无需额外锁
└─ 与部署（27） ──► 常驻任务 / 消费者关闭前必须排空 + 提交位点；leader 退出要让渡（优雅关闭对称，27 §5）
```

> 关闭排空（停常驻 goroutine / 消费者前先提交位点）的表示真相源在 [`docs-spec/27 §5`](../docs-spec/27_deployment_runtime_spec.md) / [`03 §3.4`](03_concurrency_design.md)；本视角的任务退出（leader 让渡 / 锁释放）须接入同一套优雅关闭。

## 4. 默认选型与升级路径（强制）

| 场景 | 默认选型 | 升级触发 → 升级到 | 反向纪律（防过度优化） |
| --- | --- | --- | --- |
| 后台任务多副本语义 | 先判每副本是否无害（§3.0） | 多跑一份有害 → 全局单例选协调原语 | 每副本无害 → **别上协调**（徒增脑裂面） |
| 单例协调 | 原子认领（SPOP / SKIP LOCKED） | 一批动作不可拆 → 分布式锁；长角色 → leader 选举 | 可原子认领 → **别上分布式锁** |
| 周期触发 | 进程内 cycle / crontab（每副本，配认领去重） | 需全局单次 → 外部调度器（cobra / CronJob） | 每副本安全 → 无需外部调度 |
| 崩溃 / 重复触发 | 幂等可重入 + 断点续作 | 大任务 → 位点续作 + 边界对账 | —（"假设只跑一次"禁止，无豁免） |
| 锁 / leader | 租约 TTL | 有迟到写风险 → + fencing token（§3.3） | 无独占需求 → 不用锁 |
| 任务退出 | 接优雅关闭排空（27 §5） | leader → 退出让渡 + 释放租约 | — |

## 5. 权衡矩阵 + 借纪律不借实现

### 5.1 协调原语权衡矩阵

| 原语 | 可用性 | 并行度 | 脑裂风险 | 复杂度 | 适用形态 |
| --- | --- | --- | --- | --- | --- |
| 每副本各跑（无协调） | 最高 | 高 | 无 | 最低 | 各刷各的 / 无副作用 |
| 原子认领（SPOP / SKIP LOCKED） | 高 | 高（并行领） | 无（原子去重） | 低 | 可拆成逐条认领 |
| 外部调度器（CronJob / cobra） | 中（依赖编排层） | — | 低 | 低 | 全局单次、跑完退出 |
| 分布式锁（TTL + fencing） | 中（抢不到即跳过） | 单点 | 中（靠 fencing 兜） | 中 | 一批动作原子单次 |
| leader 选举（租约续约） | 中（重选有窗口） | 单点 | 中（靠 fencing 兜） | 高 | 长生命周期单点角色 |

### 5.2 借纪律不借实现：按"是否可拆 + 角色时长"重选协调（强制）

同一条"全局只生效一份"的纪律，**原语随任务形态变**（[`00 §2.4`](00_design_overview.md)）：可拆逐条 → 原子认领；一批不可拆 → 分布式锁；长角色 → leader 选举；跑完退出 → 外部调度。把"leader 选举"硬套到"每天发一次报表"（一次性动作）是过度设计，把"假设只跑一次"留给"每副本 cycle 任务"是欠设计（会发 N 份）。**可迁移的是"先判单例 vs 每副本、能认领不加锁、用锁必带 fencing、调度必幂等可重入"这套纪律，不是某个锁实现（Redis SETNX / etcd / Zookeeper）。**

## 6. 反模式速查 + 机械闸门（引用不复制）

### 6.1 反模式速查

| 反模式 | grep 锚 | 正确方向（详见 docs-spec/10） |
| --- | --- | --- |
| 后台协程多副本下跑成 N 份（重复副作用） | `R-SCHED-MULTI-REPLICA` | 先判单例 vs 每副本（§3.0），有害则协调 |
| 能原子认领却上分布式锁（过度协调） | （选型审查） | SPOP / SKIP LOCKED 原子认领（§3.1） |
| 分布式锁无 TTL（持有者崩溃即死锁） | `R-SCHED-LOCK-NO-TTL` | 锁带租约 TTL + 续约（§3.1/§3.3） |
| 用锁 / leader 无 fencing（脑裂双主写） | `R-SCHED-NO-FENCING` | 单调 fencing token，下游拒旧 token（§3.3） |
| 任务"假设只跑一次"（重复触发出错） | `R-SCHED-NOT-IDEMPOTENT` | 幂等可重入 + 断点续作（§3.2） |
| 大任务崩溃从头重跑（无位点） | （选型审查） | 记录进度位点，重启续作（§3.2） |
| 常驻任务 / leader 退出不释放 / 不让渡 | `R-SHUTDOWN-LEAK-GOROUTINE` | 接优雅关闭排空 + 释放租约（§3.4 / 27 §5） |
| flush 类任务用分布式锁串行（本可并行认领） | （选型审查） | SPOP 多实例并行 flush（04 §9.3 / 01 §9.4） |

### 6.2 机械闸门下沉（L0）

可机械判定的违规下沉为提交前 Hook（[`00 §2.5`](00_design_overview.md)、[`PRINCIPLES §14`](../PRINCIPLES.md#14-hooks-机制l0-自动化防御--强制)）：**锁无 TTL**（分布式锁获取 grep 不到过期 / 续约参数 → `R-SCHED-LOCK-NO-TTL`）；**常驻任务漏排空**（新增 cycle / crontab 常驻协程却未接 cancel / 关闭 → 复用 `R-SHUTDOWN-LEAK-GOROUTINE`）；**后台协程含全局副作用**（cycle 任务体内有"发通知 / 写外部"且 grep 不到认领 / 锁 → `R-SCHED-MULTI-REPLICA` 软提醒）。"单例 vs 每副本判对没""fencing 全不全"等语义类违规走选型审查（软提醒）。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节系统名 / 数值 / 代码为某真实工程实现，**只借纪律不借实现**——落地工程时按自己的"单例 vs 每副本 + 角色时长"重选协调。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 把"单例"消解成"原子认领"（系统 B flush，最该抄）

```
需求：每 5min 把 Redis 脏标记 flush 落库；服务跑 4 个副本
❌ 单后台协程假设单进程 → 4 副本各跑一份 → 同一批脏数据被 flush 4 次（重复 UPDATE）
❌ 上分布式锁 → 只有 1 副本干活、另 3 闲置，且要处理锁崩溃 / 续约
✅ SPOP 原子认领：4 副本并行 SPopN(dirty, 100) → 弹出即消费的原子性 → 各领各的、不重复、无需任何锁
不变量：Redis.used = DB.SUM(used) + Redis.delta（边界对账兜重复 / 遗漏，04 §9.3）
```
**这是"能认领不加锁"的范本**：看似要单例的 flush，靠 SPOP 原子性变成"每副本安全并行"，协调被彻底消解。

### 9.2 真需单例的全局动作 → 分布式锁 + fencing

```
需求：每天 02:00 生成并发送一份全局对账报表（多发 = 重复打扰，必须全局单次）
锁：SET lock:daily-report {token} NX EX 600           // 抢到才发，TTL 防持有者崩溃死锁
fencing：token = INCR report:fence                    // 单调递增；写报表状态时校验 token ≥ 已记录值，旧 token 拒
抢不到锁的副本：跳过本次（不报错，正常）
```

### 9.3 长角色单点 → leader 选举

```
需求：本地镜像的全局 compaction 只能一个实例跑（长期角色，频繁抢锁浪费）
leader：基于租约选主（如 etcd / Redis 续约）；leader 周期续约，挂了其他副本重选
非 leader：只服务读、不跑 compaction；leader 退出（优雅关闭）主动让渡租约（27 §5 对称）
```

### 9.4 外部调度（把单例外包给编排层）

```
需求：每月 1 号预建下月分表（一次性、跑完即退）
cobra 子命令：go run main.go pre-create-table   // 不常驻进程
K8s CronJob：每月触发拉起一个一次性 Pod 跑完退出 → 单实例语义由编排层保证，进程内零协调
（对应 docs-spec/10 §7.3 单进程外部调度）
```

### 9.5 幂等可重入（崩溃 / 重复触发都安全）

```
断点续作：迁移任务记 last_migrated_id；重启 WHERE id > {last_migrated_id} 续，不从头
幂等生效：每条 UPSERT（ON DUPLICATE KEY），重复执行不产生重复行
边界对账：跑完校验 source.count == target.count，不等则告警 + 增量补（弱一致兜底，04 §3.2）
```
