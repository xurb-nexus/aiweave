# 05 - 缓存策略设计决策规范

> 回答"面对一个需求，怎么选缓存层级、统一访问层、降级方向、Key 与编码、回源与失效"。缓存是把"远端往返"压成"本地命中"的主力（IO 三维度的"往返"维，见 [`01 §1`](01_io_design.md)）。
>
> **边界先声明**：Redis-first / 三级缓存 / Key 模式 / Hash 字段映射 / Flush 机制等**表示规则的真相源在 [`docs-spec/06`](../docs-spec/06_cache_design_spec.md)**；跨实例聚合 / 单飞契约在 [`docs-spec/25`](../docs-spec/25_io_aggregation_spec.md)；熔断在 [`06`](06_resilience_design.md) + [`docs-spec/12`](../docs-spec/12_circuit_breaker_spec.md)，本篇只引用不复制。本篇新增决策资产：统一访问层职责、**读写非对称降级语义**、多级缓存判定、值编码 / Hash 分桶 / 本地镜像 / 加密 ID 的选型。

## 1. 设计目标

- **命中有效**：缓存真正的热点，命中率可估可测。
- **降级方向正确**：缓存故障行为由"缓存是加速层还是数据源"决定，不一刀切。
- **回源安全**：不被穿透 / 击穿 / 雪崩（空值 + 布隆 / 单飞 / 随机 TTL / 概率提前重算 / 兜底）——四类失效各有专属武器（见 §3.4）。
- **存储经济**：海量小映射不用海量独立 key；近静态小表可本地镜像；**计数/存在/集合用概率/位图结构（HLL / Bitmap / Roaring）而非存原值**；集合/对象控在紧凑编码阈值内（§3.5）；**值编码先选序列化格式（可调试 / 紧凑二进制 / 零拷贝）再叠压缩（§3.3）**。
- **分片可扩展**：容量随 Redis 分片数**线性扩展**——前提是无大 key / 无热 key / key 均匀分布（§3.6）。
- **键生命周期有界**：每个缓存 key 必须有界 TTL，杜绝无界增长 → 淘汰风暴；冷启 / 扩容前预热 + 启动错峰，避免空缓存启动风暴（§3.8）。

## 2. 输入信号识别

| 迹象 | 典型描述 |
| --- | --- |
| 读热点 / 计算昂贵 | "`{entity}` 高频读，底库压力大""`{result}` 计算代价高可复用" |
| 跨实例 | "缓存分布在多个实例 / 需分组读" |
| 一致性敏感 | "底库变了，缓存必须及时反映" |
| 海量小映射 | "`{name}` → `{id}` 映射数亿条" |
| 近静态小维度表 | "字典 / 枚举 / 分类，≤1 万行、可容忍陈旧、被高频 join" |
| 对外 ID | "响应里要暴露主键，怕被顺序刷库" |
| 计数 / 存在 / 集合 | "统计有多少不同 `{X}` / `{X}` 在不在 / 两集合交并" |
| 单 key 过大 / 超热 / 需扩容 | "某 key value 很大 / 单 key QPS 极高 / 容量要随分片增长" |
| 恒不存在 key 查询 | "大量查询的 `{key}` 在底库根本不存在（误查 / 被刷库）" |
| 热路径反复编解码 / 大载荷 | "缓存值高频 marshal / unmarshal，或单值文本载荷大" |
| 批量失效一整类 | "改了 `{规则 / 维度}`，要失效一整类 `{entity}` 的缓存" |
| 冷启 / 扩容 | "实例重启 / 扩容后缓存空，需避免启动回源风暴" |

## 3. 决策树（强制）

### 3.1 缓存层级与必要性

```
该不该缓存 / 缓存几级？
│
├─ 先判必要性 ────────► 命中率低 / 一致性要求极高 → 不缓存
├─ 万级 + 低频变更的判定 key ─► L1 进程内（存 struct 非 map，0 序列化 0 网络，TTL 自然失效，不主动广播）
├─ 普通读热点 ───────► L2 Redis（Cache-Aside 旁路 + 失效驱动）
├─ 跨实例热点 ───────► 按实例分组批量读（聚合器，见 01 §3.1）
└─ 近静态小维度表 ──► 本地镜像（进程内 SQLite），本地命中绕过 Redis 与一切外层缓存
```

**本地缓存准入条件（L1 进程内 KV / 强制）**：L1 在**每个实例各存一份**（内存 × N 实例）、且**只靠 TTL 自然失效、无跨实例失效广播**——这两条根因推导出准入门槛，须**同时满足**：

| 准入条件 | 根因 |
| --- | --- |
| 数据量有界且小（万级） | L1 每实例各一份 = 内存 × N 实例 |
| 低变更频率 | 无跨实例失效广播 → 变更越频，实例间越不一致 |
| 可容忍陈旧（最多一个 TTL） | 每实例靠 TTL 自然过期，业务须能接受这段陈旧 |
| 高频读 | 用大量读摊薄"× N 实例内存"的成本 |
| 非强一致 / 不要求 read-your-writes | L1 无法保证全实例瞬时一致 |
| 低基数判定类 key | 非每实体 / 每会话 / 组合键 |

> **排除清单（绝不放 L1）**：高基数 key（每实体 / 每会话 / 组合键）→ 内存 × N 爆；高频变更（计数器 / 限流 / 累加类状态）→ 陈旧 + 实例发散；强一致 / read-your-writes 需求；大对象。
>
> 本地 SQLite 镜像（关系型，§3.1 决策树末项）准入更窄：`≤ 约 1 万行 + 近静态（可容忍陈旧 ≤ 一刷新周期）+ 被高频 join + 不分表`。

### 3.2 读写非对称降级语义（强制 / 最该抄的一条）

```
缓存熔断打开 / 失败时怎么降级？取决于缓存的【定位】：
│
├─ 加速层（miss 可回源 DB） ──► 读：返回"未命中"→ 自然穿透 DB；写：静默跳过返回 nil
│                                 → "宕机"与"未命中"行为完全同构，业务无需感知
└─ 核心数据源（miss = 拒绝） ──► 快速失败（如 503），绝不穿透 DB（静默降级会返脏/返空）
```

> **写永远 best-effort（失败静默）；读的降级取决于背后有没有兜底数据源。** 降级方向不可一刀切。

### 3.3 存储形态与编码

```
├─ 海量小映射 ──► Hash 分桶（CRC32 取模控桶体积）替代海量独立 String key（省 robj/dictEntry/TTL 元数据）
├─ 值序列化格式 ──► 按读写比 / 可调试性 / 体积选：文本（可调试，默认）/ 紧凑二进制（省体积 + marshal CPU）/ 零拷贝（读密集免解码）
├─ 结构化大值 ──► 自适应压缩（小值不压 / 大值不写双阈值 + 1 字节自描述头）
├─ 批量读写 ────► MGET/MSET + Pipeline 超阈值自动分批（对调用方透明）
├─ 防雪崩 ──────► TTL 随机抖动（打散集体过期）
└─ 对外 ID ─────► 确定性加密（可缓存可比较）；解密失败统一"查无结果"无差分
```

> **先定格式再叠压缩**：序列化格式决定 marshal CPU 与 GC（热路径成本），压缩只减网络 / 内存字节——两件事正交。文本便于调试与跨语言，紧凑二进制（如 protobuf / msgpack）省体积省 CPU，零拷贝（如 flatbuffers）适读密集免解码。

### 3.4 回源保护：四类回源风暴分治（每类一种武器，互补不替代）

四种"回源风暴"是**不同的失效形态**，各有专属武器，叠加使用，不可互相替代：

| 失效形态 | 触发 | 武器 |
| --- | --- | --- |
| 并发 miss 同一 key | 多请求同时回源刚失效的同一 key | **single-flight**：合并成一次回源，其余等待共享结果（详见 [`01 §3`](01_io_design.md) / [`docs-spec/25 §3`](../docs-spec/25_io_aggregation_spec.md)） |
| 同批 key 集体过期（雪崩） | 同时写入的一批 key 同时到期 → 瞬时全部穿透 | **TTL 随机抖动**：打散到期时刻（§3.3 / §9.2） |
| 单热 key 到期悬崖（击穿） | 一个超热 key 到期，即便 single-flight，其余请求仍要等那一次回源 → 延迟尖刺 | **XFetch 概率提前重算**：临近 TTL 按概率提前刷新，热 key 永不真到期（§9.6） |
| 恒不存在 key（穿透） | 大量查询底库根本不存在的 key → 每次必穿透回源（缓存对它永远 miss） | **空值缓存**（短 TTL 占位"查无"，吸收重复查询）＋ **布隆 / 布谷过滤器**（前置闸门，判不存在直接拒，免查缓存与底库） |

> **XFetch 决策要点**：把"上次回源耗时" `{delta}` 与值一起缓存；每次读时，**越临近过期、回源越贵，越可能由某个幸运读者提前重算**，其余读者继续用尚未过期的旧值。用极小的"提前重算概率"换掉热 key 到期那一下的延迟悬崖。`{beta}` 调早晚（默认 1，越大越早刷）。与 single-flight、随机 TTL 三者**正交叠加**。
>
> **穿透 vs 其余三类**：前三类是"已存在的 key 在失效 / 到期那一刻的回源风暴"，穿透是"key 根本不存在、缓存永远兜不住"。**数据源型缓存（miss = 拒绝，§3.2）天然免疫穿透**；只有加速层（miss 回源）需要空值缓存 + 布隆闸门。空值缓存挡"重复查同一不存在 key"，布隆挡"海量不同的不存在 key"，二者互补。

### 3.5 极致省内存：紧凑编码阈值 + 概率/位图结构

两类"用一点限制换数量级省内存"的手段：

**① 紧凑编码阈值治理**：Redis 集合类型在元素数/值长低于阈值时用紧凑编码（`listpack` / `intset` / `ziplist`），超阈值退化为 hashtable/skiplist，内存翻数倍。**主动把单 key 的元素数/值长压在阈值内**（§3.3 的 Hash 分桶"控桶体积保 listpack"即此原理的分片应用）。

**② 概率/位图结构选型**：需求是"统计基数 / 判断存在 / 集合运算"而非取回原值时，别存原始集合：

| 需求 | 结构 | 极致点 |
| --- | --- | --- |
| 海量去重计数（可容忍 ~1% 误差） | **HyperLogLog** | 固定约 `{HLL-bytes}`（~12KB）顶亿级基数 |
| 稠密整数 ID 的 membership / 标志位 | **Bitmap** | 1 bit/元素 |
| 稀疏整数 ID 集 + 快速交并（标签 / 权限 / 特征） | **Roaring Bitmap** | 分块自适应压缩 + 快速 AND/OR，远省于存原始 ID 列表 |

> **判定**：需精确值 / 取回成员 → 普通结构；只问"多少个 / 在不在 / 交并" → 概率/位图结构。HLL 有误差且不可取回成员；Bitmap 适稠密整数；Roaring 适稀疏。

### 3.6 分片可扩展性：大 key / 热 key 治理 + 多分片利用

> **容量线性律**：缓存容量要随 Redis 分片数**线性扩展**，三前提缺一不可——① key 在分片间**均匀分布**；② **无大 key**；③ **无热 key**。大 key / 热 key 正是击穿分片扩展性的元凶：大 key 困在单分片无法拆，热 key 流量全砸一个分片，加再多分片也分不走。

**A. 大 key 治理**（让单分片成瓶颈、无法横向拆）

```
识别：单 value 过大（String > `{big-value}` / 集合元素数 > `{big-entries}` / 集合总字节 > `{big-bytes}`）
危害：① 占单分片内存、无法跨分片拆 ② 大 value 操作阻塞 Redis 单线程（慢命令拖垮整分片）
      ③ DEL 大 key 阻塞 ④ 扩缩容 slot 迁移大 key 慢/卡
治理：
  ├─ 拆分 ──► 大集合按子键拆（Hash 分桶 / `{key}:{part}`）；大 String 分块（§3.3 分桶即此原理）
  ├─ 阈值 ──► value 写入硬上限（§9.2）；集合元素数控在紧凑编码阈值内（§3.5）
  ├─ 删除 ──► 大 key 用 `UNLINK`（异步惰性删）不用 `DEL`；大 key 过期分批清
  └─ 检测 ──► `--bigkeys` / `MEMORY USAGE` 采样巡检 → 登记 + 告警
```

**B. 热 key 治理**（让单分片 CPU 饱和、加分片分不走）—— 按读热 / 写热分治

```
识别：单 key QPS 远超均值（占某分片流量大头）
读热：├─ L1 本地缓存 ─► 进程内挡住热 key 读流量（§3.1，准入条件已定）—— 最有效
     ├─ 读副本 ─────► 热 key 读分散到多 replica
     └─ key 复制打散 ► `{hotkey}:{0..K}` K 副本随机读 → 单 key 流量摊到 K 个 key（落不同分片）
写热：├─ 本地聚合 + 批量 flush ─► 本地累加、周期合并写（01 §3.2 delta flush 即此）
     └─ 分片计数 ─────────────► `{counter}:{0..K}` 拆 K 个分片计数、读时求和 → 写流量摊到 K 分片
检测：`--hotkeys`（需 LFU）/ `OBJECT FREQ` / 客户端采样单 key QPS
```

**C. 多分片利用（容量可扩展）**

```
├─ 分片模型 ──► Redis Cluster（16384 slot，CRC16(key)%16384）或客户端分片（一致性哈希）
├─ 均匀分布 ──► key 设计让散列均匀、无倾斜，避免大量 key 挤进少数 slot
├─ Hash Tag ─► `{tag}` 强制同 tag 同 slot（为跨 key MGET/事务/Lua 原子）——双刃：滥用造热点 slot；
│              仅在必须跨 key 原子时用、且 tag 基数要大
├─ 跨分片批量 ► MGET/Pipeline 按分片分组并行（聚合器"按实例分组批量读"，01/05）—— 别串行
└─ 扩缩容迁移 ► 一致性哈希（节点增减只迁 K/N）/ Cluster slot 迁移；**禁裸 `mod N`**（全量重哈希）
```

> **三前提任一破，容量卡在最热 / 最大那一个分片**——分片可扩展不是"加机器"，是"先消灭大 key / 热 key / 倾斜"。

### 3.7 批量失效：版本号 / 世代式 + 杜绝键空间扫描（强制）

"失效一整类实体的缓存"绝不要枚举删 key——枚举即 `KEYS` / `SCAN`，阻塞或拖垮整实例：

```
要失效一批 / 一整类 key？
│
├─ 单 key 失效 ───────► 直接 Delete（改库后删，见 §4 写一致）
├─ 一整类批量失效 ───► 版本号 / 世代式：key 带版本前缀 `v{n}:{…}`；失效 = bump 版本号（一次 INCR）
│                        ↳ 旧版本 key 无人再读 → 靠 TTL 自然过期回收（故须配有界 TTL，§1 / §3.8）
│                        ↳ "换代"替代"逐个删"：把批量失效从 O(N) 降到 O(1)，不碰存量 key
└─ 必须枚举现存成员 ──► 维护显式索引集（Set 记成员）按集合遍历；**严禁热路径 `KEYS`**，离线巡检才用 `SCAN` 限速游标
```

> `KEYS` 全量阻塞 Redis 单线程（慢命令拖垮整实例）；世代式失效用一次版本号自增"逻辑清空一代"，是把批量失效降到 O(1) 的应用层手段。Key 模式 / 版本前缀的表示登记真相源见 [`docs-spec/06 §4`](../docs-spec/06_cache_design_spec.md)。

### 3.8 键生命周期：有界 TTL + 冷启预热防启动风暴（稳定性纪律）

```
├─ TTL 纪律 ──► 每个缓存 key 必须有界 TTL（数据源型常驻 key 例外，但须登记）；杜绝无界增长 → 内存涨满触发淘汰风暴
├─ 冷启预热 ──► 实例重启 / 扩容后缓存空 → 启动期先预热热点 key 再就绪（readiness 晚于预热），避免空缓存全员回源
└─ 启动错峰 ──► 多实例同时启动 → 预热 / 首次回源加随机抖动 + 分批就绪，打散启动瞬时回源洪峰
```

> 预热触发时机 / 顺序的表示真相源见 [`docs-spec/06 §6`](../docs-spec/06_cache_design_spec.md)（缓存预热）；加速层"miss 与宕机同构"（§3.2）在预热缺失时会退化成启动回源风暴，故预热是加速层冷启的稳定性前提。

## 4. 默认选型与升级路径

| 场景 | 默认选型 | 升级触发 → 升级到 |
| --- | --- | --- |
| 读热点 | Cache-Aside + 失效 | 极热点判定 key → L1 进程内 + L2 Redis 多级 |
| 回源保护 | 单飞 + 随机 TTL | 单热 key 到期悬崖 → XFetch 概率提前重算；强击穿 → 逻辑过期 / 预热 |
| 写策略 | cache-aside 失效优先（改库后删缓存） | 写后即读 / 一致性敏感 → write-through（同步写穿缓存 + 库）；写聚合削峰（计数热点）→ write-back（写缓存异步 flush 库，见 04 §3 / 01 §3.2）；强一致 → 延迟双删 / 订阅 binlog |
| 访问方式 | 统一缓存访问层（收编编解码+熔断+降级） | —（禁止业务裸碰 Redis） |
| 近静态小表 | Redis 缓存 | ≤1 万行 + 高频 join → 本地 SQLite 镜像 |
| 海量小映射 | 独立 String key | 数亿条 → Hash 分桶 |
| 计数 / 存在 / 集合 | 普通结构存原值 | 去重计数 → HyperLogLog；整数集 membership/交并 → Bitmap/Roaring |
| 集合/对象编码 | 默认 | 撑破紧凑编码阈值 → 控元素数/值长保 listpack/intset |
| 大 key | 默认 | 单 value 过大 → 拆分（分桶 / 分块）+ `UNLINK` 删除 |
| 热 key | 默认 | 读热 → L1 / 读副本 / `{hotkey}:{0..K}` 打散；写热 → 本地聚合 flush / 分片计数 |
| 容量扩展 | 单实例 | 容量瓶颈 → 多分片（前提：无大 key / 无热 key / 均匀分布，§3.6） |
| 穿透防护 | 数据源型 miss = 拒绝（已免疫） | 加速层遇恒不存在 key → 空值缓存 + 布隆 / 布谷闸门（§3.4） |
| 值序列化格式 | 文本（可调试） | 体积 / marshal CPU 紧 → 紧凑二进制（protobuf / msgpack）；读密集免解码 → 零拷贝 |
| 批量失效 | 逐 key Delete | 失效一整类 → 版本号世代式 bump；严禁 `KEYS` 枚举（§3.7） |
| 缓存预热 | 懒加载（首次 miss 回源） | 冷启 / 扩容空缓存启动风暴 → 启动预热 + 实例就绪错峰（§3.8） |

| 方案 | 一致性 | 读性能 | 复杂度 | 适用 |
| --- | --- | --- | --- | --- |
| Cache-Aside（旁路） | 最终 | 高 | 低 | 默认读热点 |

| Hash 分桶 | — | 高 | 中 | 海量小映射省内存 |

## 6. 反模式速查

| 反模式 | 正确方向（详见 docs-spec/06 / 25 / 12） |
| --- | --- |
| 业务代码裸碰 Redis 客户端 | 统一访问层收编编解码 + 熔断 + 降级 |
| 降级方向一刀切 | 按"加速层 / 数据源"定降级（§3.2） |
| 缓存命中率低却强行缓存 | 先判必要性 |
| 回源无单飞（击穿） | single-flight 合并同 key 回源 |
| 单热 key 到期无提前重算（延迟悬崖） | XFetch 概率提前重算（§3.4） |
| 同批 key 集体过期（雪崩） | TTL 随机抖动 |
| 海量独立 String key | Hash 分桶 |
| 大集合存原始成员只为算"有多少 / 在不在" | 基数 → HLL，membership/集合 → Bitmap/Roaring（§3.5） |
| 集合/大对象撑破紧凑编码阈值退化重编码 | 控元素数/值长在 listpack/intset 阈值内（§3.5） |
| 本地命中还叠 Redis 读写 | 本地镜像即最快进程内缓存，绕过 Redis（双写产生两份陈旧度 → 脏读） |
| 高基数 / 高频变更 key 塞进 L1 进程内 | 不满足 L1 准入（万级 + 低频变更 + 可容忍陈旧，§3.1）→ 走 L2 Redis |
| 大 key（大 value / 海量元素集合）用 `DEL` 删 | `UNLINK` 异步删；大 key 先拆分（§3.6 A） |
| 热 key 流量全砸单分片（加分片也分不走） | L1 / 读副本 / `{hotkey}:{0..K}` 打散 / 分片计数（§3.6 B） |
| 滥用 hash tag 把大量 key 钉到少数 slot（热点 slot / 倾斜） | 仅必须跨 key 原子时用，tag 基数要大（§3.6 C） |
| 跨分片 MGET / Pipeline 串行逐分片 | 按分片分组并行（聚合器，01 §3.1） |
| 节点增减用裸 `mod N` 重哈希（全量迁移） | 一致性哈希 / Cluster slot 迁移（§3.6 C） |
| 跨实例逐条读 | 按实例分组批量（01 §3.1） |
| 解密失败返 4xx / 携带失败细节 | 统一"查无结果"，消除可观测差分 |
| 恒不存在 key 反复穿透回源（穿透） | 空值缓存（短 TTL 占位）+ 布隆 / 布谷前置闸门（§3.4） |
| 热路径反复 JSON 编解码吃 CPU + GC / 大文本载荷 | 按数据形态选序列化格式：紧凑二进制 / 零拷贝（§3.3） |
| 用 `KEYS` / `SCAN` 枚举删一批 key（阻塞单线程） | 版本号世代式失效 / 维护索引集（§3.7） |
| 批量失效逐个 Delete | bump 版本前缀换代（§3.7） |
| 缓存 key 无 TTL / 无界增长（→ 淘汰风暴） | 每 key 有界 TTL（§3.8） |
| 实例冷启空缓存全员同时回源（启动风暴） | 启动预热 + 就绪错峰（§3.8） |
| 写聚合热点用同步 write-through（每写穿库，失削峰意义） | 计数 / 高频写 → write-back 写缓存异步 flush（04 §3 / 01 §3.2） |
| 一致性敏感写仍用 cache-aside 失效（读到旧值窗口） | write-through 同步写穿，或延迟双删 |

> Key 模式 / Hash 字段映射 / Flush 规则正文见 [`docs-spec/06`](../docs-spec/06_cache_design_spec.md)；跨实例聚合 / 单飞契约见 [`docs-spec/25`](../docs-spec/25_io_aggregation_spec.md)；熔断接入见 [`06`](06_resilience_design.md)。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节系统名 / 数值 / 代码为某真实工程实现，**只借纪律不借实现**。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 统一缓存访问层 API + 非对称降级

```go
Get(ctx,key)(json string, hit bool, err error);  SetWithRandomTTL(ctx,key,json,baseTTLSeconds)
MGet(ctx,keys)(map[string]string,error);  MSet(ctx,items []SetItem);  HGet/HSet/Delete/Do/Pipeline...
```
熔断器以"底层连接池指针为 key"挂注册表、懒加载、实例级隔离。降级：

| 缓存定位 | 熔断打开 / 失败 | 实例 |
| --- | --- | --- |
| 加速层（miss 回源 DB） | 读返回 `("",false,nil)`，写静默 `nil` → 宕机与未命中同构 | 系统 A |
| 核心数据源（miss=拒绝） | 快速失败 503，绝不穿透 DB | 系统 B 鉴权 |

### 9.2 TTL 抖动 + 值编码

```
实际TTL = base + rand.Int64N(floor(base*0.15)+1)   // 打散 15%。实测：结构化 12h / 关系 3d / 映射 1d
值编码：[1 字节 flag][payload]  flag=0x00 原文 / 0x01 LZ4
       <128B 不压（避免膨胀），≥128B 走 LZ4（sync.Pool 复用零 GC）；写入硬上限 16KB 超限静默跳过
MGet keys>64 自动按 64 切批 Pipeline 合并，未命中 key 不进结果 map（调用方按 _,ok 分流）
```

### 9.3 Hash CRC32 分桶（海量小映射）

```
bucket = crc32.ChecksumIEEE([]byte(field)) % 16_000_000   // 1600 万桶，field 数≈总实体/16M 保 listpack
key="cname_"+bucket; field=名称; value={"1":"id1",...}（一名多 id 上限 32，id 存十进制串规避 2^53 丢精度）
```

### 9.4 本地 SQL 缓存（进程内 SQLite 镜像 · 可复用组件蓝图）

近静态小表整表镜像，远端往返(~1-3ms)→本地索引查找(~1-50µs，100-1000×)。关键细节：
- 纯 Go 驱动（`modernc.org/sqlite`）保 `CGO_ENABLED=0`；WAL 单写多读（读池长存保温、**写池 `MaxOpenConns=1`** 钳并发，详见 [`03 §9.4`](03_concurrency_design.md)）。
- 单事务原子替换（`DELETE`+分块`INSERT`，读者 COMMIT 前看旧快照、后瞬间切新，永不读半截）；`ready.Store(true)` 在 COMMIT 之后（happens-before）；刷新失败回滚继续服务旧数据。
- 单后台协程每 24h 顺序刷新全部表，每 7 次顺带 `VACUUM`；本地命中绕过 Redis（双写产生两份陈旧度）。
- 三层 fail-open：本地（就绪）> Redis > MySQL，每层故障安全跌落下一层。

**组件骨架（按职责拆 6 件，整体可迁移）**——把"本地镜像"沉淀成可复用组件而非一次性代码：

| 部件 | 职责 |
| --- | --- |
| registry | 每表注册项（源库句柄 / 模型 / 索引 / 首行超时）+ `ready` 原子位；注册集中一处可见 |
| engine | 单 SQLite 文件：`readDB`（多连接）/ `writeDB`（单连接）+ 建库 PRAGMA（WAL / cache_size / mmap）+ AutoMigrate 建表 |
| loader | 整表流式拉取（首行到达超时则跳过本表、走远端）→ 单事务原子替换写本地 |
| refresher | **唯一后台协程**：首次顺序加载 → 每统一间隔刷新全部表 → 每 N 次刷新顺带 VACUUM |
| maintainer | `compact()`：VACUUM + `wal_checkpoint(TRUNCATE)`，由 refresher 调用（非独立协程） |
| api | `DB(name)→(*handle, ready)` 路由契约：未就绪 / 未注册 / 开关关 → 回落远端 |

**原子发布不变量（本组件最该守的一条）**：`ready` **只能在整表替换事务 COMMIT 成功后**置位。读者只读 `ready`，WAL 下读事务一开始即钉住"最末一条已提交帧"快照；由 `COMMIT happens-before ready.Store(true) happens-before 读者读到 true happens-before 读者开读事务`，读者要么走远端 / 读到**完整旧版本**、要么读到**完整新版本**——**绝无半截、绝无"已切本地却读空"**。违反（COMMIT 前置位 / 跨刷新持长读事务不重开）即撕裂可见性 → `R-LOCALMIRROR-STALE-READY`。

### 9.5 确定性加密对外 ID

```
plain → PKCS7 → AES-128-CBC(IV = HMAC-SHA256(ivKey, plain)[:16]) → base64url → "A"+密文
```
确定性 IV（同明文→同密文）→ 客户端可当稳定键持久化、测试可 Decrypt 反验；`ivKey=HMAC(masterKey,"...-iv-v1")` 隔离子密钥；版本前缀预留轮换；空/零值短路返 `""`。**解密失败统一降级为"查无结果"（返 201/空），绝不返 4xx、绝不携带失败细节**——否则响应差分让加密层退化为预言机，可被批量探测有效 token（机械闸门 `R-SEC-ID-PLAINTEXT` / `R-SEC-DECRYPT-ORACLE`，表示纪律见 [`docs-spec/14 §9.5`](../docs-spec/14_status_codes_spec.md)）。

### 9.6 XFetch 概率提前重算（防单热 key 击穿）

缓存里和值一起存：`value`、绝对过期时刻 `expiry`、上次回源耗时 `delta`。每次读时算：

```
若  now − delta × beta × ln(rand())  ≥  expiry   → 由本次读者提前重算并刷新（rand ∈ (0,1]，beta ≥ 1 默认 1）
否则                                              → 直接返回缓存旧值
```

越临近 `expiry`、`delta` 越大（回源越贵），触发提前重算的概率越高 → 在到期前就被某个读者刷新，热 key 永不出现"全员等一次回源"的延迟悬崖。与 single-flight（防并发 miss）、随机 TTL（防集体过期）三者正交叠加。源：Vattani et al., *Optimal Probabilistic Cache Stampede Prevention*, VLDB 2015。

### 9.7 紧凑编码阈值 + 概率/位图结构

```
# 紧凑编码阈值（redis.conf；超过即退化为 hashtable/skiplist，内存翻倍）
hash-max-listpack-entries 128   hash-max-listpack-value 64
set-max-intset-entries 512      zset-max-listpack-entries 128
# 概率/位图结构
PFADD hll {元素}… / PFCOUNT hll          # HyperLogLog：~12KB 估亿级基数，标准误差 ~0.81%
SETBIT bm {intID} 1 / BITCOUNT bm        # Bitmap：稠密整数 ID 的 membership / 计数
# Roaring：经 RedisBloom/模块或应用层库，稀疏整数 ID 集压缩 + 快速 AND/OR
```

### 9.8 分片可扩展：大 key / 热 key / 多分片

```
# 大 key 检测 + 安全删除
redis-cli --bigkeys ;  MEMORY USAGE {key}     # 采样巡检
UNLINK {bigkey}                               # 异步惰性删（不用 DEL 阻塞单线程）
# 热 key 检测 + 读热打散
redis-cli --hotkeys （需 maxmemory-policy=LFU）;  OBJECT FREQ {key}
读：rand(0..K) → GET {hotkey}:{i}             # 单热 key 复制成 K 份，随机读，落不同分片
# 写热：分片计数（读时求和）
INCR {counter}:{rand(0..K)}                   # 写流量摊到 K 个 key
# 多分片：Hash Tag 仅在必须跨 key 原子时用（强制同 slot）
MGET obj:{1001}:a obj:{1001}:b               # {1001} 是 tag → 同 slot，可 MGET/事务；滥用会倾斜
```

### 9.9 穿透防护：空值缓存 + 布隆闸门

```
# 空值缓存：查无也写一个短 TTL 占位，吸收"重复查同一不存在 key"
GET item:{id} → miss → 回源 DB 查无 → SET item:{id} "\x00NULL" EX 30   # 短 TTL，下次直接命中"查无"
# 布隆闸门：挡"海量不同的不存在 key"，前置判断，免查缓存 / DB
if !bloom.Test(id) { return NotFound }       # 不在布隆集 → 必不存在，直接拒
# 数据源型缓存（miss=拒绝）天然免疫穿透，无需空值 / 布隆
```

### 9.10 版本号世代式失效

```
# key 带版本前缀；失效一整类 = bump 版本号（O(1)，不碰存量 key）
ver = GET dim:ver:{type}                      # 当前代号，如 7
key = "dim:v" + ver + ":" + id                # 读写都拼当前代号
INCR dim:ver:{type}                           # 失效一整类：换代（旧 v7 无人再读，靠 TTL 自然回收）
# ❌ 严禁：KEYS dim:v*  /  热路径 SCAN 枚举删
```

### 9.11 序列化格式选型

| 数据形态 | 格式 | 取舍 |
| --- | --- | --- |
| 调试友好 / 跨语言 / 偶发读 | JSON（文本） | 可读、易排障；体积大、marshal 较慢 |
| 高频读写 / 省体积省 CPU | protobuf / msgpack | 紧凑、快；需 schema / 不可直读 |
| 读密集 / 大对象免解码 | flatbuffers（零拷贝） | 读时不反序列化；写侧略重、布局固定 |

> 先定格式（决定 marshal CPU + GC），再叠压缩（只减字节）；与 §9.2 值编码头正交。

### 9.12 键生命周期 + 冷启预热

```
TTL 纪律：所有缓存 key 走 SetWithRandomTTL（§9.2），无 TTL 的常驻 key 必须登记理由
冷启预热：实例启动 → 预热热点 key（批量 MGET 回填）→ ready.Store(true)（就绪晚于预热）
启动错峰：多实例就绪加随机抖动 + 分批，避免空缓存同时回源（启动风暴）
```
