# 04 - 事务一致性设计决策规范

> 回答"面对一个需求，怎么划事务边界、选一致性模型、设计补偿/幂等，以及何时用弱一致 + 边界兜底换掉强一致的复杂度"。
>
> **边界先声明**：事务模型选型清单、事务边界清单、补偿逻辑、幂等性、一致性窗口、失败路径全景图等**规则的真相源在 [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)**；写路径削峰编排在 [`01 §3.2`](01_io_design.md)，本篇只引用不复制。本篇新增决策资产：一致性强度判定、本地 vs 分布式事务决策树、**弱一致 + 边界兜底 + 监控**的取舍方法、读写分离消除幂等复杂度。

---

## 1. 设计目标

- **边界清晰**：每个写操作的事务边界明确，不变量在边界内成立。
- **一致性恰到好处**：强一致 / 最终一致的选择有据，一致性窗口显式——**不为不需要强一致的场景付强一致的复杂度**。
- **失败可恢复**：跨资源写有补偿，所有写入口幂等（或在架构层规避幂等需求）。

---

## 2. 输入信号识别

| 迹象 | 典型描述 |
| --- | --- |
| 多写聚合 | "一个操作要改多张表 / 多个资源" |
| 跨数据源 | "同时写库和缓存 / 库和 MQ / 多个库" |
| 高频数值变更 | "高频扣减 / 增加 `{核心数值字段}`，不能错账" |
| 重试场景 | "下游可能重复投递 / 客户端可能重试" |
| 读写混杂判定 | "一个接口里既判定资格又改状态" |
| 批量写吞吐 | "大量小事务高频提交 / 写 TPS 受限于 fsync 落盘" |

---

## 3. 决策树（强制）

### 3.1 一致性强度判定

```
这个写的一致性需求有多强？
│
├─ 单库多表写 ───────────► 本地事务（一个 DB 事务包住）
│
├─ 跨数据源写 + 强一致 ──► 分布式事务模型（Saga / TCC / 本地消息表，按补偿能力选，见 21）
│
├─ 跨数据源写 + 可最终一致 ─► 异步补偿 + 对账；或写路径削峰（Redis-first + delta flush，01 §3.2）
│
└─ 高频数值变更 ─────────► 单命令原子性（无锁）在热路径，越界在【持久化边界】兜底，不在热路径拦
```

### 3.2 何时用"弱一致 + 边界兜底 + 监控"（强制取舍）

```
强一致会引入大量复杂度时（预扣 + 回退 + 定时扫描…）：
  └─► 评估"弱一致 + 边界兜底 + 监控"是否够用
       ├─ 架构层读写分离：把"读判定资格"和"写扣费"拆成两个接口
       │     → 重复读天然幂等（重试安全）；漏写=可接受损耗（监控兜底）
       │     → 省掉"预扣 + 回退 + 定时扫描"整套机制
       └─ 持久化边界兜底：热路径不拦越界（原子但允许瞬时越界），落库时夹断 + 记 WARN 当系统损耗
```

> **原则**：当强一致会引入大量复杂度时，优先评估弱一致是否够用。读写分离让重试天然安全，是在**架构层（而非代码层）**消除复杂度的典范。用极小边界损耗换热路径零锁高并发，常比"强一致预扣 + 回退 + 扫描"简单一个数量级。

### 3.3 写吞吐：批量提交 / group-commit（摊销 fsync）

```
写 TPS 卡在"每事务一次 fsync（redo / binlog 落盘）"上？
│
├─ 单条小事务高频提交 ──► 攒批：N 条逻辑写合并进一个事务 / 一次提交，把 N 次 fsync 摊成 1 次
│        ↳ 提交间隔 / 批大小是旋钮：越大吞吐越高、但崩溃丢失窗口越大（已 ack 未落盘的一批）
├─ 持久性可放宽 ──────► 组提交 + 放宽刷盘级别（写密集库，见 02 §9.1 分库分级）
└─ 纯追加极高写 ──────► 出 DB 事务，走 MQ 攒批 + 批量 INSERT（写路径见 01 §3.2）
```

> **group-commit 的本质是"用持久性窗口换吞吐"**：把多次 fsync 摊成一次，吞吐随批大小上升，代价是崩溃时可能丢"已 ack 未落盘"的一批。与 [`01 §3.2`](01_io_design.md) 的 Redis-first / delta-flush 互补——后者把"写"移出 DB 关键路径，前者把"必须进 DB 的写"摊批。批大小 / 提交间隔 / 刷盘级别按业务可丢失窗口定（强一致核心库不放宽，见 [`02 §9.1`](02_data_model_design.md)）。

## 4. 默认选型与升级路径

| 场景 | 默认选型 | 升级触发 → 升级到 |
| --- | --- | --- |
| 单库多表 | 本地事务 | 跨库 → 分布式事务模型 |
| 跨数据源 | 本地消息表 / 异步补偿 | 强一致不可让步 → TCC / Saga |
| 高频数值变更 | 单命令原子 + 边界兜底 | 绝不允许越界 → 预扣 + 回退（仅在边界损耗不可接受时） |
| 幂等 | 唯一键 / 幂等表 | 高并发去重 → 分布式锁 + 状态机；**或读写分离规避** |
| 写 TPS（fsync 瓶颈） | 逐事务提交 | 小事务高频 → 批量提交 / group-commit 摊 fsync（§3.3）；可放宽持久性 → 组提交 + 刷盘分级 |

---

## 5. 权衡矩阵

| 方案 | 一致性 | 复杂度 | 热路径性能 | 适用 |
| --- | --- | --- | --- | --- |
| 本地事务 | 强 | 低 | 高 | 单库多表 |
| Saga / 补偿 | 最终 | 高 | 中 | 跨资源、可补偿 |
| TCC | 强 | 很高 | 中 | 跨资源、强一致 |
| 本地消息表 | 最终 | 中 | 高 | 跨数据源异步 |
| 弱一致 + 边界兜底 + 监控 | 最终（含极小损耗） | 低 | 很高（零锁） | 高频写、损耗可接受 |
| 读写分离消幂等 | 读幂等 | 很低 | 很高 | 判定与扣费可拆分 |
| 批量提交 / group-commit | 强（批内原子） | 低 | 很高（摊 fsync） | 小事务高频、可容忍批丢失窗口 |

## 6. 反模式速查

| 反模式 | 正确方向（详见 docs-spec/21 / 01） |
| --- | --- |
| 热路径加锁 / Lua 拦越界 | 单命令原子 + 持久化边界 `GREATEST`/`LEAST` 兜底 |
| 用预扣 + 回退 + 定时扫描的重机制 | 先评估读写分离 / 弱一致是否够用 |
| 跨资源写无补偿 | Saga / 补偿步骤设计 |
| 写入口无幂等 | 幂等键 + 去重表，或架构层读写分离 |
| 用分布式事务包一切 | 能本地事务的不上分布式 |
| 逐条小事务高频提交（每条一次 fsync） | 攒批合并一个事务 / 组提交摊 fsync（§3.3） |
| 事务内夹着 RPC / 外部 IO（长事务持锁过网络） | 先把 IO 移出事务边界，事务只包 DB 写 |

> 补偿 / 幂等 / 一致性窗口规则正文见 [`docs-spec/21`](../docs-spec/21_distributed_transaction_spec.md)；新增 Saga 步骤代码走 `/new-saga-step`。

## 9. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意：规范本体（§1-§8）已用占位符表达；本节系统名 / 代码为某真实工程实现，**只借纪律不借实现**。本节服从 [`PRINCIPLES.md §12`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 参考示例段豁免。

### 9.1 读写分离消除幂等复杂度（系统 B 鉴权/计费）

```
把"读检测资格"和"写扣费"拆成两个接口：Verify（只读不写）+ Report（写）
  · 重复鉴权不会多扣（读天然幂等）
  · 网关未上报 = 本次免费（可接受，监控兜底）
  · 省掉"预扣 + 回退 + ZSET 定时扫描"整套机制
```

### 9.2 高并发超扣的边界兜底

```
热路径：HINCRBY used+1 / HINCRBY balance-1   // 原子但不拦超扣（性能优先，可瞬时为负 / 超 total）
落库：  GREATEST(balance+delta, 0) / LEAST(used+alloc, total)   // 持久化边界夹断，超扣记 WARN 当损耗
```
用极小边界损耗换鉴权链路零锁高并发，比"强一致预扣 + 回退 + 扫描"简单一个数量级。

### 9.3 delta 增量 flush 一致性不变量

```
不变量：Redis.used = DB.SUM(used_quota) + Redis.delta
校验：  expected = db_used + pending_delta   // 吸收 flush 窗口内并发新增量，避免误判
SPOP 脏标记：从 Set 弹出即消费的原子性 → 多实例并行 flush 不重复、无需分布式锁
```
