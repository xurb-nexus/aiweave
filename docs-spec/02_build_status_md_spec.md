# 02 - BUILD_STATUS.md 写法规范

> 规定 `docs/BUILD_STATUS.md` 的内容结构、状态图例、暂不实现模块（§0）的标准写法、维护流程。

---

## 1. 定位

`BUILD_STATUS.md` 是 **AI 在生成任何代码之前必须读的文档**。它回答一个问题：

> 这个工程的某层（model / service / controller / 路由 / 任务）当前的实现状态是什么？

没有 BUILD_STATUS：
- AI 看到 md 就生成代码 → 容易重复创建已有实现
- AI 不知道某模块是 🚫 暂不实现 → 误生成被禁用的代码
- 重构/重建时无法判断从哪个 Stage 接力

---

## 2. 顶层结构（强制）

BUILD_STATUS.md 必须按以下章节顺序组织：

- `## 0.` ⛔ 暂不实现的模块（如有，详见 §3）
- `## 1.` 状态图例（详见 §4）
- `## 2.` 整体进度快照（按代码层级，详见 §5）
- `## 3.` 模型层（models/）明细
- `## 4.` Service 层（service/）明细
- `## 5.` Controller 层（controllers/http/）明细
- `## 6.` 定时任务与 MQ 消费者
- `## 7.` helpers 工具函数
- `## 8.` 文档状态（docs/）
- `## 9.` Skill 状态（.claude/skills/）
- `## 9.5` 测试框架（test/）状态
- `## 10.` 维护流程
- `## 11.` 约束清单状态轨道（详见 §11）
- `## 12.` 运行时基线区域（如启用 runtime_profile.md，详见 §12.5）

> **完整章节骨架见** [`templates/docs/BUILD_STATUS.md`](../templates/docs/BUILD_STATUS.md)——可直接复制到工程作起点。

各 §按代码层级排列，与 `service/`、`controllers/`、`models/` 等代码目录形成 1:1 映射。

---

## 3. §0 暂不实现的模块（标准模板）

这是本规范最重要的章节之一。它把"项目期决策"沉淀成 AI 可机械检查的规则。

### 3.1 章节模板

```markdown
## 0. ⛔ 暂不实现的模块（临时决策，将来删除）

以下模块在 `docs/` 中已有完整设计，但**当前阶段明确不进入代码实现**。AI 在任何情况下**不得**为这些模块生成代码。

| 模块 | 涉及代码路径（全部标为 🚫 暂停） | 涉及文档（保留完整设计，不动） | 在调用链中的处理方式 |
|------|-------------------------------|-----------------------------|-------------------|
| **{module-name}** | `service/{...}/`、`controllers/.../{...}/`、`controllers/command/{...}.go` | `docs/service/{...}.md`、其他相关 | 在调用链中的替代实现：如"`service/{auth-module}/checker.go::checkXxx` 实现为空壳直接返回 nil" |

### 0.1 附带关系
- 哪些 Redis Key / 表 / 配置项保留 md 完整但运行时不写
- 哪些定时任务设计保留但代码不实现
- 哪些接口定义在 md 中但路由不注册

### 0.2 检查规则
- `/new-service {module}/*` → **立即拒绝**，提示用户本模块暂不实现
- `/new-controller .../{module}/*` → 同上
- `/new-scheduled-task {task}` → 同上
- `/rebuild-from-docs {module}` → 同上
- `/rebuild-from-docs all` / `service/all` → **跳过** 本模块

### 0.3 解除条件（将来删除本 §0 的时机）
当决定启用 {module} 模块时，执行以下动作即可解除本节：
1. 删除本文档 §0 整节
2. 删除 `docs/{...}.md` 顶部的警示框（如有）
3. 删除其他相关 md 的"当前简化"标注
4. 恢复 `service/{...}/{...}.go::{Func}` 的完整实现（按对应 md 伪码生成）
5. 执行 `/new-service {module}/*` + `/new-controller .../{module}/*` + `/new-scheduled-task {task}`
6. 搜索项目所有 md 中 "🚫" / "暂不实现" / "当前阶段跳过" 关键字，逐个确认并清理
```

### 3.2 必填字段

每个暂不实现模块都必须填写：

| 字段 | 说明 |
|------|------|
| 模块名 | 给一个清晰的名字，所有 md / Skill / 代码注释都用这个名字指代 |
| 涉及代码路径 | 列出所有"原本应该生成但当前不生成"的代码文件/目录 |
| 涉及文档 | 列出"完整保留不动"的 md（需要时附章节号） |
| 在调用链中的处理方式 | 关键！如何让其他模块在缺失这部分实现时仍能正常工作（通常是空壳实现/直接放行） |
| 附带关系（§0.x） | 哪些 Redis Key / 配置项 / 接口"形式上保留，运行时不触发" |
| 检查规则（§0.x） | 列出每个 Skill 命中本模块时的拒绝行为 |
| 解除条件（§0.x） | 启用时需要按顺序执行的动作清单 |

### 3.3 反例（不应该这样做）

```markdown
> {受限模块}暂时不做。   ← ❌ 模糊，AI 不知道哪些路径要拒绝、哪些设计要保留
```

正确的做法是按 §3.1 模板填全字段。

### 3.4 多个模块时

如果有多个暂不实现模块，§0 用同一表格列出所有模块；§0.1/§0.2/§0.3 分别按模块再做子小节。

## 4. §1 状态图例（标准）

```markdown
## 1. 状态图例

| 图例 | 含义 | AI 处理方式 |
|------|------|-----------|
| 🟢 | 已实现且在运行时装配（可直接 import 调用） | 按既有签名调用，勿重复创建 |
| 🟡 | 代码已落盘但未装配 / 被隔离（需要额外启用步骤） | 启用或调用前先读取代码，确认签名 |
| ⬜ | 设计已定稿，代码尚未创建（按 skill 生成） | 使用对应 `/new-*` skill，严格按文档伪码实现 |
| 🚫 | 设计已定稿，但当前阶段**明确不实现**（见 §0） | **拒绝生成**，不得在任何 skill 中触发 |
| ❌ | 未设计且未实现（文档也缺） | **停止**生成，先补 md |
```

5 个图例**不可增删**——这是 Skill 黑名单机制的契约。

## 5. §2 整体进度快照

按代码层级一行一项，列出当前状态：

```markdown
## 2. 整体进度快照

| 层 | 状态 | 已实现 / 规划 | 下一步 |
|----|------|--------------|-------|
| `go.mod` / 模块根 | 🟢 | 1 / 1 | — |
| `main.go` + `helpers.PreInit/InitResource` | 🟢 | 1 / 1 | 已就绪 |
| `conf/` 配置 | 🟢 | 3 / 3 | 无外部依赖 |
| `components/error.go` | 🟢 | 全量错误码已定义 | — |
| `models/` N 库 M 表 | 🟢 / ⬜ | M / N | 待补：核对 X / Y 与 DDL 是否一致 |
| `helpers/` 基础设施 | 🟢 | 主体全部就绪 | — |
| `middleware/` | 🟡 | 2 / 3 | 待生成 {admin_mw} |
| `service/` | 🟢 / 🚫 | {module} 🟢；{module} 🟢；... ；{disabled_module} 🚫 | — |
| `controllers/http/` | 🟡 | X / Y | — |
| `controllers/command/` | ⬜ | 0 / 11 | Stage 4 起按 skill 生成 |
| `controllers/mq/` | ⬜ | 0 / 1 | Stage 4 |
| `router/http.go` | 🟡 | X 路由实际注册 | 完成 controller 后批量注册 |
```

每行字段：

- **层** —— 代码路径或模块名
- **状态** —— 🟢/🟡/⬜/🚫 之一（如果一行包含多个子项有不同状态，用 `/` 分隔）
- **已实现 / 规划** —— 数量比，如 `28/28`
- **下一步** —— 简短说明（"已就绪" / "待 X 完成后启用" / "—"）

## 6. §3-§9 各层明细

每节列出该层的全部条目，每条标注当前状态：

```markdown
## 4. Service 层（service/）明细

| 模块 | 文档 | 方法数 | 状态 |
|------|------|--------|------|
| `service/{module-1}/` | `{module-1}_service.md` | 9 | 🟢 |
| `service/{module-2}/` | `{module-2}_service.md` | 14 | 🟢 |
| ...
| `service/{disabled-module}/` | `{disabled-module}_service.md` | 4 | 🚫 **暂不实现（见 §0）** |

**生成顺序建议**（依赖最少 → 最多）：
```
1. {audit-module}
2. {module-1}
3. {module-2}
4. {module-N-1}
5. {module-N}
6. {auth-module}
7. {disabled_module}  ← 🚫 跳过
```
```

各层的明细表必填字段：

- 模块/文件名
- 关联的 md 文档
- 数量（方法数 / 接口数 / 表数）
- 当前状态

## 7. §10 维护流程

```markdown
## 10. 维护流程

- 每次 `/new-*` skill 生成新代码后，**必须**更新本文档对应行的状态列（🟢 / 🟡 / ⬜）
- 每次 `/doc-sync-check all` 完成后，**必须**与本文档对比，发现差异立即补齐
- 新增文档或 skill 时，**必须**在 §8 / §9 中登记一行
- 删除代码时，**必须**把对应行从 🟢 改回 ⬜（如果仍计划实现）或移除（如果设计也删除）

**AI 自检**：生成代码前，如果无法快速判定目标模块当前状态，**停下**先读本文档，不要盲猜。
```

---

## 8. 与 Skill 的联动

每个 Skill 的第 0 步必须执行：

1. **检查 `BUILD_STATUS.md` §0** —— 命中即拒绝
2. **检查目标层级的状态** —— 🟢 提示已实现、🟡 提示先确认签名、⬜ 继续生成
3. **检查依赖层级的状态** —— 如 controller 依赖 service，如果 service 还是 ⬜，先生成 service

详见 `aiweave/skills-spec/01_skill_authoring_guide.md` §3。

---

## 9. 与 INDEX.md 的关系

| 字段 | BUILD_STATUS | INDEX |
|------|--------------|-------|
| 一篇 md 的存在 | §8 列出 | 各章节表格列出 |
| 一篇 md 的描述 | 不写 | 一句话描述 |
| 一篇 md 对应代码的状态 | §3-§7 标注 | 不写 |

两者**互补**：INDEX 回答"有什么文档"，BUILD_STATUS 回答"哪些已实现"。

## 11. 约束清单状态轨道

### 11.1 概念

除"代码模块状态"轨道（§3-§9）外，BUILD_STATUS.md 还维护一条**约束清单状态轨道**（§11）。

代码模块状态轨道回答"哪些代码已实现"；约束清单状态轨道回答"哪些约束类清单已设计 / 已启用"。

约束类清单包括但不限于：

- `docs/architecture/concurrency_safety.md §2 共享状态注册表`（每条共享状态一行）
- `docs/architecture/concurrency_safety.md §3 锁粒度决策表`（每把锁一行）
- `docs/service/transaction_design.md §2 事务边界清单`（每个写入操作一行）
- `docs/service/transaction_design.md §4 幂等性设计`（每个幂等接口一行）
- `docs/service/{module}_service.md §7 领域不变量`（每条不变量一行）
- `docs/architecture/ai_dev_guide.md` grep 锚 rule-id 索引（每条规则一行）

### 11.2 状态图例（沿用 §1 + 一项扩展）

| 图例 | 含义 | AI 处理方式 |
|------|------|-----------|
| 🟢 | 约束已启用（设计 + 校验代码 + 测试都齐） | 写新代码时必须遵守 |
| ⬜ | 设计已完成，但当前未启用（无校验代码 / 不在审计 Skill 中触发） | AI 可按 docs 完整理解约束，但不在 PR 中阻断未遵守的代码 |
| 🚫 | 设计完整但当前阶段明确不启用（与 §0 联动） | AI 写代码时**不引用**该约束 |

约束类轨道**允许长期保持 ⬜**——这是它与代码模块状态轨道（⬜ 是临时态，应推进到 🟢）的关键区别。

### 11.3 模板

```markdown
## 11. 约束清单状态轨道

### 11.1 共享状态约束（对应 docs/architecture/concurrency_safety.md §2）

| 约束类 | 已设计条目数 | 已启用条目数 | 状态 | 说明 |
|--------|-------------|-------------|------|------|
| 共享状态注册表 | {N} | {M} | 🟢 / ⬜ / 🚫 | — |
| 锁粒度决策表 | {N} | {M} | 🟢 / ⬜ / 🚫 | — |
| Channel 清单 | {N} | {M} | 🟢 / ⬜ / 🚫 | — |
| Goroutine 池配置 | {N} | {M} | 🟢 / ⬜ / 🚫 | — |

| 幂等接口清单 | {N} | {M} | 🟢 / ⬜ / 🚫 | — |

### 11.3 领域不变量（对应 docs/service/{module}_service.md §7）

| 模块 | 业务规则 (I-N) 数 | 隐式约束数 | 状态 | 说明 |
|------|-----------------|----------|------|------|
| `{Module-A}` | {N} | {M} | 🟢 / ⬜ / 🚫 | — |

### 11.4 跨服务合约约束（对应 docs/architecture/cross_service_contract.md）

### 11.5 grep 锚 rule-id 索引（对应 docs/architecture/ai_dev_guide.md）

| Rule-id | 含义 | 状态 | 说明 |
|---------|------|------|------|
| `R-CONC-LOCK-IO` | 锁内做网络 IO | 🟢 | — |
| `R-CONC-MAP-RACE` | map 并发写 | 🟢 | — |
| `R-TXN-NO-IDEMPOTENT` | 写入类接口无幂等 Key | 🟢 | — |
```

### 11.4 与代码模块状态轨道的区别

| 维度 | 代码模块状态（§3-§9） | 约束清单状态（§11） |
|------|--------------------|--------------------|
| 单位 | 一个文件 / 模块 / Skill | 一类约束（多条条目可聚合为一行）|
| 状态语义 | 是否已落代码 | 是否在 AI 行为中生效 |
| ⬜ 含义 | 待生成 → 应推进到 🟢 | 已设计未启用 → 可长期保持 |
| AI 联动 | 生成代码前先查 | 写代码时是否触发约束告警 |

### 11.5 维护

- 新增约束（新增 docs/architecture/concurrency_safety.md §2 一行 / 新增不变量 I-N 等）→ §11 对应类目的"已设计条目数"+1
- 约束对应的校验代码 / 审计 Skill 触发 → "已启用条目数"+1，状态转 🟢
- 启用某类约束的总开关→ 该类目状态从 ⬜ 转 🟢

## 12. §12 运行时基线区域（如启用 runtime_profile.md）

> 与 §3-§9 代码模块状态轨道、§11 约束清单状态轨道并列的第三条独立轨道。承载工程的运行时静态描述基线。

### 12.1 定位

`runtime_profile.md`（详见 [`templates/docs/architecture/runtime_profile.md`](../templates/docs/architecture/runtime_profile.md)）记录工程的流量模式 / 数据规模 / 资源消耗基线 / 关键超时链。但这些数据来自运行时，AI 无法直接感知 —— 本节维护"上次校对日期 + 数据来源 + 偏离百分比"三项元数据，让 AI 评估静态档案的可信度。

### 12.2 模板

```markdown
## 12. 运行时基线区域

| 项 | 上次校对日期 | 数据来源 | 与上次基线偏离 | 备注 |
|----|-------------|---------|--------------|------|
| §1 流量模式 | {YYYY-MM-DD} | APM / Prometheus / 人工估算 | {P}% | — |
| §2 数据规模 | {YYYY-MM-DD} | DB 实测 / 人工估算 | {P}% | — |
| §3 资源消耗基线 | {YYYY-MM-DD} | Prometheus / 容器监控 | {P}% | — |
| §4 关键超时链 | {YYYY-MM-DD} | APM / 配置 | — | 与 performance_contract.md §6.2 双向对齐 |
```

### 12.3 维护触发

- 每季度技术评审后必须更新（即使数据无变化也要更新校对日期）
- 偏离 > 30% → 触发 runtime_profile.md 重写
- 数据来源变更（如从人工估算 → APM 实测）→ 更新"数据来源"列

### 12.4 AI 行为约束

- AI 写代码时**不直接使用** §12 的数值（数值仅供工程负责人参考）
- 如代码逻辑依赖运行时假设（如"日峰值不超过 N"），必须显式标注 `// runtime-assumption: §X` 并在 §12 留痕
- §12 与 §11 不耦合：约束清单状态独立追踪，运行时基线独立维护

---

## 13. 与 INDEX.md §0 采纳进度的关系

`INDEX.md §0 AIWeave 采纳进度`追踪**整个工程对 AIWeave 演进规范的启用状态**（如 `20_concurrency_safety` / `21_distributed_txn` 是否启用）。

`BUILD_STATUS.md §11 约束清单状态轨道`追踪**单条约束条目的状态**（如 §2 共享状态注册表内"某条 mutex"是否启用对应校验）。

两者粒度不同：INDEX.md §0 是规范级（粗），BUILD_STATUS.md §11 是条目级（细）。一个规范在 INDEX.md §0 标 🟢 启用，意味着它的所有 ⬜ 条目都被纳入 §11 追踪范围；标 ⬜ 未启用则其 §11 子表整体置 ⬜。
