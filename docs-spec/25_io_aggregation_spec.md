# 25 - IO 极致与聚合并行规范

> 规定 `docs/architecture/io_contract.md` 的内容结构与维护规则。
>
> 本规范是 AIWeave 的 P1 规范，承载"IO 极致"诉求：**消灭 N+1、消灭独立串行编排，强制聚合 + 并行**。与 22 性能合约互补——22 管"单次 IO 的延迟预算"，25 管"多次 IO 的聚合与并行编排"。

---

## 1. 定位

`io_contract.md` 是 **AI 在生成任何"涉及多次数据访问 / 跨实例读取 / 批量处理 / 多依赖编排"代码之前必须读的文档**。它回答四个问题：

> 1. 本工程的 IO 极致目标是什么（每请求 / 每批的 IO 往返预算）？
> 2. 哪些数据访问必须走聚合器（批量读 + 并行回源）？
> 3. 三条 IO 铁律的边界、识别迹象与豁免条件是什么？
> 4. 聚合器原语的契约（共享只读、单飞合并、异步写回、延迟结果槽）是什么？

没有这篇文档：

- AI 在循环里逐条查询（**N+1**），把 1 次请求放大成 O(N) 次 IO 往返
- AI 把 N 个互不依赖的读 **逐个串行 await**，把可并行的 RTT 累加成串行总和
- AI 反复跨实例单条读取，不做按实例分组的批量合并
- AI 在热路径里同步阻塞等待多个下游，吞掉吞吐

## 2. 三条 IO 铁律（强制 / 最高优先级）

本规范的核心是三条**不可绕过**的铁律：一条正向总则（铁律零·批量优先）统摄两条负向强制（铁律一·禁 N+1 / 铁律二·禁独立串行编排）。任何被审计命中且无合法豁免的代码，视为未交付。

> 读 / 写两类访问的优化方向相反（读消灭往返、写削峰摊销），三条铁律主要约束**读路径与编排**；读写路径分治的决策方法论见 [`design-spec/01 §3`](../design-spec/01_io_design.md) 与 [`PRINCIPLES §13.3`](../PRINCIPLES.md#13-io-极致三条铁律--读写分治--聚合优先)。

### 2.0 铁律零：批量优先（正向总则）

> 任何"对一批 ID / key / name 的查询"必须用**批量方法一次完成**（`{Get-method}List` / `{List-method}List`）。

| 维度 | 内容 |
| --- | --- |
| 识别迹象 | 已知一批 key，却因"数据层没有批量方法"而退化为循环单查 |
| 正确做法 | **数据层若缺批量方法，先在数据层补 `{...List}` 方法**（分表按 shard 分组并行），再在上层调用——禁止"因为没有批量方法就退化为循环单查" |
| 合法豁免 | 同铁律一（编译期常量小循环 / 已是批量 API） |
| grep 锚 | 共享 `R-IO-N-PLUS-1`（违规表现即铁律一 N+1）；附加锚 `R-IO-BATCH-FIRST` 标注"缺批量方法应先补" |

> 铁律零是铁律一的正向表达：铁律一禁止"循环内单条查询"的**结果**，铁律零规定"先把批量方法补齐"的**前置动作**——两者同源，合并审计。

### 2.1 铁律一：禁止「循环内单条查询」（N+1）

> 在 `for` / `range` / 递归 体内，对同一类资源逐条发起独立查询（DB / 缓存 / RPC / 外部 API）。

| 维度 | 内容 |
| --- | --- |
| 识别迹象 | 循环体内出现 `{单条读方法}` / `{单条 RPC}` / `{单条缓存读}`，且查询 key 来自循环变量 |
| 正确做法 | 循环外收集全部 key → **一次批量读**（`IN (...)` / 批量缓存读 / 批量 RPC）→ 落 `map[{key}]→{entity}` → 循环内 O(1) 装配 |
| 合法豁免 | ① 循环次数为编译期常量且 ≤ `{Loop-safe-N}`（推荐 5）；② 调用的已是批量 API；③ 行末 `// aiweave:allow=R-IO-N-PLUS-1 reason=...` |
| grep 锚 | `R-IO-N-PLUS-1` |

### 2.2 铁律二：禁止「独立串行查询编排」

> 多个**互不依赖**的 IO 操作被逐个串行执行（前一个完成才发起后一个），而它们本可并行或聚合。

| 维度 | 内容 |
| --- | --- |
| 识别迹象 | 连续 ≥ 2 个 `{读操作}` 各自 `if err != nil` 串行展开，且后者的入参**不依赖**前者的结果 |
| 正确做法 | 用聚合器登记后**一次性并行执行**（§4），或扇出并行（worker pool / `WaitGroup` / `errgroup`）后统一收集结果与首错 |
| 合法豁免 | **真实数据依赖链**——后一次查询的入参来自前一次的结果（如先查主体拿到外键再查从属）。依赖链不算违规，但应审视能否预取/合并 |
| grep 锚 | `R-IO-SERIAL-ORCH` |

> **判定要点**：铁律二的违规核心是"独立"。只要存在真实依赖（B 的入参是 A 的输出），串行是必要的、合法的；AI 应能区分"伪串行（可并行）"与"真依赖（必须串行）"。

### 2.3 衍生约束（同源于三条铁律）

| 约束 | 正确做法 | grep 锚 |
| --- | --- | --- |
| 循环内单条写 → 应批量写 / pipeline | 收集后一次批量写 | `R-IO-LOOP-WRITE` |
| 多次跨实例缓存读未按实例分组批量 | 按后端实例分组，每实例一次批量读 | `R-IO-NO-AGGREGATOR` |
| 循环内单条 RPC / 外部调用 | 批量接口 / 并行扇出 | `R-IO-LOOP-RPC` |

## 5. §1 IO 极致目标

```markdown
| 维度 | 目标值 | 测量方式 | 超标动作 |
|------|--------|---------|---------|
| 每请求 DB 往返 | ≤ `{DB-RTT-budget}`（聚合后） | 链路追踪 span 计数 / mock 计数断言 | 超标 → 审查是否漏聚合 |
| 每请求跨实例缓存往返 | ≤ `{Cache-RTT-budget}`（按实例分组后） | 同上 | 超标 → 合并为批量读 |
| 每批处理任务的批量写次数 | 每批 ≤ `{Batch-write-N}`（pipeline / 批量） | 任务日志计数 | 超标 → 攒批 |
| 独立 IO 的并行化率 | 无依赖的 IO 必须并行 | 代码审查 + io-review | 串行 → 改聚合 / 扇出 |
```

> **占位符示例**：`{DB-RTT-budget}` 通常取 3~5；`{Cache-RTT-budget}` 通常取 1~2（按实例分组后每实例 1 次批量读）。具体由工程业务决定。

## 7. §3 聚合器模式（核心 / 强制）

聚合器（Aggregator）是消灭 N+1 与串行编排的标准武器。它把"分散的多次 IO"重写为"收集 → 批量 → 并行"的统一编排。业界有成熟实现可参考，本规范只约束其**抽象契约**，不绑定任何具体实现。

### 7.1 五阶段编排（强制）

```
① 收集阶段   登记 N 个"缓存兜底读"（cache-backed）与 M 个"直接任务"（direct），
            每次登记立即返回一个【延迟结果槽】（Slot），尚未求值。
② 批量读阶段  按后端实例（缓存实例指针 / 库）分组，每组发起【一次批量读】
            （MGET 等价 / 批量 IN），一次网络往返拿回整组命中。
③ 并行回源阶段 把"批量读未命中项"与"直接任务"合并，全部【并行执行】
            （worker pool / WaitGroup），收集首个错误，其余 best-effort。
④ 单飞合并   同一 key 的并发回源【塌缩为一次】底层查询（single-flight），
            其余等待者共享同一结果指针。
⑤ 异步写回   回源结果以【fire-and-forget】方式按实例分组批量写回缓存，
            脱离响应关键路径，失败静默（best-effort）。
```

### 7.2 延迟结果槽（Slot）契约（强制）

```markdown
- Slot 是单次查询的"延迟结果"，由收集阶段返回，求值后填充。
- nil-safe：对 nil Slot 取值返回零值（与"未找到"语义对齐），调用方无需前置判空。
- 【共享只读铁律】：单飞使多个并发请求拿到【同一指针】。下游（装配 / 渲染）
  必须将 Slot 取出的值视为【只读】，禁止就地修改其字段——否则与共享该指针的
  其它请求产生并发写竞争。如需修改先深拷贝。
```

### 7.3 抽象伪码（占位符）

```go
// 收集阶段：登记缓存兜底读 + 直接任务，返回延迟结果槽
agg := aggregator.New()
slotA := aggregator.Preload[{TypeA}](agg, {cacheA}, {keyA}, dbFetchA, {ttlA})
slotB := aggregator.Preload[{TypeB}](agg, {cacheB}, {keyB}, dbFetchB, {ttlB})
agg.Go(func() error { return loadList(ctx, &listC) }) // direct（绕缓存的列表查询）

// 执行：批量读 → 并行回源 → 异步写回，一次完成
if err := agg.Fetch(ctx); err != nil {
    return err
}

// 装配（只读使用 Slot.Value()，禁止就地改字段）
vA, vB := slotA.Value(), slotB.Value()
```

### 7.4 时序图（强制 ASCII）

```
请求
  │  收集（N 个 cache-backed + M 个 direct，各返回 Slot）
  ▼
按实例分组 ──► 实例1: 批量读(一次RTT) ─┐
              实例2: 批量读(一次RTT) ─┤── 命中直接填 Slot
              ...                    ─┘
  │  未命中项 + direct
  ▼
并行回源（worker pool）──► single-flight 同 key 合并 ──► 填 Slot / 收首错
  │
  ▼
异步批量写回（fire-and-forget，脱离关键路径）
  │
  ▼
返回（所有 Slot 已就绪）
```

## 8. §4 并行编排原语清单

```markdown
| 原语 | 适用场景 | 强制约束 |
|------|---------|---------|
| 聚合器（Aggregator） | 多个"缓存兜底读"+少量直接任务的混合编排 | 批量读按实例分组；回源并行；Slot 只读 |
| 扇出并行（Fan-out） | N 个无缓存、互不依赖的 IO | 受 worker pool 上限约束；首错聚合；ctx 透传 |
| 流水线（Pipeline） | 多阶段、阶段间无 barrier 的批处理 | 每项独立流过各阶段；慢项不阻塞快项 |
```

约束：

- **并发上限**：所有并行原语必须受 worker pool / 信号量约束，禁止裸 `go func()` 无界扇出（goroutine 生命周期见 [`20 §1.1`](20_concurrency_safety_spec.md)）。
- **错误聚合**：并行任务收集**首个错误**返回，其余任务 best-effort 跑完（一个失败不阻断其它结果计算）。
- **ctx 透传**：并行子任务必须透传请求 `ctx`；异步写回需 detach（复制 ctx）以脱离请求生命周期。

## 9. §5 数据访问聚合约束

```markdown
### 5.1 批量读优先（强制）
- 已知 key 集合的多条读 → 循环外收集 → 一次 `IN (...)` / 批量缓存读 → map 装配
- 跨实例 → 按实例分组，每实例一次批量读

### 5.2 批量写优先（强制）
- N 次同类写 → 一次批量写 / pipeline（一次 RTT）
- 攒批阈值与 flush 周期在本节登记

### 5.3 预加载 map 装配
- 批量读结果落 `map[{key}]→{entity}`，循环内 O(1) 取用，禁止循环内回查

### 5.4 JOIN vs 多查询的取舍
- 强依赖、同库、可索引 → 优先单条 JOIN（≤ `{JOIN-limit}`，见 22 §4.2）
- 跨库 / 跨实例 / 弱依赖 → 聚合器并行多查询，禁止串行
```

### 9.1 §5.4 / §5.5 / §5.6 / §5.7 表示记录槽位（读写路径分治落地登记）

`io_contract.md §5` 在 `§5.1~§5.3` 三条聚合约束之后，追加四个**表示记录槽位**——工程填空"本服务读写路径各选了什么手段"，**只记选了什么，不复制读/写两棵决策树**。决策方法论真相源在 [`design-spec/01 §3`](../design-spec/01_io_design.md)，每槽位一行反向引用回链。

```markdown
### 5.4 读写路径分治登记
| 路径 | 形态 | 本服务选定手段（登记） |
|------|------|---------------------|
| 读 | {单实体读 / 集合 enrich / 多依赖编排} | {本地命中(0 网络)>单次往返 / 消重复读派生值透传 / 分支裁剪} |
| 写 | {高频状态写 / 纯追加} | {Redis-first + 脏标记 + 定时增量 flush / 投 MQ 按 `{key}` 保序攒批} |
> 决策方法见 design-spec/01 §3.1（读路径树）/ §3.2（写路径树）。

### 5.5 shard 分组并行编排登记
| 批量方法 | 分组键 | 只查规则 | 并行约束 |
|---------|-------|---------|---------|
| `{Module}.{Get-method}List` | `{shard-key}` | 仅查有请求落点的物理表，空 shard 不发查询 | 多 shard 并行（worker pool 上限见 §6） |
> 决策方法见 design-spec/01 §3.1 及铁律零批量优先（§2.0）。

### 5.6 读路径命中不合并登记
| 本地缓存层 | 命中即返回的读 | 禁止合并进 Pipeline 的原因 |
|-----------|-------------|----------------------|
| `{L1-local-cache}` | `{Module}.{hot-read-method}` | 本地命中已 0 往返，合并 Pipeline 反劣化为 1 次 RTT |
> 决策方法见 design-spec/01 §3.1（读路径反直觉点：消灭往返 > 合并往返）。

### 5.7 批量写极致登记（Upsert / 批量装载）
| 写形态 | 选定手段（登记） | 落点（哪些写） |
|-------|---------------|--------------|
| {插入/更新混合} | {批量 Upsert：`ON DUPLICATE KEY`/`ON CONFLICT`，一次往返免 select-then-write} | `{file}:{func}` / `{Module}.{Upsert-method}List` |
| {大批量装载（冷启 / 迁移 / 离线导入）} | {`LOAD DATA`/`COPY` 批量装载} | `{file}:{func}` / `{bulk-load-entry}` |
> 决策方法见 design-spec/01 §3.2（写路径树：插入/更新混合 → Upsert；大批量装载 → `LOAD DATA`/`COPY`）。
```

> **铁律 vs 槽位的关系**：§2 三条铁律是**机械可审计的硬约束**（命中即违规）；`§5.4~§5.7` 是**工程填空记录槽**（登记本服务实际选了哪种读/写手段），二者不冲突——§5.5 正是铁律零"补 `{...List}`"在分表下的落地形态，§5.6 正是 §2 末"本地命中禁再合并 Pipeline"反向纪律的登记面，§5.7 正是写路径树"插入/更新混合 → Upsert、大批量装载 → `LOAD DATA`/`COPY`"的登记面。

## 12. AI 行为约束（强制）

```markdown
### IO 约束（强制）

- 对一批 key 的查询（铁律零·批量优先）：数据层缺批量方法时，**先在数据层补 `{...List}` 方法**（分表按 shard 分组并行）再调用，禁止退化为循环单查
- 生成"对集合逐元素访问资源"的代码时：
  - 禁止循环内单条查询（铁律一）——必须循环外批量 + map 装配
  - 禁止循环内单条写——必须收集后批量 / pipeline
- 生成"多个互不依赖的读"时：
  - 禁止逐个串行 await（铁律二）——必须聚合器或扇出并行
  - 仅当后查询入参依赖前查询结果时，串行才合法（且需审视可否预取）
- 多次跨实例读 → 必须按实例分组批量（走聚合器）
- 共享指针结果（single-flight / Slot）→ 只读，禁止就地改字段
- 新增并行原语 → worker pool 上限在 §6 登记；channel 容量与 22 §6.1 对齐
- 拿不准是否构成 N+1 / 串行违规 → 主动报告并建议聚合方案，不擅自串行
```

### 13.1 B1 反向同步规则（强制）

| 代码迹象（git diff） | 反向同步动作 |
| --- | --- |
| 循环体内新增单条 `{读方法}`（key 来自循环变量） | 命中铁律一；拒绝合并或要求改批量 + §6 往返预算复核 |
| 连续 ≥ 2 个独立读串行展开 | 命中铁律二；要求改聚合器 / 扇出，或在 §2.2 登记依赖链豁免 |

| 新增批量 Upsert（`ON DUPLICATE KEY`/`ON CONFLICT`）/ 大批量装载（`LOAD DATA`/`COPY`） | §5.7 批量写极致登记新增 / 更新对应行 |

| 修改攒批阈值 / flush 周期 | §5.2 与 §6 同步 |

## 16. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意，规范本体（§1-§15）已用占位符表达；落地工程时按业务语义替换占位符，不得直接照搬本节具体业务名作为规范。
>
> 示例服从 [`PRINCIPLES.md §12 占位符规则`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 的"参考示例段豁免"。

### 16.1 铁律一反例 → 正例（N+1）

```go
// ❌ 反例：循环内单条查询（N+1）
for _, id := range orderIDs {
    user, _ := userRepo.FindByID(ctx, order.UserID) // N 次 DB 往返
    enrich(order, user)
}

// ✅ 正例：循环外批量 + map 装配（1 次往返）
ids := collectUserIDs(orders)
users, _ := userRepo.FindByIDs(ctx, ids)           // 一次 IN(...) 查询
um := indexByID(users)                              // map[userID]→user
for _, order := range orders {
    enrich(order, um[order.UserID])                 // O(1) 装配
}
```

### 16.2 铁律二反例 → 正例（独立串行编排）

```go
// ❌ 反例：3 个互不依赖的读串行 await，RTT 累加
profile, err := profileRepo.Get(ctx, uid)
if err != nil { return err }
quota, err := quotaRepo.Get(ctx, uid)               // 不依赖 profile，却串行等待
if err != nil { return err }
settings, err := settingsRepo.Get(ctx, uid)         // 同上
if err != nil { return err }

// ✅ 正例：聚合器一次并行（RTT ≈ max 而非 sum）
agg := aggregator.New()
sp := aggregator.Preload[Profile](agg, cacheUser, profileKey(uid), profileRepo.Get, 300)
sq := aggregator.Preload[Quota](agg, cacheUser, quotaKey(uid), quotaRepo.Get, 60)
ss := aggregator.Preload[Settings](agg, cacheUser, settingsKey(uid), settingsRepo.Get, 600)
if err := agg.Fetch(ctx); err != nil { return err }
profile, quota, settings := sp.Value(), sq.Value(), ss.Value()
```

### 16.3 合法串行（真实依赖链，不算违规）

```go
// ✅ 合法：order.UserID 来自第一次查询的结果，存在真实依赖
order, err := orderRepo.Get(ctx, orderID)
if err != nil { return err }
user, err := userRepo.Get(ctx, order.UserID)        // 入参依赖前次结果 → 串行必要
```
