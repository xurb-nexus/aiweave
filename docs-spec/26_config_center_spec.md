# 26 - 配置中心与凭据加密规范

> 规定 `docs/architecture/config.md` 中"**配置上云 + 配置中心集成 + 凭据加密 + 启动期拉取落盘**"部分的内容结构与维护规则。
>
> 本规范是 AIWeave 的 P2 规范。[`04 §2`](04_architecture_overview_spec.md) 治理 `config.md` 的总体结构（分类 / 清单 / Go struct 映射），**本规范是其"配置上云 / 凭据安全"维度的真相源**——两者通过 §2 边界划分协作，不重复。

---

## 1. 定位

`config.md` 的配置中心相关章节是 **AI 在生成任何"读取配置 / 初始化资源凭据 / 接入配置中心"代码之前必须读的文档**。它回答四个问题：

> 1. 配置如何上云（哪些配置以配置中心为权威源、哪些随代码发布）？
> 2. 配置中心的连接凭据（账号 / 密码）如何加密存放、密钥从哪来？
> 3. 启动期如何从配置中心拉取配置并落盘到本地文件？
> 4. 环境维度（dev / test / prod）如何区分配置？

没有这篇文档：

- AI 把数据库 / 配置中心的账号密码**明文**写进配置文件并提交进版本库
- AI 把加密密钥硬编码进代码并随仓库分发
- AI 在配置中心拉取失败时**静默用空配置 / 旧配置启动**，掩盖故障
- AI 不区分"随代码发布的业务配置"与"云端动态的环境配置"，把两者混放

## 3. 核心原则（强制）

| 原则 | 含义 |
| --- | --- |
| **配置上云** | 环境相关配置以**配置中心为权威源**；本地文件是启动期落盘的副本/缓存，不是真相源 |
| **凭据加密** | 配置中心连接凭据（账号/密码）**禁止明文**落盘/入库，以对称加密密文存放 |
| **密钥不入库** | 加密密钥由**应用层持有**，不随密文 / 代码一起进版本库；框架库**不内置任何密钥**；每工程专属密钥 |
| **fail-fast** | 启动期拉取失败立即终止进程；**禁止**以空配置 / 旧配置静默启动 |
| **最小落盘** | 拉取内容为空视为失败，避免把本地配置清空成空文件 |
| **实现隔离** | 配置中心的网络调用只存在于"实现层"（一个可替换的 Client 实现），上层不感知具体配置中心品牌 |

## 5. §6 配置分层与环境模型

```markdown
### 6.1 权威源矩阵
| 配置类别 | 权威源 | 修改方式 | 入 git |
|---------|--------|---------|--------|
| 业务配置（随代码发布） | 本地仓库 | 改代码 + 发版 | ✅ |
| 环境配置（端口 / 资源地址 / 外部依赖） | 配置中心（云端） | 配置中心控制台 | ❌（启动期落盘副本） |
| 连接凭据（配置中心账号 / 资源账号） | 加密密文 + 应用层密钥 | 重新生成密文 | 密文可入库；密钥❌ |

### 6.2 环境维度
- 环境标记来源：`{env-source}`（如环境变量 / 环境标记文件）
- 配置标识按环境拼接：`{dataId}` + `{group}-{env}`（同一份配置在不同环境用不同分组/后缀）
```

## 6. §7 配置中心集成（抽象 / 强制）

> 本节只约束**抽象契约**，不绑定任何配置中心品牌。具体配置中心实现放在独立子包，是**唯一**进行配置中心网络调用的地方（隔离 / 可替换）。

### 6.1 Client 抽象

```markdown
- 定义最小读取接口：按"配置标识 + 分组"读取一份配置的原始文本。
- 具体实现对应一个"已连接的配置中心客户端"；上层只依赖接口，不依赖实现。
- 能力边界：当前规范只要求"读取（拉取）"能力；监听 / 发布为可选扩展，不强制。
```

```go
// 抽象 Client 接口（占位符）
type Client interface {
    // Get 按 dataId + group 读取一份配置的原始文本。
    Get(dataId, group string) (string, error)
}
```

### 6.2 连接信息结构

```markdown
| 字段 | 含义 | 安全要求 |
|------|------|---------|
| `{Addr}` | 配置中心地址 | — |
| `{Namespace}` | 命名空间 / 租户隔离 | — |
| `{Username}` | 连接账号 | 允许密文，运行期解密 |
| `{Password}` | 连接密码 | 允许密文，运行期解密 |
| `{TimeoutMs}` | 请求超时；0 表示实现默认 | 必须有上限（防启动期 hang） |
```

### 6.3 拉取 → 落盘映射与编排（强制）

```markdown
- 每条映射 = { 远端配置标识 dataId, 分组 group, 本地落盘路径 LocalPath }。
- 编排（Sync）行为约定：
  1. 逐条按 dataId + group 拉取原始文本；
  2. 自动创建落盘路径父目录，覆盖写入本地文件；
  3. 拉取内容为空 → 视为失败（避免清空本地配置）；
  4. 任一条拉取 / 写盘失败 → 立即返回错误（上层 fail-fast）。
```

```go
// 拉取→落盘映射（占位符）
type Item struct {
    DataId    string // 配置中心中的配置标识
    Group     string // 分组（已按环境拼好后缀）
    LocalPath string // 落盘的本地文件路径
}

// 编排：逐条拉取并覆盖写入本地文件，任一失败即 fail-fast
func Sync(client Client, items []Item) error { /* 按 §6.3 约定实现 */ }
```

## 7. §8 凭据加密（强制）

### 7.1 算法

```markdown
- 推荐对称加密（如 AES-256-GCM）：带完整性校验，篡改密文在解密时报错。
- 每次加密随机生成 nonce 前置到密文，整体 Base64 编码输出。
- 加密因随机 nonce 而非确定性（同明文每次密文不同），但解密确定性。
```

### 7.2 密钥来源（强制）

```markdown
- 框架库【不内置任何密钥】：基于框架以 module 引用方式建的工程不会拷贝框架代码。
- 加密密钥（如 32 字节 AES-256 key）由各工程【应用层】持有，作为参数传入 Encrypt/Decrypt。
- 每个工程使用各自专属密钥，互不通用。
- 密钥【不进版本库】：通过部署环境注入（环境变量 / 密钥管理服务 / 启动参数）。
```

### 7.3 生成与运行期使用

```markdown
- 密文【一次性生成】（Encrypt）：开发/运维用密钥把明文凭据加密一次，写入配置文件。
- 运行期【只解密】（Decrypt）：进程启动时用同一密钥解出明文凭据，构造连接信息。
- 轮换：应用层替换密钥 → 用新密钥重新生成所有密文 → 替换配置文件中的密文。
```

```go
// 加解密接口（占位符）；key 由应用层提供，不内置
func Encrypt(key []byte, plain string) (string, error) // → Base64(nonce ‖ ciphertext)
func Decrypt(key []byte, token string) (string, error) // 空串原样返回；校验失败归一为 ErrInvalidCipher
```

### 7.4 禁止清单（强制）

| 禁止 | grep 锚 |
| --- | --- |
| 配置中心 / 资源凭据**明文**落盘或入库 | `R-CONF-PLAINTEXT-CRED` |
| 加密密钥 / 密码**硬编码**在代码中 | `R-CONF-HARDCODE-SECRET` |
| 含密钥的文件 / 明文凭据文件**提交进版本库** | `R-CONF-SECRET-COMMIT` |
| 配置中心拉取失败**未 fail-fast**（静默空/旧配置启动） | `R-CONF-NO-FAILFAST` |
| 高风险 `{危险开关}` **仅靠配置档位单门禁**（无运行档位 `RUN_ENV` 硬校验叠加） | `R-CONF-DANGER-SINGLE-GATE` |

### 7.5 高风险开关双重硬门禁（强制）

> 决策方法论真相源：[`design-spec/06 §3.3`](../design-spec/06_resilience_design.md)（"危险开关双门禁"）+ §6 反模式"危险开关只靠配置"。本节是其落地表示规范。

高风险 `{危险开关}`（如"跳缓存 / 强制降级 / 危险调试开关"这类一旦在生产误生效即造成事故的开关），**禁止仅靠配置档位单门禁放行**。必须 **配置档位 + 运行档位 `RUN_ENV` 双重硬门禁叠加**，两道门同时满足才放行：

| 门 | 来源 | 约束 |
| --- | --- | --- |
| **门一·配置档位** | 配置中心档位 / 请求标记（如仅 `{test-audience}` 请求允许） | 通过配置中心或入参限定生效范围 |
| **门二·运行档位** | 运行时环境硬校验（`RUN_ENV` 等，仅非生产 env 生效） | 代码内对 `RUN_ENV` 做硬判定，生产 env 直接短路拒绝 |

- 两门是 **AND 关系**：任一门不满足即拒绝；**绝不可能在线上误生效**。
- 只实现门一（仅配置档位）即违规 → `R-CONF-DANGER-SINGLE-GATE`。
- `{危险开关}` 落地后必须登记到 `config.md` 的"危险开关双门禁登记"槽位（开关名 / 配置档位条件 / 运行档位 `RUN_ENV` 条件 / 落点）。

## 8. §9 启动期加载时序（强制 ASCII）

```
进程启动
  │  读环境标记（确定 env / 各配置落到哪个本地文件）
  ▼
读取加密凭据密文 + 应用层密钥（环境注入）
  │  Decrypt(key, cipher) → 明文 username/password
  ▼
构造配置中心连接信息（Addr / Namespace / 明文凭据 / Timeout）
  │  连接失败 ──► ✗ fail-fast（进程退出）
  ▼
Sync(client, items)：逐条拉取 → 覆盖落盘本地文件
  │  任一条失败 / 空内容 ──► ✗ fail-fast
  ▼
本地解析配置文件 → 全局配置变量
  ▼
按配置初始化资源（DB / 缓存 / MQ ...）
```

**fail-fast 点**：解密失败、连接失败、任一拉取失败、空内容——任一发生立即终止，禁止降级静默启动。

## 9. AI 行为约束（强制）

```markdown
### 配置中心与凭据约束（强制）

- 新增 / 修改配置中心凭据 → 必须是密文（§7）；禁止明文落盘 / 入库
- 加密密钥 → 必须来自应用层 / 环境注入；禁止硬编码、禁止入 git
- 新增配置中心拉取项 → 必须登记到 §6.3 拉取映射表；失败必须 fail-fast
- 接入新配置中心品牌 → 必须实现 §6.1 Client 接口，网络调用只在实现层
- 区分配置类别（业务 / 环境 / 凭据）→ 按 §5 权威源矩阵归位，不混放
- 新增 / 修改高风险 `{危险开关}` → 必须 **配置档位 + 运行档位 `RUN_ENV` 双门禁叠加**（§7.5）；仅配置档位单门禁即违规（`R-CONF-DANGER-SINGLE-GATE`），并登记 `config.md` 双门禁槽位
- 拿不准凭据是否已加密 / 密钥是否泄漏 / `{危险开关}` 是否仅单门禁 → 主动报告，不擅自提交
```

## 10. §维护流程（含 B1 反向同步规则）

### 10.1 B1 反向同步规则（强制）

| 代码迹象（git diff） | 反向同步动作 |
| --- | --- |
| 新增 / 修改配置文件中的凭据字段 | 核对是否密文；明文 → 拒绝合并（`R-CONF-PLAINTEXT-CRED`） |
| 新增 `Encrypt` / `Decrypt` 调用 | §7 凭据加密说明同步 |
| 新增配置中心 `Client` 实现 / `Get` 调用 | §6.1 / §6.3 集成说明同步 |
| 新增 `Sync` 拉取映射条目 | §6.3 拉取映射表同步 |
| 修改启动初始化时序（资源依赖配置） | §8 加载时序图同步 + `infrastructure.md` 时序对齐 |
| 新增 / 修改高风险 `{危险开关}` 判定代码（配置档位 / `RUN_ENV` 校验） | 核对是否双门禁叠加；仅配置档位单门禁 → 拒绝合并（`R-CONF-DANGER-SINGLE-GATE`）；登记 `config.md` 双门禁槽位 |
| 出现硬编码密钥 / 凭据明文 | 命中禁止清单 → 拒绝合并 |

### 10.2 维护触发

| 触发 | 动作 |
|------|------|
| 新增环境配置项 | §5 权威源矩阵归位 + §6.3 拉取映射登记 |
| 轮换密钥 | §7.3 重新生成所有密文 + 记录轮换日期 |
| 接入新资源凭据 | §7 加密 + §8 时序评估 |
| 新增 / 调整高风险 `{危险开关}` | §7.5 双门禁（配置档位 + `RUN_ENV`）校验 + `config.md` 双门禁槽位登记 |

## 13. 参考示例（仅示意，落地按业务替换）

> ⚠️ 以下为示意，规范本体（§1-§12）已用占位符表达；落地工程时按业务语义替换占位符，不得直接照搬本节具体业务名 / 具体配置中心品牌作为规范。
>
> 示例服从 [`PRINCIPLES.md §12 占位符规则`](../PRINCIPLES.md#12-占位符规则强制--双轨结构) 的"参考示例段豁免"。

### 13.1 §5 权威源矩阵（示例）

| 配置类别 | 权威源 | 入 git |
|---------|--------|--------|
| 熔断器参数 `app.yaml` | 本地仓库（随发布） | ✅ |
| 资源地址 `resource.yaml`（DB/缓存/MQ） | 配置中心 | ❌（启动落盘） |
| 配置中心账号密码 | AES-256-GCM 密文 + 环境注入密钥 | 密文可入库；密钥 ❌ |

### 13.2 §7 凭据加密（示例：AES-256-GCM）

```
明文 username/password
  │  Encrypt(key32, plain)  // 随机 12B nonce 前置，整体 Base64Std
  ▼
密文（写入环境标记文件，一次性生成）
  │  运行期 Decrypt(key32, token)
  ▼
明文凭据（构造配置中心连接信息）
```

密钥 `key32` 由部署环境注入（如环境变量 `APP_CONF_KEY`），不进仓库；轮换时换 key 重新 Encrypt 全部密文。

### 13.3 §8 启动时序（示例）

```
读 env 标记文件 → 解密配置中心账号密码 → 连接配置中心
  → 拉取 resource.yaml / api.yaml / config.yaml 落盘
  → 本地解析为 RConf/AConf → InitMysql / InitRedis / InitKafka
任一步失败 → 进程退出（fail-fast），不静默启动
```
