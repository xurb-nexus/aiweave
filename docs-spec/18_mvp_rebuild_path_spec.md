# 18 - 分阶段构建路径文档规范（通用方法论）

> 规定 `docs/architecture/mvp_rebuild_path.md` 的内容结构。
>
> **本规范把"分阶段构建路径"从绑定单一项目的 Stage 列表提升为**通用方法论****：覆盖三种使用场景——**从零建设（forward）**、**从 docs 重建代码（rebuild）**、**重构演进（refactor）**——共用同一套分阶段心智模型。

---

## 1. 定位

`mvp_rebuild_path.md` 回答四个问题：

1. **如何把整个系统拆成可独立交付的几个阶段**（Stage 1 / 2 / 3 / ...）
2. **每阶段交付什么**（最小可运行的子系统）
3. **阶段间如何衔接**（依赖、回滚、跳过）
4. **哪些场景下用这份路径**（首次开发 / 重建 / 重构 / 渐进迁移）

> **关键原则**：把"构建路径"从"工程的一次性建设事件"提升为"持续可用的分层心智模型"。它不仅在工程从零建立时使用，也在每次大重构、每次破坏性改动时复用。

---

## 2. 顶层结构（强制）

mvp_rebuild_path.md 必须按以下章节顺序组织：

- `## 1.` 文档定位（三种使用场景 / 与 BUILD_STATUS 的关系 / 何时不使用）
- `## 2.` 阶段拆分方法论（拆分原则 / 单 Stage 判定标准 / Stage 间依赖图谱）
- `## 3.` 前置约束（必须已就绪的代码层级与文档层级）
- `## 4.` 当前工程的具体阶段（Stage 1 / Stage 2 / ...）
- `## 5.` 阶段间依赖关系（**强制：DAG**）
- `## 6.` 每阶段的 AI 执行协议（**强制**）
- `## 7.` 失败回退策略
- `## 8.` 与其他 skill 的协作
- `## 9.` 三种使用场景的具体路径（forward / rebuild / refactor）
- `## 10.` 当前阶段决策记录（如有 🚫 暂不实现等临时节点）

> **完整章节骨架见** [`templates/docs/architecture/mvp_rebuild_path.md`](../templates/docs/architecture/mvp_rebuild_path.md)。下面各节给出每个章节的填充精度与必填要素。

## 3. §1 三种使用场景（强制定义）

```markdown
## 1. 文档定位

### 1.1 三种使用场景

| 场景 | 触发 | 输入 | 期望输出 |
|------|------|------|---------|
| **forward**：从零开发 | 新工程启动，docs/ 已写但代码空白 | docs/ + .claude/skills/ | 按 Stage 顺序交付，每 Stage 独立可运行 |
| **rebuild**：从 docs 重建 | 代码全部丢失或要彻底重写（如换 Go 版本） | docs/ + .claude/skills/ | 按 Stage 顺序重建，与原代码语义等价 |
| **refactor**：演进重构 | 现有工程要做大重构（拆分模块、换技术栈） | docs/ + .claude/skills/ + 现有代码 | 按 Stage 边界做切片，每 Stage 平滑迁移 |

**共用心智模型**：三种场景都遵循"最小可运行子系统 → 渐次叠加"的拆分思路。差异仅在于"上一 Stage 完成"的判定方式：
- forward：从空白开始，逐个 Stage 累积
- rebuild：从 docs 出发，每个 Stage 验证与（原始）期望一致
- refactor：每个 Stage 把"老代码"切片替换为"新代码"，灰度切流量

### 1.2 与 BUILD_STATUS 的关系

`BUILD_STATUS.md` 反映**当前实际状态**，本文档反映**目标路径**。两者交叉点：

- 看 BUILD_STATUS 知道"我现在在哪一阶段"
- 看本文档知道"下一阶段要做什么"
- 每完成一阶段，把对应行从 ⬜ 改为 🟢

### 1.3 何时不使用本文档

- 单一 bug 修复（直接改代码 + 改 md）
- 单 API 接口的迭代（走 OPERATIONS.md W2 工作流）
- 配置参数调整

本文档专为**跨多模块的重大变更**而生。
```

## 4. §2 阶段拆分方法论（最重要的章节）

```markdown
## 2. 阶段拆分方法论

### 2.1 拆分原则

按以下顺序综合判定，得到的就是 Stage 划分：

1. **依赖最少 → 最多**：纯工具函数 / 常量 → 数据访问层 → 业务 service → 接口 → 路由
2. **可独立运行**：每个 Stage 完成后，整个工程能 `go build && go test` 通过，并能跑通某个端到端用例
3. **业务价值递增**：Stage N 比 Stage N-1 多覆盖一个完整业务场景（哪怕是最小的）
4. **代码量上限**：单 Stage 代码生成量不超过 ~2000 行（AI 单轮可靠输出范围）

### 2.2 单 Stage 的判定标准

一个合理的 Stage 必须满足**全部** 4 条：

- [ ] 有明确的"完成验证"（哪个端到端用例必须跑通）
- [ ] 有限的依赖范围（不依赖未启动的 Stage）
- [ ] 有限的代码量（< 2000 行新增）
- [ ] 有清晰的回退机制（实现失败时可单独 revert）

### 2.3 Stage 间的依赖图谱

按"基础设施 → 业务层"分层：

\`\`\`
[L0]  go.mod / pkg/ / 配置文件 / components 错误码 / models / helpers 资源初始化
        │
        ▼
[L1]  公共工具函数 + 业务常量
        │
        ▼
[L2]  最小关键链路（如{核心业务链路} / {身份生命周期闭环}）
        │
        ├──▶ [L3a] 业务模块 A（如{核心实体管理} / {资源管理}）
        │
        ├──▶ [L3b] 业务模块 B（如{核心写入动作} + MQ）
        │
        └──▶ [L3c] 业务模块 C（如后台管理）
\`\`\`

L0 是前置条件，必须在所有 Stage 启动前就绪。L1 是必经路径。L2 之后可按业务优先级并行选择。
```

## 5. §3 前置约束（必填）

```markdown
## 3. 前置约束

开工前**必须**已就绪：

- [ ] `go.mod` / `module` / Go 版本 — 已定义
- [ ] `pkg/` 内部框架库 — 已存在（详见 `pkg_api.md`）
- [ ] `conf/` 配置文件骨架 — 已存在
- [ ] `components/error.go` + `components/service_error.go` — 已存在
- [ ] `models/` 各库表的 GORM struct — 已生成或可由 `/new-model all` 一次生成
- [ ] `helpers/` 资源初始化（mysql / redis / cb / gcache / kafka） — 已存在
- [ ] `middleware/` 通用中间件（cors / metrics） — 已存在
- [ ] `router/http.go` 主入口（已注册 ready 端点） — 已存在
- [ ] `main.go` — 已存在

所有项标注 🟢 来自 `BUILD_STATUS.md`。任何 🟢 项缺失意味着代码被删除，先恢复再开工。
```

## 7. §N 阶段间依赖关系（强制 DAG）

```markdown
## 5. 阶段间依赖关系

\`\`\`
Stage 1 (公共基础)
       │
       ▼
Stage 2 (最小关键链路)  ←──── 可独立部署，让外部消费方集成
       │
       ▼
Stage 3 (业务自助侧)  ←──── 可接前端
       │
       ▼
Stage 4 (业务核心闭环)  ←──── 完整业务环跑通
       │
       ▼
Stage 5 (后台)
       │
       ▼
Stage N（可选）— 启用某 🚫 暂不实现的模块（如有）
\`\`\`

每个 Stage 都可以单独合并进主干，不依赖后续阶段。Stage N 完成后删除"暂不实现"的临时节点。
```

## 8. §6 AI 执行协议（强制）

```markdown
## 6. 每阶段的 AI 执行协议

每个 Stage 开工时，AI 应：

1. **读取本文档对应章节**，确认生成范围
2. **读取 `BUILD_STATUS.md`**，标记相关项为 🟡（进行中）
3. **按 skill 顺序生成代码**：优先 `/new-model` → `/new-middleware` → `/new-service` → `/new-controller` → `/new-router` → `/new-scheduled-task` / `/new-mq-consumer`
4. **每层生成后立即 `go build ./{层}/...` 验证**
5. **全量 `go build ./... && go vet ./...`**
6. **同步新增 / 扩展测试用例**：每个新接口至少 4 类用例
7. **更新 `BUILD_STATUS.md`** 对应行为 🟢
8. **运行 `/doc-sync-check`** 对应 scope，检查反向一致性
9. **汇报交付物给用户**（含测试结果），等用户确认再进入下一阶段
```

## 9. §7 失败回退策略

```markdown
## 7. 失败回退策略

任何 Stage 开发中遇到无法解决的文档漏洞（例如某个 service 方法在 md 中伪码写错参数名），**不要硬写**：

1. 停止生成代码
2. 报告具体的 md 漏洞位置
3. 修复 md → 通过用户评审 → 再继续生成
4. 如果当前 Stage 已生成的代码依赖了错误的伪码，按"签名冲突裁决"例外（PRINCIPLES.md §5）回滚或修正

**Git 回滚约定**：每个 Stage 是一次（或几次）独立提交。Stage N 失败时 `git revert` 即可回到 Stage N-1 的状态。
```

## 10. §9 三种使用场景的具体路径

```markdown
## 9. 三种使用场景的具体路径

### 9.1 forward（从零开发）

\`\`\`
Stage 0：填充 docs/（按 docs-spec/ 顺序）
   │
   ▼
Stage 1-N：按 §4 顺序生成代码
   │
   ▼
持续维护：通过"增量同步"工作流（19_incremental_sync_spec.md）保持 docs ↔ 代码一致
\`\`\`

### 9.2 rebuild（从 docs 重建代码）

\`\`\`
前置：docs/ 已完整，代码丢失或彻底废弃
   │
   ▼
Stage 1：先恢复 L0 前置（go.mod / models / helpers / middleware）
   │
   ▼
Stage 2-N：按 §4 顺序，调 /rebuild-from-docs 逐 Stage 生成
   │
   ▼
每 Stage 跑全量测试，与文档定义的行为对齐
\`\`\`

⚠️ 不要直接 `/rebuild-from-docs all`——AI 单轮无法可靠生成 N 千行代码，按 Stage 推进。

### 9.3 refactor（基于现状重构）

\`\`\`
前置：旧代码运行中，要做架构级重构（如换技术栈、拆服务）
   │
   ▼
Stage 0：更新 docs/（先把目标架构写完）
   │
   ▼
Stage 1：在 旧代码 + 新 docs 之间建立映射（哪些旧文件对应哪个 Stage）
   │
   ▼
Stage 2-N：按本文档 Stage 顺序，逐 Stage 替换旧代码
   │       灰度切流量验证每 Stage
   ▼
完成后清理旧代码 + 旧 docs（如有过时章节）
\`\`\`
```

## 11.5 §11 安全重构方法论

> 解决痛点 #11 渐进式重构。当工程已上线，再做架构级或模块级重构时，重构本身的风险（破坏既有契约、引入回归）必须有显式约束清单管理。

### 11.5.1 N.1 重构前准备（强制清单）

```markdown
- [ ] 识别变更边界（哪些模块受影响）
- [ ] 确认测试覆盖（不覆盖的不动）
- [ ] 定义回滚方案
- [ ] 兼容期策略（新旧共存 / feature flag / 灰度）
```

### 11.5.2 N.2 重构步骤模板（强制 5 步）

```markdown
1. 新建目标代码（**不删旧代码**）
2. 接入点切换（router / 调用方指向新代码）
3. 验证期（新旧双写 / 对比，持续 N 天，详见 §N.4 灰度切流量）
4. 清理旧代码
5. 文档同步
```

### 11.5.3 N.3 AI 行为约束（强制）

```markdown
- 不得一次性删除并重写超过 200 行的已有代码
- 重构必须拆分为 ≤ 5 个独立可回滚的 commit
- 每个 commit 必须通过完整测试
- 重构破坏不变量（删除校验代码）→ 拒绝合并；要么保留校验，要么先讨论是否同步删除不变量条目
```

### 11.5.4 N.4 灰度切流量（生产系统强制）

对涉及"行为变更"（非纯重构，如算法变更 / 数据格式调整）的重构，§N.2 第 3 步"验证期"扩展为：

```markdown
| 阶段 | 行为 | 退出条件 |
| --- | --- | --- |
| Shadow read | 新代码并行执行但不返回（日志比对） | 与旧逻辑一致率 ≥ 99.9% |
| 双写 | 新旧并行执行，结果取旧 | 旁路差异告警归零 |
| 流量切分 | 按 1% → 10% → 50% → 100% 分级 | 每级稳定 N 小时无 P0 |
| 老代码降级保留 | 仅作为 feature flag fallback | 1 个发版周期后清理 |
```

### 11.5.5 N.5 与既有规范的联动

- **20_concurrency_safety.md §2**：重构涉及共享状态时，必须先读 §2 共享状态注册表；重构后 §2 同步更新
- **21_distributed_transaction.md §2**：重构涉及事务边界时，§2 必须显式登记新旧两条边界（标注"重构中"），稳定后再删除旧条目
- **22_performance_contract.md §2**：重构触碰热路径时，必须保证重构后的代码仍满足 §2 禁止操作清单
- **17_testing_design.md §4.7 / §4.9**：重构必须配套故障注入测试 + 性能回归测试（详见对应章节）

### 11.5.6 B1 反向同步规则（强制）

| 代码迹象（git diff） | 反向同步动作 |
| --- | --- |
| 提交说明含 "refactor:" 前缀 + 改动 ≥ 100 行 | 触发 §11.5 检查：是否有重构前准备清单、回滚方案、灰度计划 |
| 新旧代码并存（双写 / feature flag）超过 1 个发版周期 | 提示清理或在 §N.4 灰度切流量进度表登记延期理由 |
| 引入 framework 支撑（feature flag / shadow runner） | 同步 [`docs/architecture/helpers_api.md`](architecture/helpers_api.md) |
