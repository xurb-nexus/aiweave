根据 `$ARGUMENTS` 把一句需求过一遍 `design-spec/` 十三大决策视角，产出技术方案(TRD)。严格按以下步骤执行。

> **本 Skill 是 AIWeave 的【设计类】Skill（第 5 类 · A0 设计阶段）**——`design-spec/` 支柱的唯一执行器，位于 docs ↔ 代码生命周期的**最上游**。它**不生成代码、不写 docs/**，只产出"架构决策 + 落地索引"的 TRD；TRD 再进 A1 forward / B1 sync 落地为 docs/ 与代码。
>
> 公共步骤模板见 [skills-spec/01_skill_authoring_guide.md](path) §A-§E。本 Skill 不生成业务代码，第 5 步测试同步不适用（见末尾"测试同步"）。
>
> **与 PRD→TRD 流的关系**：本 Skill 聚焦"架构决策"，产出 TRD 的技术决策主体。若上游已有 PRD→TRD 流程，可把本 Skill 作为其中"架构设计"环节，把十三视角结论填入既有 TRD 的对应章节，而非另起一份。

## 第 0 步：拒绝规则与启用检查（公共模板见 §A，本 Skill 特定填充）

- ⛔ **拒绝规则**（§A.1）：需求触碰 `BUILD_STATUS.md` §0 中 🚫 暂不实现模块 → 立即停止
- **启用检查**：读 `INDEX.md` §0 采纳进度——`design-spec/*` 标 ⬜ 未启用 → 提示"本工程未启用 design-spec 决策方法论，可先启用或跳过本 Skill"
- **输入完整性**：需求过于模糊（无业务场景 / 无数据形态 / 无读写量级）→ 用苏格拉底式追问补齐**最小决策输入**：① 读多还是写多？② 数据规模量级？③ 一致性 / 延迟有无硬指标？

## 第 1 步：读取设计文档（公共必读见 §B）

```
1.1 读方法论（决策真相源）：
    - design-spec/00（总纲 + 四大元方法 + 统一 lens 骨架 + §6 TRD 结构）
    - design-spec/01..13（十三视角：IO / 数据建模 / 并发 / 事务 / 缓存 / 稳定性 / 尾延迟 / 异步消息 / API 契约 / 定时协调 / 演进迁移 / 容量伸缩 / 可观测性）
1.2 读硬规则真相源（仅被引用，过决策树时按需查）：
    - docs-spec/25(IO 三铁律) / 05(Schema) / 20(并发) / 21(事务) / 06(缓存) / 12(熔断) / 22(性能) / 24(跨服务) / 27(部署)
    - docs-spec/11(MQ 消费) / 07·08(API 接口·流程) / 10(定时任务) / 19(增量同步)  ← 08-12 视角的表示真相源
1.3 读本工程现状：
    - docs/INDEX.md §0（哪些规范已启用）+ docs/BUILD_STATUS.md（🚫 / 已实现）+ CLAUDE.md
1.4 读用户需求：$ARGUMENTS + 关联需求 / 业务文档（如有）
```

## 第 2 步：识别命中的决策视角（Skill 特定）

对需求逐一比对十三视角各自的 `§2 输入信号`，列出**命中视角清单**：

| 视角 | 命中信号（摘自各 lens §2） |
|------|--------------------------|
| **IO**（01） | "对每个… / 同时要… / 高频读 / 高频写 / 纯追加 / 扇出调用"——**只要涉及数据访问即强制命中** |
| 数据建模（02） | 新实体 / 关系建模 / 高频检索 / 数据量增长 / 海量小映射 / 字段演进 / 存储引擎 / 分析查询 |
| 并发（03） | 共享状态 / 并行 fan-out / 后台协程 / 取消传播 / 嵌入式库并发 |
| 事务（04） | 多写聚合 / 跨数据源 / 高频数值变更 / 重试场景 / 读写混杂判定 |
| 缓存（05） | 读热点 / 计算昂贵 / 跨实例 / 一致性敏感 / 海量小映射 / 近静态小表 / 对外 ID |
| 稳定性（06） | 外部依赖 / 热路径 / 过载风险 / 多资源接入 / 凭据 / 危险开关 |
| 尾延迟与过载（07） | P99≫P50 长尾 / 高扇出取最慢 / 共享资源排队 / 过载雪崩 / 重试放大 / 冷启尾延迟 / 严格端到端 SLA |
| 异步消息（08） | 可异步副作用 / 纯追加流水 / 跨服务事件广播 / 双写一致（库+MQ）/ 削峰 / 重复投递 / 顺序敏感 / 消费积压 / 异构同步 |
| API 契约（09） | 新增对外接口 / 一屏多份数据 / 调用方循环调 / 列表翻页 / 接口要改 / 大响应多余字段 / 写接口可能重试 / 实时推送 |
| 定时协调（10） | 周期任务 / flush 巡检 / 全局唯一动作 / 多副本部署 / 预建迁移 / 抢占处理 / 长任务选主 / 崩溃恢复 |
| 演进迁移（11） | 改既有表 / 改字段语义 / 拆并表 / 存量回填 / 换存储换实现 / 换缓存结构 / 带不兼容变更发布 / 大表加索引 |
| 容量伸缩（12） | 定容量（几副本/池多大） / 有 SLA+流量预估 / 流量起伏大 / 加副本不见效 / 伸缩抖动 / 冷启慢 / 有状态扩容 / 容量评审 |
| 可观测性（13） | 开了控制环（池 / 熔断 / 限制 / shed） / 有 SLO 要反推指标 / 要看饱和度判扩缩容 / 有指标要画面板 / 要看依赖健康 |

> **新增四条边的触发口诀**：可异步 / 双写 / 顺序 → **08**；对外接口 / 调用方循环调 / 接口要改 → **09**；多副本下的后台任务 / 全局唯一动作 → **10**；带流量改结构 / 迁数据 → **11**；定容量 / 加机器不见效 → **12**；开了控制环 / 有 SLO 要画面板 → **13**。

## 第 3 步：逐视角过决策树产出结论（Skill 特定 / 核心）

对**每个命中视角**，按其 `§3 决策树 → §4 默认与升级 → §5 权衡 → §6 反模式` 走一遍，得出该视角的决策结论：

```
对每个命中视角：
  ① 识别需求落入该视角的哪一种【形态】（§3 决策树叶子）
  ② 选定【方案】：默认选最简（§4 默认列），逐条核对升级触发是否成立 → 决定是否升级
  ③ 记录【关键权衡】（§5 矩阵）与【规避的反模式】（§6 速查 + grep 锚）
  ④ 标注该结论对应的【硬规则真相源】（如 IO → docs-spec/25 §2；只引用不复制）
```

> **IO 视角强制先做读 / 写路径分治**（[`design-spec/00 §2.3`](path) / [`01 §3`](path)）：先判读还是写，再分别走读路径树 / 写路径树——读求"消灭往返 > 合并往返"，写求"削峰 + 摊销"。一个需求同时有读写时，两条路径分别给方案。
>
> **借纪律不借实现**：决策只产出占位符化的纪律与原语选型（如"按数据形态：同质大对象→MGET，异构小字段→Pipeline"），不抄任何具体工程的魔法数字。

## 第 4 步：组装技术方案 TRD（按 design-spec/00 §6）

按以下结构输出 TRD（这是本 Skill 的产物，**不是 docs/**）：

```markdown
# {需求名} 技术方案(TRD)

## 1. 需求与边界（做什么 / 不做什么）
## 2. 触发的决策视角清单（命中哪几个 lens，各引用 design-spec/0X）
## 3. 各视角决策结论
   （每视角：识别的形态 → 选定方案 → 关键权衡 → 升级触发是否成立 → 硬规则真相源引用）
## 4. 对 docs/ 的影响清单
   （每条结论落到哪篇 docs/，按哪条 docs-spec 表示——即"落地索引"）
## 5. 对 BUILD_STATUS 的影响（本次新增 ⬜ 模块 / 是否触碰 🚫）
## 6. 风险与未决项（需用户裁决的取舍 / 待补的输入）
```

## 第 5 步：自检（Skill 特定）

- [ ] 每个命中视角都过了决策树，无遗漏（尤其 IO 视角必过）
- [ ] IO 视角已显式做**读 / 写路径分治**
- [ ] §3 每条结论都默认优先、升级有触发理由（无过度设计）
- [ ] 硬规则**只引用不复制**（双源真相：决策在 design-spec，规则正文在 docs-spec）
- [ ] §4 每条结论都标了"落地 docs/ + docs-spec 引用"（衔接下游可执行）
- [ ] TRD 用占位符 / 业务中性语言；具体数值留待落地工程按实测填

## 第 6 步：衔接下游（公共格式见 §E，本 Skill 无 go build）

TRD 不替代 docs/，是它的上游"决策记录 + 落地索引"。产出后：

| 场景 | 下游动作 |
|------|---------|
| **新工程（A1 forward）** | 按 TRD §4 影响清单，逐篇填 docs/（按对应 docs-spec）→ 调 `/new-*` 按 Stage 生成代码 |
| **存量（B1 sync）** | 把 TRD 作为"需求技术文档"，连同代码改动喂给 `/sync-feature-to-docs` |
| 落地后兜底 | `/doc-sync-check`、`/io-review`、`/performance-review` 审计决策是否被代码遵守 |

## 测试同步（不适用 / 显式说明）

本 Skill **不生成业务代码**，仅产出 TRD（架构决策文档），故无测试要求——同 `update-index` / `sync-feature-to-docs` 的维护 / 设计类例外（见 [skills-spec/01](path) §D 例外）。TRD 的"验收"是其下游：落地 docs/ 后由 `/new-test` 生成的测试用例间接验证决策正确性。

## 第 7 步：验证

无独立 `go build` / `go test`；产出即 TRD。建议用户在落地 docs/ 后跑 `/doc-sync-check` + `/io-review` 兜底，确认代码遵守了本方案的决策。
